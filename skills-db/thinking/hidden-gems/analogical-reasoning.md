# Analogical Reasoning (Structured) - Gick & Holyoak

**Source**: Gick & Holyoak - Cognitive science research
**Year**: 1980s research publications
**Obscurity**: Tier 2 (⭐⭐) - High (academic cognitive science)
**Domain**: 06-problem-solving / 08-cs-ai-ml
**Score**: 40/50

## Origin & Context

Analogical reasoning represents one of the most powerful cognitive mechanisms humans employ to solve novel problems by drawing connections between familiar and unfamiliar situations. Gick and Holyoak's groundbreaking cognitive science research established that analogical reasoning operates most effectively through **structured domain mapping** - a systematic process leveraging relational schemas rather than surface-level similarities.

Their seminal work demonstrated that when individuals successfully extract and apply abstract problem structures from source domains to target domains, success rates increase dramatically from 30% to 91% - nearly tripling problem-solving effectiveness.

## Core Principle

**Systematic domain mapping from source to target domain via relational schema extraction**

91% success with good schemas vs 30% without structured mapping.

## Core Principles

1. **Structure Over Surface**: Relational architecture matters, not surface features

2. **Schema as Bridge**: Abstract principles transfer, not specific implementations

3. **Multiple Examples Enable Extraction**: 2+ source problems with shared structure reveal underlying pattern

4. **Systematicity**: Interconnected relational systems more powerful than isolated similarities

5. **Explicit Mapping Required**: Conscious structural alignment necessary for transfer

## The Schema Mapping Process

### 5-Phase Framework

**Phase 1: Exposure to Multiple Analogs**
- Encounter 2+ source problems with different surface features but similar relational structure
- Multiple examples force abstraction away from specifics

**Phase 2: Schema Extraction**  
- Identify common relational principle across examples
- Abstract away domain-specific details to reveal transferable pattern

**Phase 3: Structural Alignment**
- Map corresponding elements and relationships between source and target
- Identify how source structure aligns with target structure

**Phase 4: Solution Application**
- Transfer source domain solution strategy to target
- Adapt for domain-specific constraints and differences

**Phase 5: Verification**
- Test whether analogical solution actually solves target problem
- If not, investigate schema mismatch or overlooked constraints

## Why Structured Works (91% vs 30%)

### Good Schemas (91% Success)

**Characteristics**:
- Recognize same abstract concept solves both problems
- Explicit relational principle articulated
- Clear structural correspondence identified

**Example**: "Distributed forces converging to overcome resistance" (radiation problem schema)

### Poor Schemas (30% Success)

**Characteristics**:
- Focus on surface features ("hero gets rewarded")
- Miss underlying relational structure
- Cannot transfer because captured wrong pattern

**Example**: "Both situations involve fighting something" (surface similarity, no solution pathway)

### Intermediate Schemas (40% Success)

**Characteristics**:
- Grasp general principle but incompletely
- "Many small forces solve problem" without full systematic understanding

## The Famous Radiation Problem Example

### Target Problem

**Challenge**: Destroy tumor with radiation without harming healthy tissue

**Constraint**: High-intensity radiation destroys both cancerous and healthy tissue

**Appears Intractable**: Standard solution (increase radiation) creates impossible trade-off

### Source Problem ("The General")

**Challenge**: General must capture fortress in territory center

**Constraint**: Large military force approaching from single direction triggers explosives on roads. Small groups can navigate safely.

**Solution**: Divide force into small groups traveling different roads simultaneously, converging on fortress together

### The Extracted Schema

**Abstract Principle**: "Many small forces can achieve what one large force cannot, provided they converge on central target without aggregating prematurely"

**Relational Structure**:
- Central target requiring concentrated effect
- Constraint preventing direct concentrated application
- Multiple pathways available
- Convergence principle enabling solution

### Solution Mapping to Target

**Transfer**: Direct multiple low-intensity radiation beams from different angles, all converging on tumor

**How It Works**:
- Each beam = low intensity (healthy tissue receives manageable dose)
- Convergence point = tumor (receives concentrated total dosage)
- Solves constraint: No single path has dangerous intensity

**Without Analog**: Only 30% solved through independent reasoning

**With Good Schema**: 91% successfully transferred convergence principle

## Deep Structure vs Surface-Level Analogy

### Surface-Level Analogy (Fails)

**Focuses on**: Superficial similarities in appearance, context, specific details

**In Radiation Problem**: "Both involve fighting something" or "Both have heroes overcoming obstacles"

**Why It Fails**: Provides no actionable solution pathway - two situations can share surface features while having entirely different relational structures

### Structured Analogy (Succeeds)

**Focuses on**: Underlying relational architecture - functional relationships, cause-effect principles, logical governing interactions

**Recognition**: Tumor and fortress are functionally identical as targets requiring concentrated force, regardless of physical differences

**Why It Works**: Penetrates beneath surface to identify transferable relational logic

**Human Tendency**: Naturally gravitate toward surface comparison, must deliberately work to extract relational structure

## Applications in Innovation and Learning

### Innovation Through Cross-Domain Transfer

Organizations/individuals practicing schema extraction from one domain and applying to novel domains generate innovative solutions others overlook.

**Convergence Principle Applications**:
- Military strategy (original) → cancer treatment
- → Network optimization
- → Manufacturing design
- → Organizational leadership

Each application required recognizing underlying relational principle and deliberately transferring it.

### Accelerated Learning

**Insight**: Learning becomes more efficient when encountering **multiple examples with different surface features but identical structure**

Rather than memorizing domain facts, learners comparing multiple examples extract abstract principles with broader applicability.

**Educational Implication**: Curricula presenting multiple structurally similar examples from different domains may produce deeper understanding and better transfer.

### Technical and Creative Problem-Solving

Engineers, architects, scientists, designers who deliberately search for structurally analogous problems in different fields access solution strategies unavailable through domain-specific reasoning alone.

**Cross-functional ideation** reflects this - bringing together people from different domains specifically to provide structurally different but potentially relevant analogies.

### Schema Development and Expertise

Experts develop increasingly sophisticated schemas representing accumulated relational understanding. They solve novel problems more effectively because rich schemas allow recognizing structural patterns and rapidly mapping solutions.

## Common Failures and Avoidance

### Failure 1: Retrieval Failure

**Problem**: Even with excellent match, people fail to retrieve potentially useful analogs (military and medical sit in entirely different mental domains)

**Avoidance**:
- Deliberately practice cross-domain thinking
- Maintain organized knowledge across multiple domains
- When facing novel problem, systematically ask "What other domains address similar goal structures?"
- Develop habits of noticing structural parallels

### Failure 2: Surface-Level Mapping

**Problem**: Focus on object similarities rather than relational structure (59% of Gick & Holyoak participants developed poor schemas)

**Avoidance**:
- Explicitly test whether correspondence involves relational structure or surface features
- Ask: "Does this source feature actually play role in solution, or merely coincidental?"
- Create explicit mappings justifying why each source element corresponds to target element

### Failure 3: Incomplete Schema Extraction

**Problem**: Extract partial schemas capturing some but not all relevant structure (40% success vs 91%)

**Avoidance**:
- Articulate schemas completely and explicitly
- Avoid vague formulations like "persistence pays off"
- Prefer precise relational statements like "distributed application of force achieves concentration at target without individual force amplification"
- Test if extracted schema predicts solutions to both source and target

### Failure 4: Rigidity in Application

**Problem**: Attempt to apply source solutions directly without adapting to target domain constraints

**Avoidance**:
- Distinguish abstract transferable principle from domain-specific implementation
- After extracting schema, engage in deliberate adaptation
- Ask: "What domain-specific factors require modification? What constraints unique to target affect implementation?"

### Failure 5: Constraint Violation

**Problem**: Analogy violates constraints in target domain that don't exist in source

**Avoidance**:
- During verification, explicitly check if proposed solution violates target constraints
- In radiation problem: single powerful beam works in military but violates medical constraint (high intensity destroys healthy tissue)
