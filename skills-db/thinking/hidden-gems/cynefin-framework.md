# Cynefin Framework - Dave Snowden's Sense-Making Model

**Source**: Dave Snowden (1999, IBM Global Services)
**Year**: 1999
**Obscurity**: Tier 2 (⭐⭐) - High (complexity/knowledge management circles)
**Domain**: 04-decision-making / 13-management-strategy
**Score**: 32/50

## Origin & Context

The Cynefin framework is a conceptual model for decision-making created by Dave Snowden while working at IBM. "Cynefin" comes from Welsh meaning 'habitat' or 'place to stand' - reflecting the framework's core idea of providing decision-makers with a "sense of place" from which to view their perceptions.

Draws on systems theory, complexity theory, network theory, and learning theories, functioning as a "sense-making device" for organizations navigating uncertain situations.

## Core Principle

**Different problems require fundamentally different approaches - categorize problems by cause-effect relationships**

Applying wrong solution strategy (no matter how good) worsens outcomes. Understanding domain is critical first step.

## The 5 Domains

| Domain | Cause-Effect | Decision Approach | Practice Type |
|--------|-------------|-------------------|--------------|
| **Clear** | Obvious to everyone | Sense-Analyze-Respond | Best practice |
| **Complicated** | Requires expertise | Sense-Analyze-Respond | Good practice |
| **Complex** | Retrospective only | Probe-Sense-Respond | Emergent practice |
| **Chaotic** | None discernible | Act-Sense-Respond | Novel practice |
| **Disorder** | Unknown which domain | Assess domain → apply | N/A |

### Clear Domain
- Best practices exist and can be universally applied
- Minimal expertise required
- Fact-based approaches sufficient
- Examples: Following recipes, routine customer service

### Complicated Domain
- Expert analysis required to discern cause-effect
- Multiple right answers may exist (therefore "good practice")
- Integration of diverse expertise needed
- Examples: Rocket to moon, medical diagnosis, enterprise software, strategic business planning

### Complex Domain
- Cause-effect only determined in retrospect
- Whole > sum of parts, constant flux
- Trial/error/learning essential, emergent practices
- Non-linear adaptive behavior
- Examples: Raising children, product innovation, culture change, scaling startups

### Chaotic Domain
- No discernible cause-effect relationships
- Immediate action required to establish order
- Goal: Stabilize into complex or complicated
- Innovation emerges from trying responses
- Examples: Crisis response, disasters, system failures, market crashes

### Disorder (Center)
- Uncertainty about which domain applies
- Multiple stakeholder interpretations
- Ambiguity about problem nature
- Explicit sense-making needed

## Step-by-Step Categorization Process

### Step 1: Assess Cause-Effect Visibility
**What**: Can stakeholders clearly see action-outcome relationships?
**How**: Ask "Do we understand how this works?"
**Output**: Visibility classification
**Example**: If yes and universal = Clear; yes with expertise = Complicated; retrospective = Complex; indeterminate = Chaotic

### Step 2: Examine Stakeholder Agreement  
**What**: Do decision-makers agree on problem definition and solution?
**How**: Survey perspectives, identify consensus/discord
**Output**: Agreement level
**Example**: High agreement + clear = Clear; low agreement = Complex or Disorder

### Step 3: Evaluate System Predictability
**What**: Can outcomes be reliably predicted before action?
**How**: Test whether analysis produces confident predictions
**Output**: Predictability assessment
**Example**: Highly predictable = Clear; unpredictable until experienced = Complex

### Step 4: Assess Historical Precedent
**What**: Have similar situations been solved with known solutions?
**How**: Research comparable cases
**Output**: Precedent classification
**Example**: Established practices exist = Clear; similar with variations = Complicated; unique = Complex

### Step 5: Consider System Dynamism
**What**: Is system stable or continuously changing?
**How**: Evaluate rate of change and adaptation
**Output**: Stability assessment
**Example**: Static = Clear/Complicated; rapidly adapting = Complex; in flux = Chaotic

## Common Misapplications

### Error 1: Treating Complex as Clear
Applying best practices to novel situations, surprised when they fail.
**Prevention**: Assess if stakeholders agree problem is identical to previous instances.

### Error 2: Treating Complicated as Complex
Excessive experimentation when expert analysis would resolve faster.
**Prevention**: Check if existing expertise could clarify requirements before assuming emergence needed.

### Error 3: Treating Chaotic as Complex/Complicated
Analyzing crisis instead of taking immediate stabilization action.
**Prevention**: Look for chaos indicators (no cause-effect, immediate threat) → act first, analyze after.

### Error 4: Framework Overkill
Using Cynefin for obvious situations creates analysis paralysis.
**Prevention**: Reserve for genuinely ambiguous situations with meaningful stakes.

### Error 5: Static Assessment
Assigning domain once, never reconsidering as situation evolves.
**Prevention**: Explicitly reassess domain during retrospectives, build re-evaluation triggers.

## When Cynefin Helps vs Overkill

### Helps When:
- Decision-making under genuine ambiguity
- Cross-functional teams perceive situations differently
- Managing portfolio with mixed problem types
- Preventing category errors (wrong strategy for problem type)
- Organizational learning about approach differentiation

### Overkill When:
- Truly obvious situations (thermometer reading)
- Stable mature operations (mostly Clear/Complicated work)
- Time-sensitive crises (act first, framework later)
- Low-stakes decisions (office catering vendor choice)
- Organizations already distinguish approaches well

## Integration with Agile/Scrum

### Complex + Scrum (Optimal Fit)
Scrum works very well for Complex where requirements emerge.
- Sprint iterations = safe-to-fail experiments
- Success = learning and pattern recognition
- Retrospectives surface emergent patterns
- Stories evolve based on evidence

### Complicated + Scrum
Requires adjustment:
- Longer investigation sprints
- Include expert specialists
- Architectural spikes reduce complexity
- Document expert decisions

### Clear + Agile
Agile overhead unnecessary for straightforward work:
- Use kanban for operations
- Minimize ceremonies
- Efficiency as goal

### Chaotic + Rapid Response
Suspend Scrum during crisis:
- Command-and-control temporarily
- Restart Scrum after stabilization
- Post-incident reviews extract learning

**Conflicts with**:

- Pure analytical approaches for Complex problems
- Pure experimental approaches for Complicated problems

**Leads to**:

- Adaptive leadership frameworks
- Organizational learning systems
