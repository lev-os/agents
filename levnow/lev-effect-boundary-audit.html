The Architectural Answer

The naming problem is real, but the deeper problem is ownership.

Current

core/orchestration is doing graph traversal, queues, topology, proof, receipt, effect declarations, effect-control, eval result projection, CDO handlers, worktree utilities, and direct shell execution. That is too many reasons to change in one package.

Ideal

Keep agent orchestration for multi-agent coordination. Build Lev runtime from bottom-up primitives: shared domain contracts, FlowMind declarations, exec as the only effectful path, an effect controller/evaluator engine, a run ledger, and projections for eval/proof/receipts.

Why `agent orchestration` is a loaded term now

Industry usage has converged on multi-agent coordination, not Lev's low-level runtime algebra.

SourceCurrent usagePressure on Lev namingURL

Naming implication

If Lev keeps a package called core/orchestration, it should own agent coordination patterns: team handoffs, group-chat/tribunal style flows, multi-agent task allocation, shared state, and governance. It should not own primitive effect sealing, proof refs, typed receipts, direct shell execution, or eval scorecards.

Run-fabric decomposition

The ledger concern is real: idempotent runs, action leases, heartbeats, retry state, replay indexes. The proof concern should move under effect/evaluator receipt sealing. That means run-fabric becomes an operational log, not the semantic proof fabric.

Next

Recommended consolidation sequence

1. Freeze the vocabulary

Write the boundary doc around existing primitives: EffectEnvelope, ObservedEffect, EffectReceipt, typed refs, exec request/response, run ledger. Avoid new type catalogs.

2. Ban shadow execution

Inventory every child_process/shell site outside exec/harness/tmux-harness and route effectful work through ExecutionRunRequest.

3. Split run ledger from proof

Keep RunService leases/idempotency; move certification semantics into effect receipt sealing with shared typed refs.

4. Rename orchestration by scope

If it is multi-agent coordination, keep orchestration. If it is graph traversal, call it traversal/loop/runtime graph. If it is effect policy, put it under effect.
