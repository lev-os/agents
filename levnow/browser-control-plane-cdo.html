Browser / VFS Control Plane

The browser is real. The projected namespace is real. But the first implementation is proof-first, not filesystem-first.

10 turns · 50 agents · final synthesis

We ran a 10-turn, 50-agent adaptive CDO on one question: what is the browser/WASM/VFS idea actually becoming in Leviathan? The run converged on a clean answer. Browser, mobile, extension, and WASM are host surfaces. AgentFS and projected filesystems are execution and projection substrates. WebMCP is a semantic page-tool adapter, not yet a namespace or scheduler. And the first thing to build is not a browser filesystem — it is a proof-first stack: explicit effect identity, fail-closed authority, receipt/finality, and only then the WebMCP A/B/C harness.

The core answer

Treat browser/WASM/VFS as host surfaces and execution/projection substrates, not as a new root runtime. The first spike is a proof-first WebMCP A/B/C harness after browser authority verification exists. Projected namespace work is deferred.

**Browser host surface:** an attended, capability-scoped control-plane client that can observe, present, approve, emit user actions, validate UI/runtime truth, and sometimes invoke bounded executors. **WebMCP:** a semantic page-tool adapter over `navigator.modelContext.tools` and `callTool()`, not yet a scheduler or namespace. **AgentFS:** a scoped mount projection plus writable overlay and execution substrate over canonical LevEvent, graph, proof, and receipt contracts. **ArchitectureIR:** classification and boundary refs only, never native runtime truth. **Proof/eval fabric:** the owner of proof obligations, witness sets, receipt policy, ratchet admission, and settlement eligibility.

WEBMCP

What WebMCP changes — and what it does not

The run was brutally clear about this.

✅

What it is

A semantic page-tool adapter. Structured eyes and hands over browser-native capabilities. Cheap, local, and useful for bounded A/B/C falsification.

⛔

What it is not

Not a scheduler. Not a namespace. Not a market. Not a filesystem. Not a durable authority layer. If it just renames callTool() as file paths, it is redundant.

ROLE MAP

Who does what

The run converged on a clean role split.

Client

Issues intent/effect requests, presents authority, receives projections and receipts, and never gets ambient executor handles.

Gateway

Owns transport bridge lifecycle, fallback equivalence, trace and receipt propagation. No native business semantics.

Worker

Executes bounded effects under an admitted envelope and emits effect results and receipts. Does not self-authorize.

Provider

Supplies native capability: CDP, WebMCP, AgentFS, model, tool, or remote service behind a contract.

Verifier

Evaluates proof and receipts, attests or rejects eligibility, but does not execute the effect and does not own native semantics.

FIRST SLICE

The proof-first implementation sequence

This is the real output of the run.

1

. Contract wave

Create core/domain/src/poly-protocol/, add only ./poly-protocol export, implement AdmissionEffectEnvelope v0, add resolver probes, root-import negatives, owner/profile tests.

2

. Receipt identity wave

Add effectId / effectIds, preserve proofSpecId and verifierResultIds, and enforce settlement-hard against quorum-only bypasses and naked IDs.

3

. Browser authority wave

Add AgentPing browser authority verifier and wire /api/v1/tool and /api/v1/cdp before any browser-cascade or WebMCP promotion.

4

. AgentFS finality wave

Add guarded-finality conformance tests and raw-handle compile-fail tests before changing storage behavior.

5

. WebMCP falsification wave

Add the local A/B/C harness only as a negative test surface. No namespace promotion until upstream verifier and shared effect_id are proven.

DEFERRED

What the run explicitly pushed out

This matters because the temptation was to jump straight into productizing the browser filesystem idea.

Deferred by the run: projected WebMCP/browser namespace, OPFS, `/browser/...` trees, generator/registry/marketplace work, selector healing, live ChatGPT canaries, full AgentFS POSIX parity, checksum/scrub, multiwriter overlay policy, Turso remote sync finality, x402 settlement execution, remote worker market overlays, reputation, and auth-sniffer redesign. The browser/VFS idea was not rejected. It was demoted from first spike to later wave until proof, authority, and receipt semantics are real.

NEXT

If you build one thing next, build this

The browser filesystem dream survives only if the substrate underneath is mechanically trustworthy.

Build proof, then projection

Do not start with namespace streaming, browser filesystem metaphors, or market overlays. Start with AdmissionEffectEnvelope v0, explicit effect_id, fail-closed browser authority, guarded AgentFS finality, and the local WebMCP A/B/C falsifier.
