THE PROBLEM

Graph Walker Stops After Step 1

The DNA Cascade has 8 action nodes connected by `next` pointers: scan-dna, plan-updates, assess-plan, execute-dna, execute-spec, execute-design, manifest-code, verify. Each node spawns a Claude agent via the SDK adapter, writes structured output to `.lev/flows/state/`, and passes the file path to the next node.

The bug: Step 1 (scan-dna) completes perfectly — the agent writes 388 lines of constraint analysis. But the graph walker never advances to Step 2. The process exits with code 0. No "Graph finished" message. No "Graph failed" message. The agent did its job. The runner didn't care.

This was **lev-pd2w**, a P1 blocker. No multi-step flow worked. The entire FlowMind execution engine was a single-step machine.

ARCHITECTURE

The Execution Stack

FlowMind

Compiles .flow.yaml declarations into GraphDocuments. Owns which step to run next via ecStart/ecNext state machine. Pure functions — same input always produces same output.

Orchestration

Owns the execution contract: start(), next(), resolveNextNode(). Pure state machine transitions. Bounded loops, circuit breakers, visit counting. 531 tests.

Harness

Runtime runner beneath @lev-os/exec. Owns process spawning, adapter selection, stream management. The while loop that calls dispatchInstruction() then ecNext().

SDK Adapter

Claude Agent SDK wrapper. Spawns claude as a child process, streams messages via AsyncGenerator, detects TASK_100_DONE promise. Returns success/failure to the graph walker.

THE FIX

Before & After

Before: Eager Registration

Signal handlers registered at import time via setupTmuxCleanupHandlers() at module level. Every lev exec call — even with SDK adapter (no tmux) — registered SIGTERM/SIGINT handlers that call process.exit(0). Any signal killed the entire process, including the graph walker's pending async work.

Result: Graph walker dies mid-execution. Process exits with code 0. No finalization message. The cascade stops after Step 1.

After: Lazy + Graceful

Signal handlers only registered when a TmuxHarness is actually created via ensureTmuxCleanupHandlers(). Guard prevents double-registration. When handlers DO fire, they use process.exitCode = 0 + a 5-second grace period timeout instead of immediate process.exit(0).

Result: SDK adapter runs uninterrupted. Graph walker completes its full loop. All 8 nodes execute. Cascade works end-to-end.

DNA GAPS

10 Missing Constraints Identified

GAP-1 — Adapter Return Contract

P1 | harness, orchestration

Adapter run() must return a structured result to the caller before the graph walker advances. Without this, ecNext receives undefined and the state machine corrupts.

GAP-2 — Walker Completion Guard

P1 | orchestration

Graph walker must not exit before reaching a terminal node or exhausting its visit budget. A 5-step graph must not stop after step 1.

GAP-3 — Null Result Handling

P2 | orchestration

ecNext must handle null/undefined lastResult gracefully — retry, fail, or produce a diagnostic instruction. Never silently skip.

GAP-4 — Promise Chain Integrity

P1 | harness, orchestration

The promise chain from adapter invocation through harness.complete() through the orchestration loop must be fully awaited before state transitions.

GAP-5 — Timeout Layer Interaction

P2 | harness, orchestration, exec

Adapter timeout, graph walker timeout, and exec timeout must have defined interaction semantics. Currently: undefined behavior at boundaries.

GAP-6 — Step Result Storage

P2 | orchestration

Step result from adapter must be stored in ExecutionState.vars before ecNext computes the next instruction. Prevents stale state.

GAP-7 — Terminal Event Emission

P3 | orchestration

When ecNext returns no instruction (cursor null), the walker must exit cleanly, emit a terminal event, and produce a session receipt.

GAP-8 — Error Classification

P3 | exec, orchestration

Adapter errors need classification: transient vs permanent, retriable vs fatal. Circuit breaker should not trip on retriable network errors.

GAP-9 — Parallel Merge Ordering

P3 | orchestration

When steps execute in parallel, results must merge into ExecutionState in a deterministic order. Non-determinism breaks C2 (non-commutation) replay.

GAP-10 — Adapter Interface Contract

P2 | harness, orchestration

The exact contract shape between harness.complete() and orchestration's step executor is unconstrained. Interface drift between layers is undetected.

Why This Matters

The graph IS the discipline. When your execution engine can run its own diagnostic cascade, the system has crossed from tool to organism.

This isn't just a bug fix. It's proof that the DNA Cascade architecture works:

**Scan** found 68 constraints and 10 gaps across the execution stack. The gaps weren't obvious from the code — they lived in the negative space between modules.

**Plan** identified the exact 4-tier update needed: DNA constraints → spec changes → design implications → surgical code fix.

**Manifest** produced 4 changed files, all tests green, no regressions.

**Verify** proved end-to-end: the graph walker now advances from node to node, the cascade completes, the fix is pushed.

The cascade didn't just fix a bug. It demonstrated that FlowMind programs can reason about their own infrastructure — a recursive loop where the system improves itself by running itself. This is what DNA-first development looks like when it works.
