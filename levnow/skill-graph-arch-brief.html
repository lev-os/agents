ARCH BRIEF

The Interview

Driver, quality attributes, ADRs

**Driver**

The 4-lane operator model is empirically load-bearing. χ=4 (TopoNetX), 95.10% Louvain agreement, hand-authored taxonomy — three independent validations. Owner stack locked (Pydantic→JSON→NetworkX→GraphML). But skill graph is only ONE of many entity types Lev tracks: content (articles, /now pages), assets (images, videos, audio), beads (tasks), flows (flowmind YAMLs), specs (dna/core), decisions (ADRs), handoffs (.lev/pm/handoffs). They all live in the filesystem. They all have frontmatter. They all reference each other via wiki-links. **They are the same graph**.

**Quality Attributes**

- **Correctness**: 7/7 gates pass, χ=4 invariant, Pydantic round-trip stable - **Scale**: 155 today, 10k after Phase 1 ingest, 100k when beads + assets + content land - **Performance**: <100ms query at 1k, sharded beyond 10k - **Cross-kind traversal**: `nx.ancestors(skill://work)` must reach `content://constraint-manifesto` via IMPLEMENTS + REFERENCES edges - **Framework attribution**: Every skill/feature tagged with source project (OMX, OOO, Lev, Hermes, OpenClaw) - **Overlap detection**: When 5 frameworks all implement 'doctor/status', the graph shows that convergence

**ADRs (locked + proposed)**

- ADR-001: Owner = Pydantic + JSON + NetworkX + GraphML ✅ - ADR-002: Sidecars may not write to owner ✅ - ADR-003: Promotion = 5 gates + human audit ✅ - ADR-004: `skill://` is canonical runtime URI ✅ - ADR-005: FalkorDB deferred to clawbuddy marketplace ✅ - ADR-006: Cytoscape.js is graph explorer UI library ✅ - ADR-007: Fast-follow ingest = filesystem as graph substrate ✅ - ADR-008: AgentPing gen-ui hosts explorer, exported as lev-now asset ✅ - **ADR-009 (new)**: Single NodeKind enum covers ALL entity types (SKILL, FEATURE, ROLE, LANE, FILE, CONTENT, ASSET, BEAD, FLOW, SPEC, DECISION, HANDOFF) - **ADR-010 (new)**: `source_project` field is canonical for framework attribution — OMX/OOO/OMC/OMO/Hermes/OpenClaw/Lev - **ADR-011 (new)**: Cross-kind edges use same EdgeKind enum — REFERENCES, IMPLEMENTS, SUPERSEDES work across SKILL→CONTENT, BEAD→FLOW, etc.

UNIFIED VISION

Six Graphs → One Substrate

Each entity type is a NodeKind; all share the owner stack

Skill Graph

155 nodes today. 46 skills + 81 features + 24 roles + 4 lanes. Tracks capabilities and their framework origins.

Content Graph

~200 /now pages + articles. Constraint manifesto, lifecycle venn, 3-part series. Published to here.now. Each a CONTENT node.

Asset Graph

~500 images + videos + diagrams. ~/.agents/diagrams/, levnow/ media, /now renders. Each an ASSET node.

Beads Graph

BD issues + dependencies. Cross-reference with skills (this bead uses skill://work). Each a BEAD node.

Flow Graph

~150 .flow.yaml files. FlowMind topology already graph-native. Each flow becomes a FLOW node with COMPILES_TO edges.

Decision Graph

ADRs, specs, handoffs. .lev/pm/decisions/, docs/specs/, .lev/pm/handoffs/. Each a DECISION or HANDOFF node.

INTELLIGENCE · OUTLIERS

Where the Graph Disagrees With Itself

Skills that straddle lanes — candidates for Lane.CROSS

**Spike 07 (NetworkX community detection) found 3 outliers** — skills where Louvain, Label Prop, and Greedy Modularity disagree with hand-authored lane assignment:

- **`cdo`** — labeled Shape, but community detection places it in Deliver. It's a deliberation tool used in both alignment AND execution phases. - **`work`** — labeled Plan, but clusters across Plan + Deliver. It's THE lifecycle orchestrator — spans all phases. - **`arch`** — labeled Plan, but it's used in Shape (interview) + Plan (ADRs) + Close (review).

**Proposal**: Add `Lane.CROSS` to the enum for multi-lane skills. These three get re-tagged as CROSS. The community detection agreement drops slightly (95.10% → maybe 96%+) and the model becomes more honest about real-world usage.

**Also**: `lev-align` appears in both Plan and Close in the taxonomy (spike 7 found it as the boundary-crossing node). It IS the bridge between crystallization and validation. Keep it dual-tagged or promote to CROSS.
