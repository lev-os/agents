CANDIDATE 3

FlowMind for_each — Collection Iteration

The #1 missing capability per NAAC dogfooding, but with significant design ambiguity

FlowMind doesn't support for item in collection yet. I wrote a bash script to do the outer loop. That works, but it means the orchestration is split: FlowMind handles the inner graph, bash handles iteration. This is exactly the kind of thing FlowMind should handle natively. — NAAC Dogfooding Report

What Exists vs What's Needed

Exists Today

Until-loop (repeat until condition, circuit breaker, idle detection). GraphNode has unused foreach? field — parser recognizes it but executor ignores it. LoopExecutor injection pattern via AdapterEnv. Execution contract handles kind: 'loop'. Parallel execution with concurrency limits works.

Needs Creation

Collection iteration — zero support anywhere. Domain type additions (GraphNode fields, InstructionKind). Collection resolver (shell to array). Execution contract for_each detection. ForEachExecutor interface + fulfillForEach(). Collection runner with concurrency. ~1,710 LOC across 8-10 features.

#Design QuestionOptionsImpact
1Collection sourceShell output? Knob array? File? JSONL? All?High
2Item variable scopingKnob override? New variable scope?High
3Per-item error handlingSkip? Abort? Retry? Failure policy?Medium
4Progress / resumabilityCheckpoint at item 1500 of 3000?Medium
5Result aggregationHow are per-item results collected?Medium
6Instruction kindNew 'for_each' kind or extend 'loop'?High
7Relationship to parallelDynamic targets from collection? New topology?High
Viable but needs a design doc first

The infrastructure is solid and the pain point is real. But building without resolving these 7 questions risks building the wrong thing. A 1-page design doc answering each question would unblock implementation. The NAAC report even includes proposed YAML syntax.

The codebase has significant execSync debt beyond V5+V7. But those two are cleanly scoped: V5 is an orphaned handler with no surface route (genuinely dead code), and V7 is a 15-line function that was extracted FROM harness TO orchestration and needs to go back. Neither has hidden dependencies.

Deleting cdo.ts removes 1 of 24 handler files in core/poly/src/handlers/. The no_poly_handlers validation gate will still fail. But it's tracked separately with an active alignment session (poly-design-spec-alignment).
