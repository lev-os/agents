EXECUTIVE READ

The Decision Frame

The page answers what each external platform contributes, where their primitives overlap, which edges are rare, and what Lev should actually build.

Run Fabric is the product spine

The system object is the run lifecycle, not lev exec, a dispatcher service, a dashboard, or a provider-specific adapter.

Executable events need admission

A flow appearing in the project cannot make an event executable by accident. Admission resolves profile, adapter, model, variant, idempotency, retry, and evidence policy before dispatch.

RunFusion adds state classes

Append-only events, strong leases, queued deliveries, local handles, projections, and forbidden secrets need different coordination rules.

Lev keeps proof as the edge

Receipts prove execution happened. GateProofs prove claims. Proof bundles assemble promotion evidence. Most external stacks do not make that distinction hard enough.

Lev Shape From The Corpus

The consolidated shape is not Airflow plus Orloj plus Fusion. It is a Lev-owned run lifecycle with state classes and proof gates.

NEXT SLICE

Consequence For Event Daemon

The daemon path starts from admission, not flow matching. This preserves one local JSONL ordering property while blocking accidental execution.

event-admission-spine.txt
LevEvent fact
  -> adapter/project TriggerSpec
  -> executable request
  -> admission resolver
  -> resolved RunEnvelope or typed rejection
  -> event log append
  -> daemon dispatch consumes only admitted RunEnvelope records
  -> Run/Action/Attempt lifecycle
  -> receipt/trace/replay/proof projection
Why this fixes daemon brittleness

A passive event can be appended as fact. An executable event is only executable after project/root policy resolves it into a complete ExecutionIntent and RunEnvelope. Missing model/profile/adapter/variant becomes a typed rejection, not a best-effort dispatch.

# Design: MVP Run Fabric Vision

## Executive Summary

Lev should not ship another agent wrapper, prompt runner, or workflow toy. The
MVP should establish a **Run Fabric**: a proof-bearing orchestration control
plane where agents, tools, events, cron jobs, UI actions, SDK calls, MCP calls,
webhooks, replay, and direct operator commands all lower into one durable run
lifecycle.

The product target is "Hyperagent-class team infrastructure, but Lev-native":
agents that do real work across the team, learn business context, run from
Slack/MCP/cron/SDK/UI-like surfaces, and expose fleet-level oversight. The Lev
differentiator is not the landing page; it is the substrate: local-first,
poly-surface, event-addressable, replayable, receipts-first, and gate-proven.

Phase 1 must be meaty enough to feel inevitable: boring plumbing, strong
interfaces, generated surfaces, retry/backoff/log/trace/receipt/proof semantics,
and one self-managing SDLC proof. This is not greenfield. The repo already has
major pieces of the evidence layer; the MVP is about unifying them into one
product contract and making every surface use it. UX/product feel can iterate on
top of this in Phase 2 without re-litigating the runtime.

## Problem Framing

### Current State

Lev already has most primitives, but they are not yet composed as one product
spine:

- **Function** exists as Poly handlers and plugin/module commands.
- **Trigger** exists in FlowMind declarations, hooks, schedules, and event
  concepts.
- **Worker** exists through plugins, daemons, tmux/session substrates, SDK
  adapters, and provider registries.
- **Execution evidence** exists through receipts, traces, gate runs, artifacts,
  and Pentagon/eval work.

The gap is not another CLI command. The gap is that these primitives are not yet
obviously one system. Events, direct execution, SDK calls, UI actions, MCP
tools, and replays should all create the same kind of run, produce the same
observable state, and end in the same proof model.

### Existing Assets

This plan should treat roughly half of the boring substrate as already built,
but fragmented.

| Area | Existing Asset | Phase 1 Work |
|---|---|---|
| Receipts | `lev receipt validate`, execution receipt records, graph/gate observability fields, dry-run receipt validation | Make receipt creation/query a first-class Run API contract instead of a handler-local artifact |
| Traces | `lev trace show` reports flow, graph, nodes, branches, gate runs, stdout/stderr, evidence refs | Promote trace continuity to a required invariant across CLI/SDK/MCP/HTTP/UI/replay |
| Gate evidence | `core/exec/src/gate-run.ts`, `lev gate validate`, command/reviewer gate runs with evidence refs | Rename/productize as `GateProof` / `ProofBundle`; keep structural validation separate from semantic truth |
| Logs/artifacts | `.lev/agentfs/exec/artifacts/<exec-id>`, stdout/stderr/output paths, event JSONL | Add a stable log stream/query contract; do not leave logs as only filesystem knowledge |
| FlowMind graph events | `exec.graph.*`, `exec.gate.run`, graph validation/entry/node metadata | Bind graph execution to the same `Run` aggregate and replay packet |
| Managed execution | `ExecutionBinding`, managed tmux proof paths, opt-in tmux session control | Make substrate selection descriptor-first and prove tmux parity before defaulting |
| Proof tooling | orchestration proof helpers, testing proof helpers, Pentagon/eval concepts | Collapse into one acceptance vocabulary for promotion: receipt + trace + gate proof + replay |

The hard part is not inventing receipts/traces/proofs. The hard part is making
them impossible to bypass.

### Desired Experience

An operator should be able to define an agent once, then invoke it from anywhere
their team already works. The run should be inspectable while active,
cancelable, retryable, resumable when supported, and auditable after completion.
The output should not be "the agent said it worked." The output should be a run
record, logs, trace, receipt, gate verdicts, replay context, and a proof bundle
when promotion is required.

The team should feel the product as:

1. create or import agents with persistent skills, tools, memory, model, and
   budget;
2. trigger agents from user action, event, schedule, webhook, SDK, or MCP;
3. watch runs with reliable status, logs, cost, retries, and evidence;
4. evaluate quality with configurable rubrics and gate proofs;
5. promote useful skills/memory only through reviewable, bounded learning.

## Product Requirements

### Problem Statement

Agent products are easy to demo and hard to run. Most stacks collapse execution,
orchestration, UI, provider quirks, memory, and proof into one fragile surface.
That creates hidden state, untraceable failures, provider-specific branches,
and dashboards that look impressive but cannot explain what actually happened.

Lev needs to win the category by making agent work operationally real:
repeatable, inspectable, governable, replayable, and surface-agnostic.

### Solution Direction

Build the MVP around a Run Fabric, not a CLI command.

```text
SDK Contract
  agent.run / flow.run / run.status / run.logs / run.proof
    ->
Poly Binding
  one operation manifest projected into CLI, MCP, HTTP, UI, replay
    ->
Surface Hosts
  CLI router | daemon-hosted MCP/HTTP | UI gateway | event replay
    ->
Intent + Admission
  normalize request/event, apply policy, resolve agent/flow/task/workstream
    ->
Run Fabric
  compile control plan, schedule work, lease resources, coordinate retries
    ->
Execution Substrate
  tmux | headless ACP | sandbox | local SDK | browser | remote worker later
    ->
Evidence Fabric
  logs | trace | receipt | GateProof | replay packet | proof.yaml
    ->
Projection
  command center | CLI | MCP | HTTP | SDK | notifications
```

`lev exec` remains an ingress and operator projection, not the architecture. The
system object is the run lifecycle. There is no `lev dispatch start` product
path: dispatch is an internal event/trigger concern that creates an
ExecutionIntent, and a run starts through the SDK/Run contract.

### MVP Daemon Blocker Proposal

The immediate execution-ready cut is a boundary repair, not a new surface
framework:

1. **SDK-first contract**: `@lev-os/sdk` exposes the run operation and handle
   shape; SDK responses carry run identity and evidence handles.
2. **Poly binding projection**: Poly owns command/tool/route/action metadata for
   the same operation, but does not create run envelopes or own HTTP/MCP
   lifecycle.
3. **Daemon SurfaceHost**: daemon owns real HTTP/MCP handles, ports, health,
   status, stop semantics, auth/config hooks, and supervision. A surface either
   starts a real handle or returns explicit `not_implemented`.
4. **Evidence through Run**: daemon-hosted HTTP/MCP calls resolve Poly binding
   metadata, invoke the SDK/Run contract, and return run identity with
   receipt/trace/proof lookup.
5. **Boundary review**: after implementation, run an anti-slop review focused on
   duplicate runtime logic, fake lifecycle state, direct provider bypasses, and
   new generic abstractions.

This deliberately avoids a generic `SurfaceRuntime` or Poly-owned served
runtime. The daemon is the host; Poly is the binding map; SDK/exec is the run
contract.

### User / Operator Stories

1. As a founder/operator, I want to save an agent with tools, skills, memory,
   model, and budget so that my team can run it repeatedly without bespoke
   setup.
2. As an engineer, I want the SDK call to define the run shape, with CLI, MCP,
   HTTP, replay, and UI projected from the same Poly binding, so debugging does
   not depend on which surface launched it.
3. As a reviewer, I want receipts, logs, traces, gate verdicts, and replay
   context so that "done" means evidence-backed, not vibes.
4. As a platform owner, I want provider/runtime quirks behind PolyCard and
   ExecutionBinding-like contracts so that new agents/substrates do not fork the
   product.
5. As a product designer, I want command-center UX to sit on stable run APIs so
   that UI iteration does not corrupt runtime semantics.

### Implementation Decisions

#### Phase 1 System Nouns

- **AgentDefinition**: persistent agent configuration: prompt/system
  instructions, tools, skills, memory refs, model policy, budget, permissions,
  retention, and default execution profile.
- **ExecutionIntent**: normalized reason to run: `direct`, `event`, `schedule`,
  `webhook`, `sdk`, `mcp`, `ui`, `task`, `workstream`, or `replay`.
- **TriggerSpec**: rule turning a source event or schedule into an
  ExecutionIntent.
- **Run**: durable aggregate for one execution attempt tree.
- **RunHandle**: stable handle for status, wait, cancel, retry, logs, trace,
  receipt, proofs, and children.
- **SessionHandle**: live handle for interactive/substrate session behavior:
  attach, stream, message, capture, resume, nudge, kill where supported.
- **Worker**: any process/substrate that can execute a Function/PolyOp under
  policy.
- **Function / PolyOp**: stable unit of callable work exposed by modules,
  plugins, agents, or generated surfaces.
- **EvidenceRef**: pointer to logs, artifacts, traces, receipts, gate proofs,
  replay packets, screenshots, files, or measurements.
- **GateProof**: gate-scoped verdict with claim/gate IDs, evidence refs,
  trace/receipt refs, replay/falsification context.
- **ProofBundle**: eval-owned `proof.yaml` manifest assembling receipts,
  GateProofs, traces, replay hashes, and artifact hashes.

#### Phase 1 Modules / Ownership

| Concern | Owner | Phase 1 responsibility |
|---|---|---|
| Public SDK contract | `@lev-os/sdk` over `@lev-os/exec` | First-class `agent.run`, `flow.run`, status, logs, proof, and wait/cancel/retry handles |
| Public run creation contract | `@lev-os/exec` / core exec surface | Create and inspect runs from normalized intents |
| Intent normalization | exec + event-dispatch boundary | Convert events/schedules/webhooks/surfaces into ExecutionIntent |
| Trigger matching | event-dispatch component, not user CLI | Match LevEvent/cron/webhook/git events to TriggerSpec |
| Flow semantics | FlowMind | Compile/control the graph, not own runtime ingress |
| Surface binding | Poly | Generate CLI/MCP/HTTP/UI bindings from one SDK/PolyOp manifest |
| Served surface lifecycle | daemon | Host and supervise MCP/HTTP surfaces, ports, health, auth, and real server handles |
| Runtime substrate | harness/tmux/acp/sandbox providers | Execute with SessionHandle/Worker contracts |
| Operational state | right tool per concern | Scheduling/current state may be DB/table/queue/file if mirrored into evidence |
| Evidence | exec + trace + receipt + eval | Logs/traces/receipts/GateProof/proof bundle |
| UX | Phase 2 command center | Build on run APIs, not own orchestration |

#### Required Plumbing

- idempotency key per source event / invocation;
- correlation ID, run ID, session ID, traceparent, source event ID;
- retry policy with max attempts, retryable error taxonomy, backoff, jitter, and
  dead-letter behavior;
- timeout policy for run, node, substrate spawn, no-heartbeat, and user wait;
- lease model for files, secrets, tools, worker slots, and mutable state;
- log stream with stdout/stderr/artifact segmentation;
- trace stream with event -> intent -> run -> node -> substrate -> gate edges;
- receipt generation for execution finality;
- GateProof generation for semantic claims and promotion gates;
- replay packet for event/run/proof reproduction at the relevant proof level.

### Testing Decisions

Phase 1 is not done because APIs compile. Phase 1 is done when these gates pass:

1. **Intent parity gate**: SDK, direct CLI projection, daemon-hosted MCP/HTTP,
   event replay, and FlowMind run all produce the same Run shape.
2. **Trigger proof gate**: a real/replayed git or SDLC event becomes an admitted
   ExecutionIntent and creates a run without special-case CLI glue.
3. **Run lifecycle gate**: status, wait, cancel, retry/backoff, timeout,
   heartbeat, logs, trace, receipt, and gate refs are observable.
4. **Surface generation gate**: CLI/MCP/HTTP/UI projections are generated from
   the same Poly binding, call the SDK/Run contract, and do not own runtime
   behavior.
5. **Daemon surface gate**: `daemon surfaces start http|mcp` starts real
   supervised surface handles or reports `not_implemented`; it must not mark
   intent-only state as running.
6. **Proof gate**: one self-managing SDLC demo emits receipt, trace, gate
   verdict, replay context, and proof bundle.
7. **Failure gate**: duplicate event, out-of-order event, missing traceparent,
   provider-card mismatch, unauthorized state patch, no heartbeat, and retry
   exhaustion all fail visibly.
8. **Boundary review gate**: no implementation slice may leave duplicate Run
   schemas, Poly-owned runtime semantics, daemon-local provider launch paths,
   fake handle state, or broad framework code not required by HTTP/MCP v1.

### Out of Scope

- Full command-center UX polish.
- Full dynamic worker registration.
- Full headless ACP parity across all adapters.
- Full remote mesh/BYOC execution.
- Signed/WORM compliance.
- Freezing daemon/gateway/event APIs before Union/Flyte and Orloj lessons are
  folded into the contracts.

## Constraints

- Surfaces are projections; they cannot own business/runtime semantics.
- SDK first: the public programmatic contract defines the operation, Poly binds
  it to surfaces, and daemon hosts long-lived MCP/HTTP surfaces.
- FlowMind is control/policy semantics; it is not the dispatcher daemon and not
  the worker substrate.
- Poly is not the HTTP/MCP runtime. Poly describes operations and projections;
  daemon owns served surface lifecycle.
- Operational state can be boring, but significant mutation must be
  evidence-backed and replayable enough for its proof level.
- Receipts prove execution happened; GateProofs prove gate claims; proof bundles
  assemble evidence for promotion.
- Local-first remains the default. Remote/cloud/BYOC execution must fit the same
  contracts later.
- UI must not be allowed to smuggle a second orchestration system into the
  product.

## Design Direction

### Core Idea

The product is not "run a prompt." The product is "turn intent into governed
work across a team."

Lev should present a simple, ambitious surface:

- define agents;
- run agents from anywhere;
- watch the fleet;
- teach them through gated memory/skill promotion;
- prove important outcomes.

Internally, every action becomes a Run. The Run Fabric is the narrow waist.

#### Lev Shape From The Corpus

The correct Lev synthesis is not "Airflow plus Orloj plus Fusion." It is:

```text
AgentDefinition / Function / TriggerSpec
  -> executable event admission
  -> ExecutionIntent
  -> RunEnvelope / Run
  -> Action / Attempt records
  -> Worker claim + SessionHandle
  -> substrate BackendDescriptor
  -> EvidenceRef stream
  -> Receipt + GateProof + replay packet
  -> projections: CLI / SDK / MCP / HTTP / AgentPing / now / dashboard
```

RunFusion adds a second dimension that the earlier Run Fabric draft did not
fully carry: every state transition also has a state class.

| State class | Lev examples | Coordination rule | Source pressure |
|---|---|---|---|
| append-only | LevEvent, receipt audit, trace frames, proof facts | Append, hash, replay, never mutate in place | RunFusion, Flyte watch, Gas Town events |
| strong | work claim, worker slot, cache reservation, provider lease, approval grant | Claim/lease/fence; reject stale owner | RunFusion, Orloj, Flyte, Symphony |
| queued | external event delivery, remote artifact fetch, offline mesh write | Preserve order, classify retry, expose pending/stale | RunFusion, Airflow, Flyte |
| local | tmux pane handle, browser instance, local process PID, live socket | Node-local; replicate only summaries/evidence | RunFusion, Mngr, Sandcastle |
| projection | dashboard row, now page, MCP config diff, command-center view | Rebuild from canonical state; never own execution | AionUI, Airflow UI, Union console |
| forbidden | auth material, raw provider tokens, unscoped secrets | No mesh/event replication; only scoped lease/audit | RunFusion caution, Union security, AgentGuard |

#### Consequence For Event Daemon And MVP

The daemon path should start from admission, not from flow matching. A passive
event can be appended as fact. An executable event is only executable after a
project/root adapter resolves it into a complete `ExecutionIntent` and then a
RunEnvelope with profile, adapter, model, variant, idempotency, retry policy,
lease requirements, and evidence expectations. If any of those cannot resolve,
the bus records a typed rejection or blocked/stale state and no worker runs.

That makes the MVP event path:

```text
LevEvent fact
  -> adapter/project TriggerSpec
  -> executable request
  -> admission resolver
  -> resolved RunEnvelope or typed rejection
  -> event log append
  -> daemon dispatch consumes only admitted RunEnvelope records
  -> Run/Action/Attempt lifecycle
  -> receipt/trace/replay/proof projection
```

This preserves the single local JSONL ordering property while preventing
ambiguous "a new flow appeared so the event now does something" behavior.

### Key Tenets

1. **Runs are first-class.** SDK calls, prompts, events, flows, tasks, UI
   actions, and MCP calls all create Runs through the same contract.
2. **Worker/Function/Trigger is the runtime algebra.** Everything else is
   vocabulary around those three nouns plus evidence.
3. **Surfaces are projections.** SDK is the canonical API; CLI, MCP, HTTP,
   Slack, cron, and UI are bindings into the fabric, not separate execution
   stacks.
4. **Boring state, hard proof.** Use DB/table/queue/file state when right, but
   mirror important transitions into events, receipts, traces, and proofs.
5. **Substrates are pluggable but governed.** Tmux, headless ACP, sandbox,
   browser, local SDK, and remote workers all need declared capabilities and
   evidence outputs.
6. **State class is explicit.** Append-only evidence, strong leases, queued
   deliveries, local handles, projections, and forbidden secrets follow
   different coordination rules.
7. **Learning is admitted.** Skills/memory can accumulate, but promotion must be
   reviewable, bounded, and reversible.
8. **The UI is a command center over facts.** It can be beautiful later because
   Phase 1 gives it stable run facts now.

## Alternatives Considered

### Option A: CLI-first `lev exec` product

- Strengths: fast to explain to engineers; uses existing surface.
- Weaknesses: too small; makes events/UI/SDK feel bolted on; invites CLI-shaped
  architecture.
- Rejected: `lev exec` should remain an ingress/projection, not the product.

### Option B: Dispatcher service as separate product spine

- Strengths: clear event-routing vocabulary.
- Weaknesses: creates a second execution system and awkward chains like
  "dispatch start then exec".
- Rejected: `lev dispatch start` should not exist. Dispatch is an internal
  event/trigger component that normalizes stimuli into ExecutionIntent and
  creates Runs through the SDK/Run contract.

### Option C: Run Fabric as narrow waist

- Strengths: absorbs CLI, SDK, UI, events, schedules, MCP, replay, and proof into
  one lifecycle; aligns with all major prior-art clusters.
- Weaknesses: requires disciplined vocabulary and more up-front contract work.
- Chosen: strongest route to "team agent infrastructure" without turning every
  surface into its own runtime.

## Architecture Lock

Phase 1 should be a **library-first contract spine with durable evidence**, not
a new daemon-first platform. A daemon, gateway, command center, queue, or remote
worker can come later, but none of them should define the MVP architecture.

The architectural center is the `Run` aggregate:

```text
ExecutionIntent
  -> admission + policy
  -> Run
  -> control plan
  -> substrate session / worker attempt
  -> EvidenceRef stream
  -> Receipt + GateProof + replay packet
```

### Ownership Boundaries

| Boundary | Owns | Must not own |
|---|---|---|
| `core/sdk` | Product-facing run API: `agent.run`, `flow.run`, handles, status/log/proof queries | Provider internals, CLI/MCP/HTTP-specific semantics |
| `core/exec` | `ExecutionIntent`, `Run`, `RunHandle`, admission, status, retry policy, evidence queries | Provider launch internals, tmux mechanics, UI workflows |
| FlowMind | Control graph semantics and node policy | Public ingress, daemon ownership, provider-specific fallback |
| Poly | PolyOp/Binding manifests and generated CLI/MCP/HTTP/UI projections | Runtime semantics, generic HTTP/MCP execution wrappers, provider-specific execution paths |
| daemon | Served MCP/HTTP surface lifecycle, real server handles, ports, health, auth, config, supervision | SDK contract ownership, Poly schema generation, provider/substrate internals |
| Harness / substrates | `SessionHandle`, worker execution, tmux/headless/browser/sandbox mechanics | Product-level run state or proof semantics |
| Evidence fabric | Logs, trace index, receipt records, GateProof refs, proof bundle assembly | Claiming semantic truth from receipt existence alone |
| Event/trigger layer | Event normalization, TriggerSpec match, schedule/webhook/replay ingress | A separate "dispatch then exec" product path |

### Quality Attribute Drivers

| Driver | Phase 1 measure |
|---|---|
| Surface parity | SDK, CLI projection, daemon-hosted MCP/HTTP, FlowMind, and event replay all create the same Run envelope fields |
| Operational truth | Every terminal Run has receipt, trace path, log/artifact refs, and explicit proof/gate status |
| Failure visibility | Duplicate events, missing capabilities, retry exhaustion, timeout, and no heartbeat fail closed with evidence |
| Local-first extensibility | The MVP works without cloud control plane, but leaves slots for remote workers, resources, secrets, leases, and approvals |
| Product velocity | UI and command center can iterate on stable Run APIs without redefining execution |

### Decision Rules

1. If a surface can launch a provider without creating a Run, it is a bug.
2. If a receipt is treated as semantic correctness, it is a bug.
3. If FlowMind, SDK, CLI, daemon-hosted MCP/HTTP, or replay produce different run envelopes,
   it is a bug.
4. If logs only exist as undocumented filesystem trivia, it is not product-ready.
5. If a new daemon API cannot represent Union/Flyte resources or Orloj-style
   workers, leases, approvals, and policies, do not freeze it.
6. If `daemon surfaces start` reports `running` without an actual supervised
   server handle and health path, it is a bug.
7. If MCP/HTTP Run behavior lives in a generic Poly wrapper instead of a
   daemon-hosted surface over Poly bindings, it is a bug.

## Phase Boundary

### Phase 1: Run Fabric MVP

Phase 1 is plumbing with a proof demo.

- AgentDefinition and ExecutionIntent vocabulary.
- SDK-first run API: `agent.run`, `flow.run`, status, wait, cancel, retry,
  logs, trace, receipt, and proof handles.
- RunHandle and SessionHandle contracts.
- TriggerSpec -> ExecutionIntent path.
- Run lifecycle: status, wait, cancel, retry, backoff, timeout, heartbeat.
- Logs, traces, receipts, GateProof refs, replay packet.
- Poly-generated bindings for CLI/MCP/HTTP/UI projections.
- Daemon SurfaceHost for real MCP/HTTP server lifecycle: start, stop, status,
  health, port, auth/config hooks, and Run evidence propagation.
- One self-managing SDLC proof: event -> run -> FlowMind/Poly -> execution ->
  receipt/trace/gate/proof.

Phase 1 should feel boring in implementation and powerful in consequence.

### Phase 2: Command Center + Live Platform

Phase 2 makes it feel like the product category killer.

- Agent builder and team command center.
- Fleet view: active runs, costs, quality, model mix, failures, A/B tests.
- Human review queues and approval surfaces.
- Dynamic worker registration and live external workers.
- OTel exporter and cross-worker trace propagation.
- Headless ACP parity and remote/BYOC execution.
- Rich AgentPing/Slack/channel integrations.
- Broader proof apps and Pentagon suites.

### Union / Orloj Daemon Freeze

"Daemon freeze" means freezing daemon/gateway/event-dispatch API contracts so
other modules and surfaces can build against them without churn.

Do not freeze those APIs until two inputs are represented:

- **Union/Flyte input**: task environments, resources, secrets, cache policy,
  reports, traces, app/run lifecycle, remote data-plane isolation, local-to-remote
  execution vocabulary.
- **Orloj input**: Agent, AgentSystem, Task, Worker, Tool, Memory, Policy,
  Approval, lease/claim lifecycle, tool authorization, schedule/webhook task
  factories, MCP/resource discovery.

Phase 1 does not need to implement full Union or Orloj. It must reserve the
right contract slots so daemon/gateway APIs do not freeze around a toy local
runner worldview.

## Risks and Unknowns

- **Vocabulary risk**: `exec`, `run`, `dispatch`, `flow`, `session`, and `agent`
  can blur. The design must keep Run as the durable aggregate and Session as the
  live substrate handle; `dispatch` is never a public start command.
- **Surface drift risk**: CLI/MCP/HTTP/UI could grow separate semantics if Poly
  bindings are not generated from one SDK/PolyOp manifest and daemon does not
  host served surfaces.
- **Proof theater risk**: receipts can be mistaken for semantic correctness.
  GateProof and proof bundle boundaries must stay explicit.
- **State dogma risk**: forcing everything into graph/event stream would be as
  bad as hiding everything in a DB. Use the right store and mirror evidence.
- **UX timing risk**: building command center too early could freeze weak runtime
  facts. Build the run facts first.
- **Daemon freeze risk**: freezing daemon/gateway APIs before Union/Orloj inputs
  would make Phase 2 remote/resource/approval features expensive to retrofit.

## Fitness Functions

1. **No surface-owned runtime logic**: static check that CLI/MCP/HTTP/UI handlers
   call the SDK/Run contract instead of launching providers directly.
2. **Intent parity**: golden tests show SDK, direct, event, schedule, daemon MCP,
   daemon HTTP, and replay intents produce the same Run envelope fields.
3. **Trace continuity**: every Run has run_id, correlation_id, traceparent or
   explicit no-trace reason, source intent, and artifact root.
4. **Receipt/GateProof split**: receipt validation must pass for structural
   execution records even when semantic gate verdict fails; proof bundle assembly
   must include GateProof for promoted claims.
5. **Retry observability**: retry exhaustion must emit visible terminal state,
   attempts, backoff schedule, last error class, and dead-letter/replay pointer.
6. **Substrate capability gate**: provider/session startup fails closed when the
   selected substrate lacks required capabilities or evidence outputs.
7. **Replay gate**: the SDLC proof demo must include a replay command and a
   deterministic proof packet enough to falsify the claimed path.
8. **Daemon surface truth gate**: `daemon surfaces start http|mcp` must either
   start real supervised server handles with health/status evidence or fail
   explicitly as `not_implemented`; intent-only `running` state is forbidden.

## Recommendation

Name the Phase 1 spine **Run Fabric** and make it the narrow waist of the MVP.

Build boring, evidence-backed plumbing first. Let every surface create, watch,
cancel, retry, replay, and prove Runs. Then build the command center and product
experience on top. This is how Lev becomes team agent infrastructure instead of
another agent demo.
