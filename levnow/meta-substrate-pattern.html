Declaration -> Compiler -> IR -> Runtime

A recurring Lev pattern, but not a universal religion.

April 2026 · core / plugins / specs sweep

PATTERN

The shape that keeps reappearing

The strongest subsystems share a four-layer structure, with an optional fifth layer for discovery or federation.

The pattern is simplest in plain English: authored truth becomes normalized meaning, then a stable intermediate form, then a runtime or projection layer consumes that stable form. FlowMind, Poly, orchestration, graph, and UI all already lean this way. The useful part is not that every module must have a named compiler package. The useful part is that serious subsystems stop being mushy when they can answer four questions honestly: what is the declaration, where is the lowering seam, what is the stable form, and who owns runtime.

flowchart TD
  A[Declaration\n.flow.yaml / config / DSL / schema] --> B[Compiler / Lowering / Normalization]
  B --> C[Stable Form / IR\nGraphDocument / PolyIR / LevUI IR / typed result]
  C --> D[Runtime / Projection / Execution\nexec / CLI / Stack / renderer / adapter]
  D --> E[Optional Discovery / Federation / Composition]

The reusable shape

FIT CLASSES

Where the pattern is strong, partial, or forced

The sweep only becomes useful if we name the exceptions.

✅

Strong fit

FlowMind, Orchestration, Poly, Graph, UI, Prompt-Stack, Validator, Timetravel, Browser Cascade, LevUI.

These own a real declaration surface, a real lowering seam, and a stable form that multiple consumers can reuse.

🟨

Partial fit

Memory, Config, Build, EventBus, Storage, Telemetry, Defendr, System Dashboard, Code Graph.

These usually have normalization and stable envelopes, but not always a first-class named IR or full compiler stage.

⛔

Non-fit

Domain, Logger, thin wrappers like Graph Adapters, and some runtime-only helpers.

Do not invent fake compilers or fake IRs just to satisfy the pattern. Some modules are contracts or utilities and should stay that way.

SCAFFOLDING

How to apply it going forward

This is the actionable doctrine for plugin scaffolding and dynamic generation.

design-checklist.txt
1. What is the declaration surface?
2. Where is the lowering or normalization seam?
3. What is the stable intermediate form?
4. Who owns runtime or projection?
5. Is there a discovery / federation layer above runtime?

If you cannot answer those honestly, the subsystem is probably still mushy.
If the answers are fake, do not force the pattern.
⚡

Immediate impact

For the current workstream, nothing changes about the next execution step: implement Poly protocol nouns, the Poly compiler seam, fixture goldens, and the CLI proof. This note is a doctrine lens, not a reason to reopen the Poly architecture.
