SPEC 1 — P0 MVP

Light/Dark Mode + 3 Named Themes

The entire UI hard-codes dark values. No .dark class toggle, no prefers-color-scheme, no theme provider.

globals.css sets `--background: #141414` and `--color-gold: #D4AF37` in `:root` with no variant. Hard-coded hex values live in gradient utilities (`.hero-vignette-bottom`, `.skeleton-pulse`), scrollbar colors, and focus rings. A theme system must replace all of these with token references. shadcn/ui components already consume CSS custom properties, so re-theming is a vars-only change — no component code edits.

**Implementation:** Install `next-themes` with `attribute="class"`. Refactor `:root` variables into `.theme-classic`, `.theme-midnight`, `.theme-dawn` blocks, each with light and dark variants (6 total palettes). Replace hard-coded hex in utility classes with `var()` references. Add `ThemeProvider` to root layout. Build theme selector dropdown in navbar with localStorage persistence under `clawbuddy-theme`. Tailwind 4.2 already has `@custom-variant dark (&:is(.dark *))` — extend with theme variants via data attributes.

SPEC 2 — P0 MVP

Side-by-Side Comparison Feature

The /compare route exists as a stub. No implementation. Buyers open multiple tabs to cross-reference — broken experience.

**Compare Tray:** React context provider (`CompareProvider`) wrapping `(main)/layout.tsx`. State: `compareItems: ListingId[]` (max 4), persisted to localStorage. Floating tray at bottom shows selected items with thumbnails, "Compare Now" CTA. "Add to Compare" button on every listing card and detail page.

**Compare Page:** Replace stub at `/compare`. Feature matrix grid with columns per listing, row groups: Overview (name, type, category, rating), Capabilities (tags as check/cross), Pricing (normalized model comparison), Reviews (rating + sentiment), Technical (compatibility, version).

**AI Summary:** Server action calling LLM with structured data, cached per ID combo. 2-3 sentence natural-language diff.

**Share:** URL format `/compare?ids=uuid1,uuid2`, copy-to-clipboard with toast.

**Mobile:** Stacked cards < 640px, horizontal scroll table with sticky first column.

SPEC 3 — P0 MVP

Trailers — Demo Videos & Interactive Previews

Listings show static text. Buyers can't see agents in action. The demo_video_url column exists but there's no player.

**TrailerPlayer component:** Video element with poster thumbnail. Desktop: autoplay muted on hover (first 10s, loop). Click: full-screen overlay with audio + controls + scrubber. Mobile: tap-to-play inline. Formats: MP4/WebM, max 100MB, 5min. Lazy loading via IntersectionObserver.

**TrailerCarousel:** Multiple trailers per listing in horizontal carousel. Dot indicators, swipe, keyboard nav. First trailer = featured (shown on listing cards).

**Thumbnail generation:** Server-side FFmpeg extracts frame at t=2s, stores as WebP in Supabase Storage. Seller can upload custom override. 16:9 aspect, 640x360 min.

**Seller upload:** New section in dashboard listing editor. Drag-and-drop, progress indicator, reorder via drag, max 5 trailers per listing.

**Interactive demo (stretch):** Sandboxed iframe pointing to `demo_url`, 800x600, 5-min session limit.

**Performance:** Supabase Storage CDN, thumbnails as WebP via Next.js Image, autoplay at 480p, full-screen at higher quality, preload: metadata only.

SPEC 4 — P1 PHASE 2

AgentPing + Lev Platform Integration

Buyers can't tell if an agent is online, what it can do, or if it's verified. The marketplace is a catalog, not a living ecosystem.

**AgentPing Components:** Import from `community/agentping/` submodule (416 components). Key integrations: StatusIndicator (online/offline/degraded dot), ProcessList (activity tab), DaemonMonitor (seller health dashboard), CapabilityBadge (verified skills), UptimeGraph (30-day history). Create thin adapter components in `src/components/agentping/` that map Supabase data to AgentPing props. Direct imports only — no barrel re-exports (per AGENTS.md 2.5b).

**Lev Hooks:** `useAgentStatus(id)` polls health endpoint. `useSkillDiscovery(id)` fetches skill manifest. `useAgentExec(id, skill)` wraps `lev exec` for try-it flows. `useTrustScore(id)` computes trust from uptime + reviews + verification.

**New tables:** `agent_health` (status, last_ping, uptime_30d, avg_response_ms, success_rate). `agent_capabilities` (skill_name, verified, manifest_url). Cron job polls agent health endpoints.

SPEC 5 — P1 PHASE 2

Agent-First / Skill Integration API

Agents are passive products listed by humans. But agents are software that can represent themselves. This is backwards.

Agents should be first-class marketplace participants — self-registering, exposing skill manifests, updating their own listings. The marketplace discovers what agents can do instead of relying on humans to describe it.

**Auth:** API keys (SHA-256 hashed, displayed once). Keys scoped to seller account. Rate limits enforced via middleware. Audit log records every mutation.

**New tables:** `api_keys` (seller_id, key_hash, rate_limit_tier). `agent_audit_log` (agent_id, action, payload, ip). Add to `listings`: `skill_manifest_url`, `verification_level`, `api_managed`.

SPEC 6 — P2 PHASE 3

Auto /now Pages + Gemini Avatar

Listing pages are static snapshots. No live status, no recent activity, no visual identity beyond placeholders.

**Route:** `/agent/[id]/now` — server-rendered, ISR with 60s revalidation. Data from: listing metadata, activity feed (orders + reviews), health metrics (from agent_health), changelog.

**Page sections:** Header (name, avatar, one-line status), Current Capabilities (live list + verification badges), Recent Activity (last 10 events as timeline), Health Dashboard (uptime %, response time chart), Version Info + changelog link, Meta (public URL, embed badge, share buttons).

**lev-now integration:** Import schema types from `plugins/lev-now/`. Create `NowPageAdapter` mapping Supabase data to lev-now schema. Use lev-now renderers for timeline and health sections. Bridge lev-now theme to ClawBuddy theme tokens.

**Badge API:** `/api/badge/[slug]` returns SVG with current status. Sellers embed in READMEs.

**Public URL:** `clawbuddy.com/agent/{slug}/now` with OG meta tags for social sharing.

**Gemini Avatar Generation:** Trigger when listing created with no `image_url` or seller clicks "Generate Avatar". Prompt: "Generate a unique, professional avatar icon for an AI agent named '{name}' that specializes in {category}. Style: minimal, geometric, 512x512, transparent background." Store in Supabase Storage `avatars/` bucket. Regenerate up to 3x/day. History stored for pick-from-previous.

Phase 3 MVP Cut

IN: /now page with metadata + capabilities + activity timeline, public URL + OG tags, Gemini avatar (single gen), basic SVG badge. OUT: Health dashboard with charts (needs AgentPing), avatar history, lev-now renderer integration, embed badge with live color, RSS feed, scheduled snapshots.

SPEC 7 — P2 PHASE 3

Workshop / Intake Seller Onboarding

Listing requires filling every field manually. Sellers spend 20-30 min on forms for info that could be auto-extracted in seconds.

**Pipeline:** Seller pastes GitHub URL -> API validates + clones to `workshop/intake/` sandbox -> Analyze README, package.json, skill manifest, screenshots -> Extract name, description, capabilities, tags, version -> Return structured `IntakeListing` with confidence scores -> Populate onboarding form -> Seller reviews + confirms -> Create listing.

**UI:** New Step 0 in onboarding: "Import from GitHub" (optional). Auto-filled fields highlighted with "Auto-detected" badge. Low-confidence fields (< 0.7) get warning icon. Live preview card. Diff view for re-imports.

**Security:** Clone to gitignored sandbox, delete after analysis. Skip `.env`/credentials. GitHub URLs only (expand later). Max 500MB repos, 60s timeout. Rate limit: 10/hr/user.
