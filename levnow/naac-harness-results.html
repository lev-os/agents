INSIGHT

The FMM Reject Cascade

The dominant failure is a cascade: intent parser correctly identifies the instruction type but extracts wrong or incomplete parameters. FMM's fail-closed `_match_rule` correctly rejects because parameters don't match the scenario fixture rule. No FMM accept means no readback generated.

This is the **anti-cheat design working as intended**. Before session 30, the contractor's intent-only fallback would have accepted these — inflating the pass rate to 98%. The 49.4% is the honest number.

**The fix is not in the harness.** It's in the pipeline: better parameter extraction, scenario rule alignment, and compound splitter coverage.

DIAGNOSIS

Root Cause Analysis

FMM Reject Cascade (37 turns)

Intent is classified correctly but parameters are wrong or incomplete. FMM's fail-closed _match_rule correctly rejects. Fix: DeBERTa slot extractor fine-tuning, WFST parameter grammar, scenario rule alignment.

Intent Misclassification (11 turns)

Notable patterns: direct_to vs route_amendment (3), landing_clearance misclassed (2), acknowledgment vs emergency_declaration (1), ifr_clearance missed (1). Fix: boundary model retraining, intent taxonomy refinement.
