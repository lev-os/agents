chiDEV Middleware Prior Art and Branch Review

What the middleware idea was, what exists now, what pr86 and pr87 still add, and where retained ideas should live.

Repo: leviathan | Date: 2026-04-10

The core answer is: yes, middleware exists in current main, but the repo canon explicitly treats it as a thin transitional runtime wrapper. FlowMind, hooks, LevEvent, and orchestration remain the real ownership surfaces. The right question for the old chiDEV branches is therefore not whether to merge middleware, but which ideas or examples are still worth preserving as references.

PROVENANCE

What the prior art says about middleware vs hooks / FlowMind

The repo contains both the experiment and the correction. The DeerFlow branch work introduced a middleware chain model, but the canonical docs later constrain that surface and explicitly move declarative policy back to FlowMind and orchestration. There is also a later freeze note that says Poly owns the WHAT and FlowMind owns the WHEN.

SourceStatusFindingWhy It Matters

docs/specs/spec-harness.mdVerifiedFlowMind owns context composition, lifecycle routing, loop policy, and guard semantics. Harness owns transport/runtime glue.This is the broader owner map, not just the middleware-specific one.

Plain-English answer

You were remembering the decision correctly. The repo still has middleware code, but the repo no longer treats middleware as the place where policy truth should live.

LIVE REALITY

Do we already have middleware?

Yes. Main has a real middleware chain and concrete modules under core/harness. What matters is how we classify them. The repo already classifies some as enabled runtime-only helpers and others as disabled policy-ish experiments.

GRAPH DIRECTION

What the branch/worktree warm-cache and graph direction already has

There is real repo prior art for a graph-backed or graph-plus-vector direction. The strongest pattern is not “put all markdown into one giant vector store and stop.” The stronger direction is graph-backed primary structure with vectorized markdown summaries and/or chunks as secondary retrieval aids.

AreaEvidenceImplication
Graph + vector hybridworkshop/papers/INTEGRATION-ROADMAP-2026-01-21.md and workshop/pocs/graph-storage-adapters/*The repo already leans toward hybrid retrieval, not vector-only.
Harvester as graph producergraph-draft.jsonl output in Harvester runsHarvester already emits graph-facing artifacts that could back prior-art lookup.
Markdown storageMultiple workshop analyses and graph-storage adapter notesMarkdown should likely stay filesystem-backed, with graph nodes carrying summaries, refs, hashes, and optionally embeddings/chunk projections.
Warm branch/worktree cacheCurrent repo has many worktrees and branch-local archaeology pain; no clean branch cache product surface exists yet.A warm cache of branch/worktree graph projections would make prior-art queries much cheaper than brute-force grep across every ref.
