R7 EvaluatorFlowParser is a domain port, not an allowlist widening high reversal cost

The S1 worker stopped correctly on a real contract conflict: execution.yaml:234 demands eval parse flow_ref through the REAL FlowMind parser, while canon pins eval to allow:[core/domain]. Ruled: domain declares an EvaluatorFlowParser port; eval's registry admission REQUIRES the injected port and hard-errors at load time; core/flowmind exports the real-parser adapter. Same inversion applied to cap.embed (embedding port, provider adapter in flowmind).

This made ports 4 and 5 of the cutover's inversion idiom, after RunLedgerWriter, EffectfulPort, verdictRefResolver, and MeasurementStreamWriter. flowmind already legally imports domain, so the adapter adds zero new edges. The 'REAL parser' contract intent is preserved at composition time — admission's load-time hard-error contract is about WHEN parsing must succeed, not WHO imports the parser. Widening eval's allowlist was rejected because flowmind imports orchestration+effect: the brain would depend on the control plane, a tier inversion. Moving the parser into eval was rejected because it drags orchestration/effect parsing concerns into the truth-decider. xhigh round-1 review CONFIRMED the inversion. Reversing now reworks landed slice-4 registry admission, the flowmind adapter, and the integration proof.

N1 Seal-misclassification: a transport-failed run got a healthy seal new · truth-integrity

When the first S2 dispatch died at codex boot (exit 1), the run was sealed outcome: candidate_effect, command_success: true with an intent_conservation verdict_ref — a failed transport run got a healthy truth record in .lev/runs/runs.jsonl. The exec success-detection path (success ? runOutcome.class : 'failed') computed success=true despite the nonzero exit. P1 bug filed; fix currently HELD.

This is one-brain-adjacent: the RunSeal IS the run's truth record, and the caller (exec) is the classifier per the seal doctrine. The defect is in exec's success detection, not the ledger — the ledger faithfully recorded what it was told. Severity hinges on blast radius: the mis-seal landed on a throwaway dispatch with no downstream consumer, and the seal doctrine keeps outcome caller-classified (the ledger didn't invent a truth bit). But 'a failed run looks successful in the run ledger' is exactly the false-green class the cutover exists to kill. Timing: the original hold reason was that S2's certify sweep rewrites the exec/run classify files (collision risk); S2 has now landed, so the collision window is closing. The controller lean: treat as P1 (real, not corrupting committed truth) and fix as a dedicated follow-up right after S2 closes — keeping the S2 rework scoped to its four review violations rather than folding an unrelated classify fix into it.

N3 Characterization-test deletion precedent for cutover slices new · methodology

S2's pre-cut characterization tests asserted that the legacy certify/shadow-brain branch EXISTS — which the cutover then deletes, making them impossible to keep green. The closure reviewer ruled deletion acceptable in principle under invariant 12, BUT that an equivalent post-cut lock is owed in the same package (S2's reject partly turned on the harness side having no replacement lock). This pattern recurs in every hard-cutover slice.

The clean rule to ratify: a characterization test that locks a soon-to-be-deleted branch may be deleted in the same commit that removes its subject — provided the same package gains a post-cut lock asserting the NEW behavior (absence of the old vocabulary / presence of the typed replacement). The git-log ancestry still proves the pre-cut behavior existed and passed before the cut. Same decision class as R6 (what proof a node owes at close). Alternative: require char tests to always survive as renames (S1's precedent) — cleaner ancestry but forces awkward 'assert the thing is gone' rewrites. Or leave it case-by-case per reviewer, which is what produced the S2 ambiguity.
