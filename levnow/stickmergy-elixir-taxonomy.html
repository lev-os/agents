MODULE TAXONOMY

The Elixir Mapping

Every core/ module has an OTP counterpart. Here's the Rosetta Stone.

🔷

graph → GenServer + Registry + PubSub

Holds entity state, dispatches events, registers flows. The FlowRegistry bridges GenServer state management with PubSub event distribution.

⚙️

flowmind → Application + Compiler

Loads YAML IR, compiles to executable graph operations, manages sessions. The Application behavior spec that boots the system's brain.

🔀

poly → Registry + Supervisor dispatch

Named handler lookup, surface routing (CLI / MCP / HTTP / gRPC / WS). RENAME CANDIDATE — it's really the handler registry + surface multiplexer.

🎯

orchestration → Supervisor + Task + GenStage

DAG waves, queues, execution contracts, loop machinery. The heavy-duty process orchestration layer.

📐

config → Application.get_env

Fractal cascading config resolution. Reads from DNA, project, user, and global scopes — just like Application.get_env with nested fallback.

🧪

harness → Supervisor + DynamicSupervisor

Process lifecycle, agent sessions, multi-provider dispatch. DynamicSupervisor because agent sessions are started and stopped at runtime.

🧠

memory → Registry + GenServer facade

Backend registry, hybrid orchestrator. Routes to the right memory backend (episodic, procedural, semantic) through a unified GenServer facade.

📡

event-bus → Phoenix.PubSub

Event routing. Topic-based publish/subscribe with the same semantics as Phoenix.PubSub — fire, subscribe, broadcast.

🤖

afk → Supervisor with max_restarts kill gates

Autonomous run supervision. Like a Supervisor that counts restarts and kills the child permanently when it crosses the threshold.

🌳

daemon → Application / Supervisor root

Daemon lifecycle, startup tree. The root of the supervision tree — Application.start that boots every other process.

NODE ARCHITECTURE

The Rich Node Model

Each graph node accumulates facets over time. Here's what sticks — and who reads it.

DNA

Constraint definitions, behaviors, failure modes, validation rules. Written by architects, read by every runtime process.

Config

Resolved configuration values from fractal cascade. Written by config resolver, read by handlers and orchestration.

Hooks

Lifecycle callbacks — pre/post execution, validation gates, side-effect triggers. Written by plugin authors, read by harness.

Exec History

Run records, pass/fail, timing, resource usage. Written by orchestration, read by FlowMind and dashboards.

Agent Trace

Which agent touched this node, when, what it did. Written by harness sessions, read by coordination and audit.

Bulletins

Graph-addressed mail with TTL decay and cascade radius. Written by any agent, read by agents traversing the region. NEW.

Manifestation

Final rendered output — the artifact this node produces. Written by surface renderers, read by users and downstream consumers.

MAIL ROUTING

Three Addressing Modes

From point-to-point to topological proximity — three ways to route information.

📬

Direct

to: agent://A3 — Point-to-point delivery. Only A3 sees it. Standard levmail behavior today.

📢

Broadcast

*to: graph:// cascade: inf** — Everyone sees it. Current all-agents broadcast. No topology awareness.

📌

Bulletin

to: graph://<node> cascade: N — Topological proximity with bounded spread. Agents within N hops encounter the message. NEW.

DNA ↔ OTP

DNA as Supervision Specs

DNA constraint files map directly to OTP child_spec fields — fault tolerance, observability, and restart policy.

validationTelemetry / health checksObservability
TOPOLOGY

FlowMind Zone Heat Map

Bulletin density and agent activity create four zone types across the graph.

🔴

Hot Zone

Many recent bulletins, multiple agents active. Action: Coordinate before touching — high collision risk.

🔵

Cold Zone

No bulletins, no claims. The output of findGaps(). Action: Unexplored territory — safe to claim and start fresh.

⚡

Conflict Zone

Contradictory bulletins on adjacent nodes. Action: Route to CDO / architecture review before proceeding.

🟢

Settled Zone

Old bulletins, consistent, low activity. Action: Stable ground — safe to build on without coordination overhead.

UPGRADE PATH

levmail Upgrade Path

Three layers — from zero-code-change adoption to full graph-backed resolution.

Layer 1 — Zero Code Change

Use graph:// strings in the to: field now. The file format already holds it. Agents and humans can start writing bulletin-addressed mail today with no tooling changes.

Layer 2 — ~30 Lines Bash

levmail bulletin <node> --cascade 2 --ttl 48h — Validates the graph:// prefix, adds frontmatter fields (cascade, TTL, posted-at). A thin wrapper, not a rewrite.

Layer 3 — Stigmergy

Full graph-backed resolution: cascade hops, agent-claim registry, TTL decay, zone heat maps. Depends on S4 EventBridge wiring. The complete topological memory system. Design doc →

PRIOR ART

Precedent

Four decades of indirect coordination research that converge here.

🐜

Stigmergy (Grassé, 1959)

Ant pheromone trails. Indirect coordination through environmental modification. Agents don't talk to each other — they modify the shared environment and others react. Read →

📋

Blackboard Architecture (Hayes-Roth, 1985)

Shared workspace, independent knowledge sources post partial solutions. No direct communication — all coordination flows through the board. Read →)

📦

Tuple Spaces (Gelernter, 1985)

Linda coordination language. Processes communicate by writing and reading tuples in a shared associative space. Decoupled in time and space. Read →

⚡

Elixir/OTP (Erlang, 1986+)

Lightweight processes, message passing, supervision trees, let-it-crash philosophy. The production-proven realization of actor-model fault tolerance. Read →
