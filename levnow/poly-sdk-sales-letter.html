If your team is tired of “it worked in the demo thread,” you need the same thing production software always needed: **routing you can explain**, **handlers you can swap**, and **contracts that don’t silently fork**.

WHY THIS MATTERS

Chat is alignment. Execution is the product.

Leviathan is built for teams who want receipts: declarative surfaces (CLI, MCP, HTTP, gRPC, and more), graph-aware work, and a harness that treats `lev exec` as first-class—not a bolt-on script.

Most “agent platforms” optimize for the message UI. Leviathan optimizes for the **dispatch loop**: who is allowed to run, how overload is handled, what events get emitted, and how capability is described as data (cards, topology, manifests) instead of tribal knowledge.

**Poly** is the SDK-shaped entry to that loop: one `call(provider, method, ctx)` that resolves through a cached registry and lands on the right transport—HTTP daemons, local binaries, or registered `lev://` handlers—without turning your repo into a maze of one-off integrations.

THREE PILLARS

What “runtime-grade” looks like here

Concrete mechanisms, not slogans.

🔀

Registry-first routing

Load registry-runtime.yaml / registry.yaml, match daemons (HTTP/S when healthy), fall back to binaries, then runners keyed as provider:method. Same mental model as internal service discovery—because agents are services.

📜

DNA-locked exec contract

exec-contract defines how @lev-os/exec behaves: bounded concurrency, backpressure, event sequences (exec.started, exec.completed, exec.failed), and a hard line—no sneaking orchestration into exec. Adapters stay honest; the control plane stays separate.

🧭

Surfaces, not a default chat protocol

Domain types describe capabilities and surfaces (CLI, MCP, HTTP, gRPC, …). The product story is “route work to the right interface,” not “everything is a thread.” Chat overlays exist where they help; they don’t define the architecture.

UNDER THE HOOD

The Poly dispatch order (the part buyers audit)

Fast path → cold path → codegen handler path.

1

Daemons

If a provider maps to a healthy HTTP/HTTPS daemon, call across the wire with executeHandler — same primitive the SDK uses for remote surfaces.

2

Binaries

On failure (or when configured), fall back to registered binaries — spawned processes with structured args, not shell improvisation.

3

Runners

Registry runners resolve provider:method to handlers — lev:// modules dynamically imported, or bin:// paths with manifest-mapped flags.

CLARITY

Two URI worlds—on purpose

Strict route objects vs. filesystem-backed handler modules.

Route.parse (lev://module/fractal/item)

The domain Route type enforces a three-segment shape for structured addressing—great for graph-native links and tooling.

Handler executor (lev:// → import path)

The codegen executor resolves lev:// to workspace-relative modules (plugins/, core/, packages/) and invokes modulemethod. When you need both, document the hand-off—Leviathan’s codebase is honest that these segment rules differ today.
