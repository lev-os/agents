From PolyIR To Architecture Ontology

The full arc of the conversation: freeze the Poly substrate, separate authoring from runtime, discover the compiled-declaration spine, then widen into a many-to-many architecture ontology without collapsing the system into one family.

April 2026 · Poly, FlowMind, Stack, OMX, ontology, CDO, skills corpus

The session started as a FlowMind and Poly design problem, turned into a compiled-substrate problem, then escalated into a broader question: what recurring architecture shape is Lev actually discovering? The final result is two layers of truth. First, a concrete Poly implementation path with compiled IR, derived runtime, and proof-sequence-first execution. Second, a broader architecture ontology where **Compiled Declaration Architecture** is only one family inside a larger many-to-many map of planes, families, layers, and failure modes.

✅ Bottom line

Poly architecture is ready for implementation. The bigger discovery is not a replacement architecture for the whole repo, but a scoped family plus a richer ontology. The final split is: Poly as compiled capability contract, FlowMind as workflow/program layer, lev exec / Stack as runtimes, and Architecture Ontology as the analysis/reconstruction layer above them.

**Frozen implementation truth:** authored roots -> compileToPolyIR() -> deriveRegistryRuntime() -> CLI / Stack / FlowMind / surfaces. **Frozen runtime truth:** Stack owns steering first, FlowMind owns lowering later, receipts close runtime loops, topology stays outside Poly core. **Frozen ontology truth:** Compiled Declaration Architecture is real, but only as one family in a larger architecture ontology. Logger, the LevEvent spine, projections, and federation are all important, but they are not the same family.

FULL ARC

The conversation decompiled into phases

This is the whole session, compressed into the decision sequence that actually changed the architecture.

Phase 1

FlowMind help-text problem becomes declaration-contract problem

The first correction was to stop treating the failure as generic lev help UX. The diagnosis moved down a layer: the invoked thing did not carry enough docs, validation, steering, and remediation semantics.

Phase 2

Poly source-of-truth strategy converges

The architecture settled on authored roots remaining canonical, compiled PolyIR as the semantic core, RegistryRuntime as a derived operational projection, and core/poly as consumer only.

Phase 3

poly-protocol and poly-compiler boundaries freeze

The repo converged on core/domain/src/poly-protocol/* for nouns, core/poly-compiler/* for compile and derive, and one-way shims for arm.ts and invoke-envelope.ts.

Phase 4

Static packet vs runtime packet split

The design separated static render packets derived from RegistryRuntime from Stack-emitted runtime authoring packets, preventing a second semantic system from appearing by accident.

Phase 5

Oracle contradiction cleanup

Small but concrete corrections landed: resume-safe Task linkage, union-by-id semantic profile merge, journey_projection, ordered Tick, stable non-file adapter IDs and digests, and tighter OMX boundary language.

Phase 6

Implementation planning, not more architecture

The architecture pass closed and became a proof-sequence-first implementation plan: poly-protocol, poly-compiler, fixtures, lev poly build|compile|derive, CLI proof, then Stack-backed runtime proof.

Phase 7

Meta-substrate discovery

A repo-wide sweep across core, plugins, and specs showed a recurring declaration -> lowering -> stable form -> runtime pattern, strongest in the FlowMind/Orchestration/Poly/Graph/UI spine.

Phase 8

CDO reframes the doctrine

The pattern survived adversarial testing, but only as a spine-scoped architecture family. The result: Compiled Declaration Architecture externally, compiled-declaration spine internally, lens discipline for governance.

Phase 9

Many-to-many ontology replaces one-family fit thinking

The conversation moved beyond 'does it fit the spine?' into planes, families, layer axes, and failure classes. Runtime became an axis that applies widely; architecture family membership stopped being treated as singular.

Phase 10

OMX/OMC boundary clarified

tmux panes, worktrees, worker counts, merge/retry/resume policies, and provider scheduling stayed in execution topology. Poly kept mailbox, task, report, heartbeat, summary, and receipt-shaped contracts at the adapter boundary.

Phase 11

Architecture ontology / ArchitectureIR question emerges

The GPT synthesis pushed one level higher: primitives, meta-structures, architecture families, and patch algebra. That reframed the missing layer as ArchitectureIR: a reconstruction and planning IR above subsystem-native IRs.

What changed the most

The deepest pivot was from feature design to ontology discipline. First the session froze a real compiler-shaped architecture for Poly. Then it realized that the same shape is only one member of a broader architectural ecology. That is why the later half of the conversation is less about commands and more about families, planes, projection discipline, and patch operators.

⚠️ The big correction

Runtime is not the deciding question. Nearly everything has a runtime. The useful question is: which planes, which family mix, which stable form, and which failure modes does the subsystem actually participate in?

**ArchitectureIR** only stays useful if it remains an analysis / reconstruction / planning IR. It collapses the moment it tries to replace Graph ontology, execution ontology, protocol ontology, or authored truth. The safe framing is: system reality -> architecture decompiler -> ArchitectureIR -> family classifier -> tradeoff atlas -> patch algebra -> refactor or extension plan.

OMX / OMC sidequest

The OMX thread mattered because it proved a real runtime shell boundary. Bootstrap was not fully headless, but post-launch control was machine-drivable. The useful outcome was not 'adopt OMX semantics as core.' It was the sharper split: topology and swarm mechanics stay outside Poly core, while mailbox, event, summary, monitor, and receipt-shaped adapter contracts can still surface as runtime ops.

Skill / research lane

The architecture ontology conversation spawned a parallel research lane. That lane corrected a wrong move too: instead of inventing a new engineering hub, it fell back to the existing nested skills-db hub structure, preserved external framework excerpts and GPT synthesis as research artifacts, and kept active skill mutation out of the architecture kernel discussion.

BOTTOM COMPARISON

GPT’s analysis vs my analysis

This is the comparison section you asked for at the bottom: where GPT was freakishly close, where it pushed the discussion forward, and where I would still tighten the framing.

GPT’s strongest contributions

GPT correctly identified the kernel as boring and small: capability / invoke / task / frame / receipt / gate / projection. It also pushed the conversation one level higher by articulating ArchitectureIR, architecture primitives, meta-structures, family signatures, and patch operators. The best parts were the primitive list, the meta-structures, the family table, and the patch algebra for refactor/extension.

My strongest corrections and constraints

I kept forcing scope discipline: do not widen Poly core, do not let topology semantics leak into substrate nouns, do not treat runtime as a family discriminator, and do not let ArchitectureIR replace subsystem-native ontologies. The key correction is that ArchitectureIR should be a reconstruction / analysis / planning IR, not a new root ontology or runtime substrate.

3 NEXT STEPS

What to do next

Exactly three, and they follow the sequence implied by the whole conversation.

1

. Ship the Poly first slice

Stop expanding architecture and execute the frozen plan: create core/domain/src/poly-protocol/, create core/poly-compiler/, add fixture roots + goldens, wire lev poly build|compile|derive, and prove author_flowmind through the existing Stack path.

2

. Write ArchitectureIR v0

Create one focused design note for ArchitectureIR as an analysis / reconstruction / patch-planning IR. Keep it above Graph ontology, execution ontology, and protocol ontology. Start with primitives, meta-structures, family signatures, patch operators, and evidence_refs.

3

. Build the architecture decompiler

Run the first many-to-many classifier on a small set: FlowMind, Poly, Event Bus, Logger, and one runtime shell. Output planes, families, layer axes, failure modes, and patch candidates. Use that to replace the remaining spine-centric comparisons with a real architecture ontology workflow.
