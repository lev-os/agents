THE THESIS

The Reactive Graph IS the AI

FlowMinds listen. Loops manifest. Evals validate. Workstreams persist.

You update a node. You make a proposal. The system accepts the proposal. The graph patches itself. A loop manifests the update. Connected nodes trickle-update through the mesh. The loop continues until the change is real. This is not a special system. It's just how graphs work.

This applies to **everything**:

A developer changes a button → a loop manifests the change on the website. A blog node detects no article for a week → auto-researches → publishes. A new feature lands → related docs auto-update → a blog post auto-writes about it. Training metrics improve → model auto-checkpoints → deployment auto-triggers.

The reactive graph is the unifying abstraction. Not agents. Not sessions. Not swarms. The graph. Sessions are just segments with compaction artifacts. Agents are just execution slots. Swarms are just concurrent loops. The graph is the thing that persists, evolves, and manifests.

THE PARADIGM SHIFT

Why Agents Are Dead

The industry picked the wrong abstraction. Here's why it doesn't survive contact with production.

Every AI framework in 2024-2025 converged on the same idea: give an LLM a persona, tools, and memory, and call it an "agent." Then build orchestration to coordinate multiple agents. Then build meta-orchestration to coordinate the orchestrators.

This is object-oriented thinking applied to a graph problem. You don't need Agent objects with inheritance hierarchies. You need a graph algebra with composition rules.

A "coding agent" is `Loop(heartbeat) + Eval(L1+L2+L3) + Session(accumulated) + Effect(git commit)`. A "research agent" is `Loop(metric-exploratory) + Eval(metric|judge) + Session(accumulated) + Effect(publish)`. The behaviors are identical at the primitive level. The only difference is the composition.

**Prompts converge. Harnesses differentiate.** And harnesses are just graph compositions.

THE EXECUTION ALGEBRA

Seven Primitives. Arbitrary Behavior.

Three L0 generators (Node, Loop, Eval) and four L1 derived primitives (Edge, Session, Effect, Intent). Every composition in the system reduces to these seven.

Node

L0 Generator · 12% of actions

The atomic unit of work. Everything is a node — a task, a file, a config, a gate, a subgraph. Four irreducible node types: agent (has prompt), function (has action), gate (has eval + branches), subgraph (has nested graph). Property-inferred, not class-inherited.

Edge

L1 Derived

Runtime transition relation connecting compiled nodes, scopes, or subgraphs. Two species: StructuralEdge (compile-time topology) and Transition (runtime guards). Carries 5 concerns that must be factored cleanly or the algebra silently fractures.

Loop

L0 Generator · 1.3% of actions

Bounded repetition primitive. Owns iteration state, exit semantics, and retry/eval cadence. Three canonical topologies: Ralph (spec-gated), Autoresearch (metric-exploratory), Until (semantic NL exit). Low action frequency because loops are structural — one loop drives thousands of node executions.

Eval

L0 Generator · 42% of actions

The judgment primitive. Decides branch, continue, terminate, or escalate. Three layers: L1 self-eval (DNA validation, free), L2 trajectory (behavioral ticks, cheap), L3 quality (LLM-as-judge, expensive). The most frequently invoked primitive by a factor of 3x.

Session

L1 Derived

Bounded execution window within a workstream. Owns runtime continuity, not PM identity. Sessions are segments with compaction artifacts — they exist so the graph can forget safely. Status follows LevPhase: pending, active, suspended, terminal.

Effect

L1 Derived · 31% of actions

Declared side-effect: git commit, file write, API call, model update, graph mutation. Observable and attributable. Unlike tool calls in agent frameworks, effects are first-class — they have types, they produce receipts, and they're gated before execution.

Intent

L1 Derived · 14% of actions

Cascading behavior spec. Governs child entities via governs links. Hard constraints are additive (never relaxed by children), steering constraints are overridable. Validated by every formalism's governance gap. The constraint propagation primitive.

COMPOSITION

Complex Behavior From Simple Algebra

Every system in the Leviathan ecosystem is a formula over seven primitives. No exceptions.

  formula: Loop(curriculum) + Eval(reward|metric) + Session(checkpoint) + Effect(model update)

  formula: Loop(attempt) + Eval(success rate) + Session(theory) + Effect(strategy mutation)

Notice the pattern. Every composition is `Loop + Eval + Session + Effect` with different parameters. The "agent" disappears. What remains is a graph node with a loop topology, an eval policy, a session boundary, and a set of declared effects.

**"autoresearch" is a marketing surface. The primitive is: theory → flowmind → heartbeat.** One engine — `core-autoresearch` — runs everything. The compositions above are just execution profiles over that engine.

This is not a theoretical claim. The algebra was validated empirically on 78 actions across three codebases (Lev, Sofia, os/) during CDO Round 2. The seven primitives account for 100% of observed execution behaviors.

EXECUTION MODEL

Workstreams, Not Sessions

You spawn workstreams, not agents. The graph does the rest.

A workstream is the graph's unit of sustained intent. It persists across sessions, accumulates state, and self-heals when interrupted. Sessions are just bounded execution windows within a workstream — segments with compaction artifacts.

When you "start a coding task" in Leviathan, you're not launching an agent. You're patching the graph: adding a workstream node, connecting it to intent nodes (constraints), attaching loop nodes (execution strategy), and declaring effect nodes (what it's allowed to do). The graph topology IS the orchestration. No coordinator agent needed.

Graph patches replace agent orchestration. You don't tell Agent A to call Agent B. You connect Node A to Node B with an edge. The tick pipeline handles the rest.

THE TICK

Every Message Is a Tick

Seven phases. No exceptions. No shortcuts. The tick pipeline is the heartbeat of the reactive graph.

1

. INGEST

Accept input. Parse. Validate schema. Attach provenance. Every message enters the system through this gate.

2

. OBSERVE

Read graph state. Gather context. Load relevant nodes, claims, evidence. Build the observation window.

3

. PROPOSE

Generate candidate graph mutations. Every change is a proposal — a pull request against graph state. Operations list, not imperative code.

4

. GATE

Run the proposal through the gate pipeline: scope, safety, feasibility, standards, progress. Binary pass/fail. No negotiation. Gates are the loss function.

5

. APPLY

Accepted proposals mutate graph state through the compositor: extract → classify → validate → apply. Append-only. No silent rollbacks.

6

. UPDATE

Trickle-update connected nodes. Propagate state changes through edges. Recalculate affected views and projections.

7

. EMIT

Produce observable effects. Git commits, file writes, API calls, UI updates. Every effect is declared, typed, and receipted.

PROGRESSIVE DISCLOSURE

The Depth Model: L0 – L3

The graph reveals itself at the depth you need. Overview first, runtime last. Never force the user to see more than they asked for.

L0 — Overview

What is this?

The 30-second answer. Entity names, top-level status, high-level intent. What a stakeholder sees. What the /now page shows.

L1 — Structure

Components, boundaries, relationships.

The architecture view. Which nodes connect to which. What owns what. Dependency graph. Module boundaries. The view an architect needs.

L2 — Details

Implementation, specifics, how things work.

Type signatures. Schema shapes. Eval criteria. Gate configurations. The view an implementer needs to build or fix something.

L3 — Runtime

Behavior, state transitions, live data.

Tick-by-tick execution traces. Session state. Loop counters. Eval results. Effect receipts. The view you need when something is broken at 3am.

CDO IS COGNITION

Skills Are Weights. Agents Are Neurons.

Synthesis is the forward pass. Eval is backprop. CDO is not a tool. CDO is cognition.

The CDO you are running RIGHT NOW is a composition of these primitives. You are a neural net thinking in real time. Skills are weights. Agents are neurons. Synthesis is the forward pass. Eval is backprop.

Multi-agent deliberation (CDO) is the clearest proof that "agents" are the wrong unit. A CDO session is `Loop(adaptive turns) + Eval(synthesis) + Session(accumulated) + Effect(FINAL.md)`. Three or more isolated agents generate divergent perspectives. A synthesis agent merges them. An eval pass checks convergence.

This is literally a neural network forward pass: multiple neurons (agents) fire, their outputs are weighted (synthesis), and the loss function (eval) determines whether to continue or terminate. Backpropagation is the eval signal flowing backward to reshape the next round of deliberation.

CDO is not a collaboration tool. It's the graph thinking out loud.

CONTROL THEORY

Gates = Loss Function

Whoever controls gates controls what the graph learns. Gate design IS reward signal design.

Gates = Loss Function. Whoever controls gates controls what the graph learns. .lev/validation-gates.yaml is not a safety document — it's the training objective.

This was Breakthrough B1 from CDO Round 1, scored 9/10 — the highest-confidence insight in the entire deliberation.

Every accepted proposal is a policy gradient step. The policy IS the environment. Modifying graph topology modifies decision-making. The gate pipeline determines which proposals survive — and therefore which direction the graph evolves.

This is not a metaphor. It's a formal mapping: states = graph snapshots, actions = proposals, transitions = compositor pipeline, reward = truth_state promotion. The graph implements Generalized Policy Iteration (Sutton & Barto) without naming it. Three codebases (Lev, Sofia, os/) all exhibit this pattern independently. The convergence proof already exists.

**Bounded speculation under constraint** — that's the unnamed GPI pattern. Every autoresearch loop, every CDO deliberation, every TELO cycle is bounded speculation converging under gate constraints.

Five Gate Types

The loss function has five dimensions. Each gate type enforces a different aspect of graph health.

Scope Gate

Is this within project/workstream boundary? Prevents graph mutations from bleeding across ownership lines. The locality constraint (C5) enforced at runtime.

Safety Gate

Secrets, destructive actions? The irreversibility check. Non-commutation (C2) means you can't undo a leaked secret. Gate it before it happens.

Feasibility Gate

Tools available? Permissions held? Finitude (C1) means you can't assume infinite capability. Check what's actually possible before proposing.

Standards Gate

Rules, formatting, invariants? Doc-lint, spec-linter, test suites. The ratchet convergence (C4) enforcer — quality only moves forward.

Progress Gate

Did last tick measurably advance claims? The anti-stall gate. Loops without progress are loops without learning. Terminate or escalate.

THE PATH FORWARD

Bounded Speculation Under Constraint

The graph is a policy. Every patch is a gradient step. Every gate is a loss signal. The system learns by building.

The graph is alive because it learns. Not through fine-tuning. Not through RAG. Through **structural evolution under constraint**.

Every accepted patch changes the graph topology. Changed topology changes what proposals are generated next. Changed proposals change what gates evaluate. Changed evaluations change what gets accepted. This is a feedback loop — a gradient descent on graph structure, with gates as the loss function and patches as the parameter updates.

The convergence proof is borrowed from reinforcement learning (Generalized Policy Iteration, Sutton & Barto 1998). The novelty is applying it to a knowledge graph where the policy and the environment are the same object. The graph is simultaneously the thing being optimized and the thing doing the optimizing.

**Constraints are the product.** Gates are the loss function. The graph is alive.
