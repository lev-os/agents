2 — Candidates at a Glance

REST

  • Universal tooling and ecosystem support
  • Human-readable, trivially debuggable with curl
  • Caching via HTTP semantics (ETags, CDN, proxies)
Over-fetching and under-fetching without careful API design

GraphQL

  • Client-driven queries eliminate over-fetching
  • Strong type system with introspection
  • Single endpoint simplifies client routing
Query complexity attacks and N+1 problems require careful resolver design

gRPC

  • Binary protobuf encoding — 5-10x smaller payloads
  • HTTP/2 multiplexing and bidirectional streaming
  • Code-generated clients with strict contracts
No native browser support; requires a proxy layer for web clients

4 — Deep Dive: Latency

Why gRPC wins on latency

Protobuf binary encoding eliminates JSON parse overhead. HTTP/2 multiplexing avoids head-of-line blocking. Here's a benchmark fetching the same user profile 10,000 times:

benchmark results (p99 latency, ms)
Protocol     p50    p95    p99    Payload Size
─────────    ───    ───    ───    ────────────
REST/JSON    4.2    8.1    14.3   847 bytes
GraphQL      3.8    7.6    13.1   412 bytes (query-specific)
gRPC         1.1    2.3     3.8    94 bytes (protobuf)

The gRPC call definition that produces this result:

user_service.proto
syntax = "proto3";

service UserService {
  rpc GetUser (GetUserRequest) returns (User);
  rpc ListUsers (ListUsersRequest) returns (stream User);
}

message GetUserRequest {
  string user_id = 1;
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
  int64 created_at = 4;
}

Compared to the equivalent REST call returning the same data as a 847-byte JSON payload, gRPC's protobuf encoding compresses the response to 94 bytes without any additional compression layer. Over thousands of inter-service calls, this compounds.

5 — Deep Dive: Schema Evolution

GraphQL deprecation vs REST versioning

REST typically requires URL-based versioning (/v1/users → /v2/users), which means running parallel endpoints during migration. GraphQL handles this with additive changes and the @deprecated directive:

schema.graphql — evolving without versioning
type User {
  id: ID!
  name: String!
  email: String!

  # Old field — still works, but clients see the deprecation warning
  fullName: String @deprecated(reason: "Use `displayName` instead")

  # New field — added without breaking existing queries
  displayName: String!
  avatarUrl: String
}

Protobuf achieves similar backward compatibility through field numbering. Adding field 5 never breaks clients that only know fields 1-4, and reserved field numbers prevent accidental reuse:

user.proto — backward-compatible evolution
message User {
  string id = 1;
  string name = 2;
  string email = 3;
  int64 created_at = 4;

  // Added in v2 — old clients simply ignore this field
  string avatar_url = 5;
  string display_name = 6;

  // Field 7 was removed; reserve it to prevent reuse
  reserved 7;
  reserved "legacy_role";
}
6 — Recommendation
High Confidence

There is no universal winner. The right choice depends on where the API sits in your architecture. Most mature microservices platforms end up using two or all three.

Choose REST if...
  • You need a public API with broad third-party adoption
  • HTTP caching and CDN support are critical (content-heavy apps)
  • Your team is small and needs the fastest path to production
  • You want maximum observability with standard tooling
Choose GraphQL if...
  • Multiple clients (web, mobile, partner) need different data shapes
  • You want to avoid API versioning and evolve schemas additively
  • Frontend teams need autonomy to build features without backend changes
  • Data comes from many backend services (federation / BFF pattern)
Choose gRPC if...
  • Internal service-to-service communication dominates your traffic
  • Latency and throughput are non-negotiable (real-time, financial, gaming)
  • You need bidirectional streaming (live feeds, multiplayer state sync)
  • Strict API contracts enforced at compile time are a requirement
Pragmatic Default

gRPC internally, REST or GraphQL externally. Use gRPC for all inter-service communication where latency and type safety matter. Expose a REST API for public consumers who need simplicity and cacheability, or a GraphQL API for client teams who need query flexibility. This is the pattern Netflix, Uber, and most large-scale platforms have converged on.
