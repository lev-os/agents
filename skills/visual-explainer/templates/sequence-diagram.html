OAuth2 Authorization Code Flow

RFC 6749 Section 4.1 — the most secure grant type for server-side applications

The authorization code flow separates authentication from token exchange. The user authenticates with the authorization server, which issues a short-lived authorization_code to the browser. The backend exchanges this code for an access_token and refresh_token, keeping credentials off the client entirely.

Authorization Code Flow

sequenceDiagram
    participant U as User
    participant B as Browser
    participant A as Auth Server
    participant R as Resource Server
    participant T as Token Store

    Note over U,T: Phase 1 - Authorization Request

    U->>B: Click Sign In
    activate B
    B->>A: GET /authorize with client_id, redirect_uri, scope, state
    activate A
    A->>A: Validate client_id and redirect_uri
    A-->>B: 302 Redirect to login page
    deactivate A

    Note right of B: User sees consent screen

    U->>B: Enter credentials and grant consent
    B->>A: POST /login with credentials
    activate A
    A->>A: Authenticate user and verify consent

    alt Authentication Success
        A-->>B: 302 Redirect to redirect_uri with code and state
        deactivate A
        deactivate B

        Note over U,T: Phase 2 - Token Exchange

        B->>A: POST /token with code, client_secret, redirect_uri
        activate A
        A->>A: Validate authorization code
        A->>T: Store refresh token
        activate T
        T-->>A: Confirm stored
        deactivate T
        A-->>B: 200 with access_token and refresh_token
        deactivate A

        Note over U,T: Phase 3 - Resource Access

        B->>R: GET /api/user with Bearer token
        activate R
        R->>A: Validate token signature and expiry
        activate A
        A-->>R: Token valid with scopes
        deactivate A
        R-->>B: 200 with user data
        deactivate R
        B-->>U: Display user profile

        Note over U,T: Phase 4 - Token Refresh

        loop Every 15 minutes before expiry
            B->>A: POST /token with grant_type refresh_token
            activate A
            A->>T: Lookup and rotate refresh token
            activate T
            T-->>A: New refresh token
            deactivate T
            A-->>B: 200 with new access_token and refresh_token
            deactivate A
        end

    else Authentication Failure
        A-->>B: 302 Redirect with error access_denied
        deactivate A
        deactivate B
        B-->>U: Display error message
    end
PKCE extension. For public clients (SPAs, mobile apps), add code_verifier and code_challenge parameters to prevent authorization code interception. The server verifies the challenge during the token exchange step.

Step-by-Step Breakdown

1

Authorization Request

The browser redirects to /authorize with the client_id, requested scope, a CSRF-preventing state parameter, and the redirect_uri where the code will be delivered.

User → Browser → Auth Server
2

User Authentication

The auth server presents a login page. The user enters credentials and reviews the requested scopes on a consent screen. Multi-factor authentication happens here.

User → Auth Server
3

Authorization Code Issued

On success, the auth server issues a short-lived authorization code (typically 60 seconds) and redirects back to redirect_uri?code=xyz&state=abc. The code is single-use and bound to the requesting client.

Auth Server → Browser
4

Token Exchange

The backend sends the authorization code plus its client_secret to the token endpoint. This server-to-server call keeps the secret out of the browser. The auth server returns an access token (short-lived) and a refresh token (long-lived).

Browser → Auth Server → Token Store
5

Resource Access

The access token is sent as a Bearer header with each API request. The resource server validates the token signature and checks that its scopes authorize the requested operation.

Browser → Resource Server → Auth Server
6

Token Refresh

Before the access token expires, the client uses the refresh token to obtain a new pair. The auth server rotates the refresh token on each use, invalidating the previous one. If the refresh token is compromised, rotation ensures the attacker's copy is quickly revoked.

Browser ↔ Auth Server ↔ Token Store

Error Scenarios

Step Failure Error Code Impact Severity
Authorization Request Invalid redirect_uri does not match registered URI invalid_request Auth server refuses to redirect; user sees error page on auth server domain HIGH
Authorization Request Unknown client_id unauthorized_client Flow terminates immediately; no code issued HIGH
User Authentication Invalid credentials or MFA failure access_denied Redirect to client with error; user must retry login MEDIUM
User Authentication User denies consent on scope screen access_denied Redirect with denial error; application cannot access requested resources LOW
Token Exchange Authorization code expired (older than 60 seconds) invalid_grant Token exchange fails; user must re-authenticate from the start MEDIUM
Token Exchange Code reuse detected (replay attack) invalid_grant All tokens issued from this code are revoked; possible security incident HIGH
Resource Access Expired or invalid access token 401 Unauthorized API returns 401; client should attempt token refresh LOW
Resource Access Insufficient scopes for requested endpoint 403 Forbidden User must re-authorize with additional scopes MEDIUM
Token Refresh Refresh token revoked or rotated by another session invalid_grant User must re-authenticate; all sessions for this grant are invalidated HIGH
Token Refresh Token Store unavailable server_error Refresh fails; existing access token remains valid until expiry, then user must re-login MEDIUM
Defense in depth. Always validate state to prevent CSRF, enforce redirect_uri exact-match (no wildcards), and use PKCE for public clients. Token rotation with reuse detection catches stolen refresh tokens within one refresh cycle.
