Implementation Plan

WebSocket Connection Pool

Scope: src/net/ws/ — connection lifecycle, pooling, and graceful degradation

Context

The Problem

Each component creates its own WebSocket connection. Under load, the browser hits the 6-connection-per-origin limit, requests queue silently, and users see stale data with no error feedback. A shared connection pool with lifecycle management eliminates the bottleneck.

Current
1 Component mounts, calls new WebSocket(url) directly
2 Each subscription opens a dedicated socket — no sharing
3 Browser hits 6-connection limit; new sockets queue behind closed ones
4 Silent failures: components show stale data, no reconnect logic
Proposed
1 Components call pool.acquire(channel) — pool manages sockets
2 Multiplexed channels share a single socket per origin
3 Pool enforces maxConnections with backpressure queue
4 Automatic reconnect with exponential backoff; health status exposed
Architecture

Connection State Machine

        graph TD
          IDLE[Idle] -->|acquire| CONNECTING[Connecting]
          CONNECTING -->|open| READY[Ready]
          CONNECTING -->|error| BACKOFF[Backoff]
          READY -->|message| READY
          READY -->|close| DRAINING[Draining]
          READY -->|error| BACKOFF
          DRAINING -->|drained| IDLE
          BACKOFF -->|retry| CONNECTING
          BACKOFF -->|max retries| DEAD[Dead]
          DEAD -->|reset| IDLE

Implementation

Modified Functions

acquire

src/net/ws/pool.ts
async acquire(
  channel: string,
  opts?: AcquireOpts
): Promise<PooledSocket> {
  const key = this.originKey(channel);
  const existing = this.connections.get(key);

  if (existing?.state === 'ready') {
    existing.refCount++;
    existing.channels.add(channel);
    return existing;
  }

  if (this.connections.size >= this.maxSize) {
    return this.enqueueRequest(channel, opts);
  }

  return this.createConnection(key, channel);
}

Reuses existing sockets when possible. If the pool is full, the request enters a backpressure queue with an optional timeout from opts.timeout.

release

src/net/ws/pool.ts
release(
  channel: string
): void {
  const key = this.originKey(channel);
  const conn = this.connections.get(key);
  if (!conn) return;

  conn.channels.delete(channel);
  conn.refCount--;

  if (conn.refCount <= 0) {
    conn.state = 'draining';
    this.scheduleDrain(key, conn);
  }
}

Decrements the reference count. When no channels remain, the connection transitions to draining with a configurable grace period before actual close.

reconnect

src/net/ws/backoff.ts
async reconnect(
  key: string,
  conn: PooledSocket
): Promise<void> {
  if (conn.retryCount >= MAX_RETRIES) {
    conn.state = 'dead';
    this.emit('dead', { key });
    return;
  }

  conn.state = 'backoff';
  const delay = baseDelay * 2 ** conn.retryCount;
  await sleep(Math.min(delay, MAX_DELAY));

  conn.retryCount++;
  conn.state = 'connecting';
  await this.openSocket(key, conn);
}

Exponential backoff with a ceiling. After MAX_RETRIES, the slot is marked dead and an event is emitted so the UI can show a degradation notice.

Interface

API Surface

Function Parameters Returns Description
pool.acquire channel: string, opts?: AcquireOpts Promise<PooledSocket> Get or create a socket for the given channel. Queues if pool is full.
pool.release channel: string void Release a channel subscription. Socket closes when refCount hits zero.
pool.health — PoolHealth Snapshot of pool state: active, idle, draining, dead counts.
pool.reset key?: string void Force-reset a dead connection slot (or all if no key). Transitions to idle.
pool.drain — Promise<void> Gracefully close all connections. Resolves when all sockets are closed.
pool.on event: PoolEvent, handler: Function Unsubscribe Subscribe to pool events: ready, dead, degraded, recovered.
Risks

Edge Cases

Scenario Expected Behavior Severity
Server sends close frame during active subscription Pool enters backoff for that slot; subscribers receive onReconnecting event. Messages buffer until reconnect succeeds or max retries exhausted. High
Component unmounts before acquire resolves Pending request is removed from backpressure queue. If socket already opened, immediate release triggers drain. Medium
Network goes offline then back online All active connections enter backoff simultaneously. Reconnect attempts are jittered to avoid thundering herd on the server. High
Pool full and backpressure queue timeout expires Queued request rejects with PoolExhaustedError. Component receives the error and can fall back to HTTP polling. Medium
Two components acquire same channel simultaneously First acquire creates connection; second finds it in connecting state and awaits the same ready promise. No duplicate sockets. Low
Health check finds zombie socket (open but no pong) Pool force-closes the socket, transitions to backoff, and logs a warning with the last activity timestamp. Medium
Guidance

Implementation Notes

Backward Compatibility
The existing useWebSocket(url) hook remains functional but becomes a thin wrapper around pool.acquire. Existing components require no changes. The ws property on AppContext is removed — any direct references (grep for ctx.ws) need updating before merge.
Critical Warning
Do not share a pool across workers. Each Worker or SharedWorker must instantiate its own pool — WebSocket instances are not transferable via postMessage. Attempting to pass a pooled socket to a worker will silently fail with a DataCloneError.
Performance Note
The health check interval defaults to 30s. Under high-throughput scenarios (1000+ messages/sec), reduce the ping/pong timeout to 5s to detect zombie connections faster. The healthCheckInterval config accepts milliseconds. Memory overhead per pooled socket: ~2KB baseline + message buffer.
