1 Core Insight

CDO adaptive and "swarm" are the same primitive at different layers. CDO fans out reasoning across N agents and picks the best answer. Swarm fans out execution across N agents and picks the best result. The only difference is the evaluation function.

CDO Adaptive

Deliberation Layer

N agents deliberate on a question. LLM-as-judge scores reasoning quality. Best reasoning wins.

fan_out(question, N_agents)
  → each reasons independently
  → llm_judge.score(reasoning)
  → best_score wins
Swarm

Execution Layer

N agents compete on a task. Domain metric scores execution quality. Best result wins.

fan_out(task, N_agents)
  → each executes independently
  → domain_metric.score(result)
  → best_metric wins

The real primitive: competitive fan-out with convergent scoring. CDO and swarm are profiles of the same thing. Once Lev is stable, HyperSpace = a plugin + some FlowMind declarations.

2 What is CRDT

A data structure that always merges without conflicts. Two agents modify their local copies independently. Merge. Same result regardless of order. No coordinator needed. No consensus protocol. Just math.

□

LWW-Register

Last write wins. Timestamp breaks ties. Simplest CRDT.

Lev has this: ClassifyFn V0
▲

G-Counter

Grow-only counter. Each agent increments its slot. Sum = total. Never conflicts.

New: agent scores, token counts
◉

LWW-Map

Per-key last-writer-wins. Each agent owns its keys. Merge = union of all keys.

New: entity graph convergence

Where CRDT Plugs Into Lev

Lev's ClassifyFn is the extension point for graph entity merge strategies. V0 is LWW (last-writer-wins). V1 is temperature scoring. CRDT is just V2 — a new classify strategy, not a new system. The interface already exists in core/graph/src/classify/.

7 The Config Surface

Three knobs. Zero new systems. This is the entire delta between Lev-today and Lev-with-swarm.

YAML — FlowMind execution config
# 1. Execution topology
execution:
  type: swarm                    # new enum value
  merge: crdt                   # ClassifyFn strategy
  metric: val_loss              # convergence metric (domain-specific)
  metric_direction: asc         # lower = better

# 2. Event bus transport
event_bus:
  transport: gossip             # instead of 'jsonl'
  # or: transport: shared-sqlite (for single-machine fleet)

# 3. Orchestration constraints
orchestration:
  immutability: relaxed         # C4 ratchet off for experimental runs

What Each Knob Controls

type: swarm

Agents self-select work. No coordinator. Results converge via merge strategy. Unbounded agent count.

transport: gossip

Cross-process event broadcast. Every agent sees every event. Replaces mail/inbox polling.

immutability: relaxed

C4 ratchet off. Mutations are experiments, not commits. Failed hypotheses are pruned, not preserved.

8 The Endgame

Once Lev's core runtime is stable, HyperSpace-like behavior is achievable with a domain plugin and FlowMind declarations. No new systems. No new contracts. Just implementations behind existing extension points.

Recipe: HyperSpace on Lev

  1. Domain plugin — plugins/swarm-research/ or plugins/hyperspace/
  2. FlowMind declarations — autoresearch loop as .flow.yaml
  3. ExecutionStrategy.type: swarm — wired through orchestration
  4. Gossip transport — or shared-SQLite for single-machine
  5. CRDT classify — V2 ClassifyFn for convergent entity graph

CDO adaptive is exactly what people are calling "swarm" today. It's all topologies and runtime behaviors. The industry is reinventing composition patterns and giving them new names. Lev already has the architecture — it just needs the enum values and a few adapter implementations.
