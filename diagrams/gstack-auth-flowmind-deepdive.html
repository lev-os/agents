Browser auth + FlowMind extraction

Direct Chromium cookie reads are the fragile part. The workflow is the real design problem.

`gstack` reads Chromium cookies by opening the browser's SQLite cookie DB, decrypting values with a Keychain-derived AES key, and injecting them into the active Playwright context. That is why it can feel seamless for QA. It is also why “access denied” happens: the direct path depends on the cookie DB, the Keychain secret, the correct profile, and local filesystem access all lining up. Lev already has a better extraction backend in auth-sniffer. What it is missing is the smoother in-place hydration UX and the SDLC seat packaging around it.

Bottom line use auth-sniffer as backend, steal `gstack`’s picker + hydration feel, and wire the rest through `core-sdlc` + `browser-cascade` flows

01

Why the Chromium DB path often fails

The code is simple in concept but brittle in practice. It is not one permission check. It is a chain of dependencies that can each fail differently.

File path problem

Wrong profile or no DB where the code expects it

`gstack` looks under ~/Library/Application Support/<browser>/Default/Cookies. If your active login lives in a different profile, the DB may exist but not contain the right cookies.

DB lock problem

Browser has the SQLite DB busy

`gstack` tries to mitigate this by copying the DB, WAL, and SHM files to /tmp. If even that cannot be read or copied cleanly, the import path still fails.

Keychain problem

The actual secret comes from macOS Keychain

The code calls security find-generic-password -s "<browser Safe Storage>" -w. If you deny the dialog, miss it, or macOS reports “interaction not allowed,” decryption fails even if the DB read succeeds.

Most likely when you see “can’t access it”
  • Keychain timeout/deny — the code explicitly throws if the dialog is not approved within 10s.
  • DB locked — if copy-open still fails, it reports the browser DB as locked.
  • Wrong browser/profile — DB exists, but the cookies you want are not there.
Why CDP is safer

`auth-sniffer`’s robot mode can relaunch a browser with CDP and ask the browser for cookies plus web storage directly. That avoids most of the “read SQLite + derive key + decrypt row-by-row” fragility when the goal is a reusable auth artifact.

02

The actual `gstack` cookie import flow

This is the flow from the browser picker to the authenticated Playwright session. 

flowchart TD
  A["User runs cookie-import-browser"] --> B["List installed browsers"]
  B --> C["Open localhost cookie picker UI"]
  C --> D["Read domain counts from Cookies SQLite DB"]
  D --> E["User selects domains"]
  E --> F["Read Chromium Safe Storage secret from Keychain"]
  F --> G["PBKDF2 derive AES key"]
  G --> H["Decrypt v10 encrypted_value rows"]
  H --> I["Convert rows to Playwright cookie objects"]
  I --> J["page.context().addCookies(...)"]
  J --> K["Active QA browser becomes logged in"]

  D -. lock .-> L["Copy DB/WAL/SHM to /tmp and reopen"]
  F -. deny/timeout .-> M["Keychain error or timeout"]
Where the DB read happens
getCookieDbPath(browser, profile)
  → ~/Library/Application Support/<browser>/Default/Cookies

openDb(dbPath)
  → readonly sqlite open
  → if SQLITE_BUSY: copy db + wal + shm to /tmp
  → reopen the copy
Where the decrypt happens
security find-generic-password -s "<service>" -w
  → password

pbkdf2Sync(password, "saltysalt", 1003, 16, "sha1")
  → AES key

AES-128-CBC decrypt(encrypted_value[3:])
  → strip padding
  → skip first 32 bytes auth tag
  → cookie value
03

`gstack` vs `auth-sniffer`

These are complementary, not substitutes. One is better UX for current-browser login. The other is a better Lev-native backend for structured extraction and reuse.

Dimension `gstack` direct import `auth-sniffer` robot mode Takeaway
Primary job QA UX
Log the active QA browser in right now.		 Extraction backend
Capture reusable auth artifacts deterministically.		 Use `auth-sniffer` as backend, borrow `gstack` UX.
Cookie source Direct SQLite DB read + local decrypt. Direct query strategy and/or CDP-assisted extraction path. `auth-sniffer` already has broader extraction paths.
Web storage Weak
Not first-class in the picker import flow.		 Strong
Explicit localStorage + sessionStorage extraction.		 Do not regress to cookie-only thinking.
Runtime bridge Injects directly into the current Playwright context. Exports to Playwright, PinchTab, browser-use, env, and runtime helpers. Lev already owns the better bridge surface.
Operator experience Best
Picker UI, domain counts, import/remove loop.		 Tooling-heavy
CLI-first, deterministic, less friendly for QA.		 This is the main thing to learn from `gstack`.
04

How the extracted workflows connect through the SDLC

The clean version is not one monolith. Browser/session flows belong to browser-cascade. Specialist reframe/review flows belong to core-sdlc. 

flowchart TD
  A["User request"] --> B["office-hours skill\nproduct reframe"]
  B --> C["office-hours.flow.yaml\ncore-sdlc"]
  C --> D["eng-review skill\narchitecture + tests"]
  D --> E["eng-review.flow.yaml\ncore-sdlc"]
  E --> F["implementation / exec"]
  F --> G["cross-model-challenge skill"]
  G --> H["cross-model-challenge.flow.yaml\ncore-sdlc"]
  H --> I["browser-session-import skill"]
  I --> J["browser-session-import.flow.yaml\nbrowser-cascade + auth-sniffer"]
  J --> K["browser-qa-report skill"]
  K --> L["browser-qa-report.flow.yaml\nbrowser-cascade"]
  L --> M["browser-qa-fix skill"]
  M --> N["browser-qa-fix.flow.yaml\nbrowser-cascade + core-sdlc"]
  N --> O["review-readiness.flow.yaml\ncore-sdlc"]
  O --> P["ship / release"]
Ownership rule

Browser mechanics stay out of core FlowMind

FlowMind declares the workflow. Orchestration runs it. The browser-specific runtime belongs under plugins/browser-cascade/**, not core/flowmind/**.

Why this graph works

Each step has a clear owner and a clear artifact

Reframing, review, browser hydration, QA, and readiness are all separate concerns. That keeps the prompt layer thin and the runtime layer testable.

The first FlowMind programs to write

browser-cascade

Browser-owned programs

  • browser-session-import.flow.yaml — call `auth-sniffer` runtime adapters, resolve domains, hydrate the active session.
  • browser-qa-report.flow.yaml — run report-only browser QA.
  • browser-qa-fix.flow.yaml — QA plus fix plus re-verify loop.
core-sdlc

Seat-owned programs

  • office-hours.flow.yaml — product reframing, premise challenge, alternatives.
  • eng-review.flow.yaml — pre-implementation engineering review with test plan output.
  • cross-model-challenge.flow.yaml — explicit second-opinion / adversarial review seat.
  • review-readiness.flow.yaml — freshness/staleness aggregation for dashboard and ship gating.
