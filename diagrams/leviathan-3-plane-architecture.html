Leviathan Runtime Architecture

Three Planes of Separation

Leviathan separates concerns into three planes: what should happen (Definition), how it executes (Execution), and who/where state lives (State). Each plane owns distinct types, has clear file boundaries, and communicates through well-defined data contracts.

FlowMind

Definition Plane

Defines what should happen. Owns the schema, serialization formats, and the graph intermediate representation. A .flow.yaml file IS the serialization of a GraphDocument. No execution logic lives here.

Owns

  • Schema + validation
  • YAML / JSON / GraphML serialization adapters
  • GraphDocument IR (the canonical data structure)
  • Execution contract: start() / next() moving to Orchestration
  • nodeToInstruction() mapping moving to Orchestration

Key Types

  • GraphDocument name, entry, nodes, subgraphs
  • GraphNode prompt, eval, op, branches, next, ...
  • GraphNodeType agent function gate subgraph
  • PromptPipeline providers + policy + renderer

Key Files

  • core/flowmind/src/schema.ts
  • core/flowmind/schemas/
  • core/flowmind/src/execution-contract/

Inference Rule

  • subgraph_ref|inline → subgraph
  • eval + branches → gate
  • prompt → agent
  • else → function

Orchestration

Execution Plane

Decides how things run. Owns topology walking, DAG scheduling, loop execution with until-semantics, parallel dispatch, A2A job dispatch, and the entity pipeline. This is where start()/next() is migrating to.

Owns

  • Topology walking (start() / next() state machine)
  • DAG scheduling (Kahn's algorithm, critical path, runtime dynamic)
  • Loop execution (iterative runner, idle detection, circuit breaker)
  • Parallel dispatch + fan-in/fan-out
  • A2A job dispatch
  • Entity pipeline (discover → validate → promote → budget)

Key Types

  • InstructionKind 10 values:
    agent tool exec notify gate
    hitl loop parallel done fail
  • Instruction kind + id + inputs + requirements
  • ExecutionState cursor + vars + visitCounts
  • StepResult ok | failed | timed_out | rejected
  • AgentAdapter execute(), detectPromise()

Key Files

  • core/orchestration/src/graph/
  • core/orchestration/src/loop/
  • core/orchestration/src/a2a/
  • core/orchestration/src/entities/
  • core/orchestration/src/queue/

nodeToInstruction() Mapping

  • terminal: true → done
  • targets[] → parallel
  • until_condition → loop
  • requires_approval → hitl
  • prompt → agent
  • eval+branches → gate
  • op|action → exec
  • fallback → tool

Graph

State Plane

Manages who and where. Owns the entity graph, lifecycle FSM, knowledge (claims, evidence, links), and context projection. Entirely separate from FlowMind's behavior graph — this is the world model, not the program.

Owns

  • Entity graph (Entity, Claim, Link, Evidence)
  • Lifecycle FSM: ephemeral → captured → classified → crystallizing → crystallized → completed
  • 4-phase compositor: extract → classify → validate → apply
  • Context projection (views, soft lineage)
  • Pluggable adapters (bd, code, file, memory, lineage)

Key Types

  • Entity id, type, source, depth (L0-L3), claims, links
  • V0EntityType spec task proposal session commit agent intent episode
  • GraphAdapter extract() + apply() + query()
  • ClassifyFn FlowMind policy slot (D8)
  • GraphReader read-only graph access

Key Files

  • core/graph/src/compositor.ts
  • core/graph/src/types.ts
  • core/graph/src/adapters/
  • core/graph/src/bridge/
  • core/graph/src/projectors/

Compositor Pipeline

  • Extract: adapters produce raw GraphOperation[]
  • Classify: ClassifyFn applies depth/merge policy
  • Validate: hooks enforce schema + gates
  • Apply: persist to EventStore + all backends

What Flows Between Planes

The three planes are connected by typed data contracts, not shared state.

GraphDocument
The intermediate representation. FlowMind produces it from YAML; Orchestration consumes it to drive the start()/next() state machine.
FlowMind → Orchestration
InstructionKind
The dispatch envelope. Orchestration produces typed instructions from graph nodes; adapters consume them to execute real work. 10 kinds cover agent calls through terminal states.
Orchestration → Adapters
LevEvent
The causality spine connecting all three planes. Every significant transition emits a LevEvent. The Graph plane's compositor consumes them to build the world model. Orchestration emits them. FlowMind declares them.
FlowMind ↔ Orchestration ↔ Graph

Cross-cutting Infrastructure

Three concerns span all planes without belonging to any single one.

LevEvent
Canonical event envelope (version: 1, id, type, source, time, data, metadata?). Defined in @lev-os/domain. Provides causal ordering via correlation_id threading. Every plane reads, writes, and reacts to LevEvents.
ContextProvider
Declared in PromptPipeline (FlowMind schema), bridges Graph memory into Orchestration execution. Providers fetch ContextBlocks; policies rank and budget them; renderers format for the LLM.
AgentFS
FUSE/NFS filesystem mount backed by SQLite. Sits beneath all three planes, providing the persistent substrate that entities, flows, and execution state are stored in. Lives in crates/lev-agentfs/.

Active Migration: Execution Contract Relocation

The start()/next() state machine and nodeToInstruction() currently live in core/flowmind/src/execution-contract/. These are moving to core/orchestration/ to complete the plane separation — FlowMind should define what, not how to traverse it. After migration, FlowMind exports only GraphDocument + schema; Orchestration owns all runtime traversal. The InstructionKind type and Instruction interface are already defined in the execution-contract types, ready for the move.
