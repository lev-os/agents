R2: Cloud Provider Patterns

CDC + Outbox Across AWS, Azure, GCP

All three clouds converge on the same architecture: operational store + built-in CDC + fan-out to projections. Azure wins on simplicity (Change Feed is magic), AWS on cost (on-demand DynamoDB), GCP on analytics (BigQuery unlimited queries).

AWS
DynamoDB Streams + EventBridge Pipes + Lambda projections.

CDC: DynamoDB Streams auto-captures; EventBridge Pipes handles retry + DLQ.
Cost: On-demand DynamoDB is cheap for small scale.
Azure
Cosmos DB Change Feed + Event Hubs + Functions + materialized views.

CDC: Change Feed is built-in, zero-overhead, millions/sec.
Best for: Simplest path to event sourcing.
GCP
Pub/Sub (ordered by aggregate_id) + BigQuery event store + scheduled queries.

CDC: Pub/Sub subscription streams to BigQuery sink.
Best for: Analytics + unlimited replay history.

Projection Strategies

Three Ways to Build Read Models (from Marten)

Phase 2 should support all three strategies per projection target. Match strategy to consistency requirements.

Live

On-demand aggregation from events. No persistent projection. Always consistent but slower reads. Best for: admin dashboards, debugging.

Inline

Projection written synchronously at event append time. Strong consistency. Best for: critical state where lag is unacceptable.

Async

Background daemon updates projection. Eventual consistency. Best for: filesystem views, search indexes, secondary stores.

The Dual-Sync Realization

BD as CQRS/ES Node

BD is both event producer (hooks emit events on write) and potential projection consumer (could receive events from other adapters). This is textbook CQRS. The enterprise patterns that solve it are:

Outbox Pattern (Recommended for Phase 2)
BD writes state + event in same Dolt transaction. Separate relay publishes to event bus. External consumers (Notion, graph) subscribe independently. Mark event origin (source_system: "bd") to prevent loops.
CDC Fallback (If Moving to SQLite/PG)
Debezium auto-tails DB transaction logs → event streams. No application code changes needed. Applicable when BD moves to a backend where WAL tailing is native.
