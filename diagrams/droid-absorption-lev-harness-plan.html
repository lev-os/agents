Droid Pattern Absorption Into Lev Harness

Visual plan for harness / FlowMind absorption · answer: some yes, not all

The clean answer is: yes, several Droid-derived behaviors can live as harness or FlowMind settings, especially routing, execution bounds, approval policies, and skill-loading rules. But the most valuable Droid patterns are not settings-only problems. Repo readiness, trust UX, judge-backed self-improvement, long-session compaction quality, and enterprise-safe proactivity all need runtime machinery and/or product surfaces in addition to configuration.

Settings That Fit Cleanly

These are the areas where another agent could reasonably proceed by shaping config, schemas, or policy surfaces before inventing a new subsystem.

Harness config already exposes
  • model, adapter, harness, host, and adapter variants.
  • Model → adapter routing already exists in policy.yaml.
  • Execution has explicit limits, retry, timeout, and heartbeat knobs.
FlowMind already exposes
  • Provider defaults and detection order.
  • Policy fields like determinism, approval_required, and max_iterations.
  • Confidence routing and escalation patterns in docs/capabilities.
Good candidate additions
  • Named routing policies: default, review, cheap, private.
  • Explicit review trigger lists for risky actions.
  • Skill source precedence and conflict policies as first-class config.
Mechanism vs policy rule: if it changes which backend, tier, threshold, or escalation path gets used, it probably belongs in FlowMind/harness settings. If it changes whether the system can do the thing at all, it probably needs a new primitive or product surface.
5

What Should Not Be Pretended Into Settings

These are the traps. They may end up with settings later, but they should not be sold to yourselves as “just config work.”

Repo readiness
  • A real workflow with scans, scoring, remediation output, and framework-aware rules.
  • Not a single harness flag.
  • Best expressed as a new workflow that later consumes existing routing/skill settings.
Trust UX
  • Users feel safety through visible rationale, scoped approval, and risk preview.
  • Config can power it, but UX is the deliverable.
  • Closest existing Lev concept is ExecutionReview, not raw YAML.
Self-improvement loop
  • Needs telemetry, trace capture, friction mining, evals, and proposal generation.
  • Specs exist adjacent to this idea.
  • Still a pipeline, not merely a policy block.

YAML Sketches

These examples show the right split. The first two are real settings-style surfaces. The third is a workflow artifact that would consume those settings, not replace them.

Harness Settings

harness:
  routing:
    default_adapter: auto
    model_policy:
      review: gpt-5
      default: claude-sonnet
      cheap: gemini-flash
      private: ollama/llama3.2:3b
    fallback_chain: [claude-sonnet, gpt-5, gemini-flash]
  execution:
    max_iterations: 8
    timeout_ms: 300000
    heartbeat_interval_ms: 30000
    retry:
      max_retries: 2
      retry_on: [timeout, stall]
  review:
    approval_required: true
    require_review_for: [filesystem.delete, git.push, network.external]

FlowMind Settings

flowmind:
  providers:
    default:
      chat_model: claude-sonnet
      embed_model: ollama/nomic-embed-text
  routing:
    confidence_thresholds:
      inline_execution: 0.90
      moderate_research: 0.80
      minimum_proceed: 0.70
  steering_policy:
    escalation_policy:
      low_confidence: request_more_context
      risky_action: execution_review
    compaction:
      strategy: selective
      pre_compact_hook: preserve_artifacts

Workflow Using Them

name: agent-readiness
policy:
  determinism: true
  require_reasoning: true
  max_iterations: 4
steps:
  - id: scan_repo
    action: exec
    command: lev check repo-readiness --json
  - id: classify_gaps
    prompt: Classify autonomy blockers by severity and fixability
    output: gaps
  - id: propose_fixes
    prompt: Generate remediation plan from {{gaps}}
    output: plan

Absorption Plan

If another agent is already working in this space, the clean sequencing is below. Don’t start by inventing the biggest subsystem first.

Phase 1 — Configurable Now
  • Provider/model routing policies
  • Execution bounds: iterations, timeout, heartbeat, retry
  • Skill injection source/priority/composition rules
  • Approval-required and escalation policy fields

These can become harness + FlowMind settings immediately without inventing a new runtime.

Phase 2 — Settings Over New Machinery
  • Context compaction strategy and hook-driven memory injection
  • Queue routing, checkpoint timing, and proactive work steering
  • Review gate placement and confidence thresholds

These need both declarative policy and real queue/memory/runtime implementation.

Phase 3 — New Workflow / Product Surfaces
  • Agent-readiness repo workflow
  • Judge-backed self-improvement pipeline
  • Trust UX / approval surfaces
  • Enterprise-safe isolation, identity, and compliance layers

These should not be mislabeled as “just settings.” They are product/runtime initiatives.

Recommended framing for your other agent: “Use FlowMind/harness settings for routing, limits, approvals, and skill steering. Use new workflow specs for repo-readiness and self-improvement. Use runtime/product work for queues, compaction quality, and trust surfaces.”
