1 Graph Fundamentals

Before diving into Lev’s architecture, you need three CS concepts that underpin every modern agent framework: directed acyclic graphs, the actor model, and message passing.

DAG

Directed Acyclic Graph. Nodes connected by one-way edges, with no cycles (you can never follow edges back to where you started). This is how workflows flow: step A → step B → step C, never looping back implicitly.

Why it matters: DAGs guarantee termination. If your workflow is a DAG, it will always finish. Loops require explicit “until” conditions to prevent infinite execution.

Actor Model

Computation organized as independent actors that each have: a mailbox (queue of messages), state (private, never shared), and behavior (how to handle each message).

Why it matters: Actors don’t share memory. They communicate only through messages. This makes concurrency safe — no race conditions, no locks, no deadlocks.

Message Passing

Instead of calling functions directly, components send messages to each other through channels. The sender doesn’t wait — it fires and forgets (or optionally waits for a reply).

Why it matters: Decouples who does the work from who requests it. An agent can send a message to “run tests” without knowing which agent picks it up.

Three types of graphs you’ll encounter. Lev’s FlowMind uses DAGs for workflows, with explicit loop nodes when cycles are needed.

Actor Model Message Passing

Each actor processes one message at a time from its mailbox. No shared state means no race conditions. 

sequenceDiagram

    participant B as Agent Actor B

    O->>A: Execute step 1
    Note right of A: Mailbox receives msg
    A->>M: Store findings
    A->>O: Step 1 complete

How this applies to Lev: In FlowMind, each GraphNode is essentially an actor — it has a type (behavior), receives input from upstream nodes (messages), and produces output for downstream nodes. The LevEvent is the message envelope. The executor walks the DAG, dispatching work to each node-actor.

3 Actor Graph Model

What if every GraphNode was an actor with its own mailbox? Here’s how the current FlowMind executor compares to a true actor-kernel design.

Current: Sequential Walker

  • Executor follows entry → next chain
  • One node runs at a time (except parallel type)
  • State passed via shared context object
  • Branching via branches map on gate nodes
  • No independent lifecycle per node
  • LevEvent emitted at lifecycle boundaries

Proposed: Actor Kernel

  • Each node is an actor with its own mailbox
  • All nodes run concurrently (natural parallelism)
  • State is private per actor, shared via messages only
  • Branching via message routing to downstream actors
  • Each actor manages its own lifecycle and failure
  • LevEvent IS the message envelope between actors

Key Architectural Mapping
LevEvent = Actor Message
The existing LevEvent type already functions as a message envelope with source, target, payload, and metadata. In an actor model, it becomes the native message type.
GraphNode.type = Actor Behavior
Each GraphNodeType (agent, function, gate, etc.) defines how the actor processes messages. The type IS the behavior definition.
ProtocolAdapter = Mailbox Adapter
A2A protocol adapters translate external messages into LevEvent format — exactly what an actor mailbox adapter does.
Why actors make A2A trivial: In the current model, inter-agent communication requires a separate A2A protocol layer. In an actor model, agents ARE actors — sending work to another agent is just sending a message to another actor. The AgentJob type from core/orchestration/src/a2a/types.ts maps directly to an actor message with prompt as payload and AgentJobResult as the reply.
4 Step Type Hierarchy

FlowMind has three document formats, each with its own step type. Should the next version use base type extension or a typed envelope?

Current Step Types

GraphNode
Graph format (v2). Object-keyed nodes with entry point. Type is GraphNodeType (9 variants). Properties: prompt, eval, action, next, branches.

Used by: cc-session.flow.yaml and the executor runtime.
StepBasedStep
Step format (v1.5). Flat array of steps with IDs. Type is StepActionType (6 variants: exec, tool, load, store, pass, escalate). Properties: command, tool, params, prompt.

Used by: dor-lev.flow.yaml, shuffle.flow.yaml.
TemplateFlowStep
Template format (v1). Nested under flow constructs with metadata. No type enum — uses type as a freeform string. Properties: action, command, condition, steps.

Used by: Legacy template documents. Being normalized.

Two Design Proposals

Option A: Base Step Extension

One base Step interface, extended by each variant. Classic OOP inheritance. 

interface BaseStep {
  id: string
  type: StepType
  description?: string
  next?: string
  on_before?: LifecycleHook[]
  on_after?: LifecycleHook[]
}

interface AgentStep extends BaseStep {
  type: 'agent'
  prompt: string
  output_schema?: string
}

interface GateStep extends BaseStep {
  type: 'gate'
  eval: string
  branches: Record<string, string>
}

interface FunctionStep extends BaseStep {
  type: 'function'
  action: string
}
  • Type narrowing via discriminated union on type
  • Each variant only has fields relevant to it
  • Compile-time safety: can't set prompt on a gate
  • Harder to add cross-cutting concerns

Option B: Typed Envelope

One envelope with a payload discriminated by type. Like LevEvent. 

interface StepEnvelope {
  id: string
  type: StepType
  description?: string
  next?: string
  hooks?: {
    before?: LifecycleHook[]
    after?: LifecycleHook[]
  }
  payload: StepPayload
}

type StepPayload =
  | { kind: 'agent'; prompt: string;
      output_schema?: string }
  | { kind: 'gate'; eval: string;
      branches: Record<string, string> }
  | { kind: 'function'; action: string }
  | { kind: 'loop'; until: string;
      max_iterations?: number }
  • Single type to serialize/deserialize
  • Payload is a clean discriminated union
  • Cross-cutting concerns on the envelope, not payload
  • Matches how LevEvent already works

Recommendation: The typed envelope (Option B) is the stronger choice for Lev. It matches the existing LevEvent pattern, makes serialization trivial (one codec), and cleanly separates routing concerns (envelope) from execution concerns (payload). The current GraphNode is already envelope-shaped — it just needs the payload discriminated more cleanly.
5 GraphDocument Serialization

A GraphDocument is a FlowMind workflow. It serializes to YAML for humans and deserializes back to TypeScript objects for execution. Here’s what that looks like and the CS concepts involved.

GraphDocument Interface

interface GraphDocument {
  name: string              // Human-readable workflow name
  entry: string             // ID of the first node to execute
  nodes: Record<string, GraphNode>  // All nodes, keyed by ID
  subgraphs?: Record<string, GraphDocument>  // Nested workflows
  context?: ContextAssemblyConfig  // Default context pipeline
}

As YAML (Real Example)

Based on the actual cc-session.flow.yaml in the codebase: 

# cc-session-classifier — a real FlowMind GraphDocument
name: cc-session-classifier
entry: parse-events

nodes:
  parse-events:
    type: agent           # LLM call — classifies tool events
    prompt: |
      You are a classifier analyzing Claude Code
      tool usage patterns...
    output: entity_candidates
    next: classify

  classify:
    type: gate            # Decision node — routes by confidence
    eval: |
      IF confidence >= 0.7: structured
      ELSE IF confidence >= 0.5: freestyle
      ELSE: low_confidence
    branches:
      classified: assess-lifecycle
      low_confidence: assess-lifecycle

  assess-lifecycle:
    type: gate            # Maps action patterns to lifecycle state
    eval: |
      action_types == ["observe"] → lifecycle = "ephemeral"
      "execute" in action_types → lifecycle = "manifested"
    branches:
      assessed: emit-footer

  emit-footer:
    type: function        # Code execution — assembles JSON output
    action: |
      Build footer structure with sense, intent, state...
    next: done

  done:
    type: terminal        # End of workflow

CS Concepts: Serialization Round-Trip

Serializer

Object → Bytes. Converts a TypeScript GraphDocument object into a YAML string that can be stored on disk. Also called a “writer” or “encoder.”

yaml.dump(graphDoc) → string

Deserializer

Bytes → Object. Parses a YAML string back into a typed GraphDocument. Also called a “reader,” “parser,” or “decoder.”

yaml.load(string) → GraphDocument

Codec

Serializer + Deserializer bundled as one unit. The codec is a “port” in hexagonal architecture — it defines how your domain types cross the boundary to/from external formats.

FlowCodec.encode() / .decode()

Hex Architecture Mapping: In hexagonal (ports and adapters) architecture, the GraphDocument interface is a port — it defines the shape of your domain. The YAML parser/serializer is an adapter — one of many possible implementations. You could swap YAML for JSON, TOML, or a database without changing the domain types. The type guards (isGraphDocument(), isStepBasedDocument(), isTemplateDocument()) detect which adapter produced the input.
Three Document Formats and Their Type Guards

FlowMind supports three input formats, each detected by its type guard:

Graph Format
isGraphDocument() checks: nodes is an object (not array) + entry is a string. This is the active format.
Step Format
isStepBasedDocument() checks: has steps array, does NOT have nodes. Flat list format.
Template Format
isTemplateDocument() checks: has metadata object + (flow_constructs or output_template or escalation_policy). Legacy.
