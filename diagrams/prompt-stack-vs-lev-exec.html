Prompt Stack vs Lev Exec

Two execution modes for SDLC work. One is a step-at-a-time instruction sequencer with schema-validated reports. The other is a profile-driven execution harness with adapter dispatch. Both exist. Neither is complete. Here's where they stand.

Prompt Stack

Step-at-a-time instruction sequencer

Shows the agent ONE step prompt at a time. Agent completes it, writes a schema-validated report, and the stack advances. Future steps are hidden. Contract-first: every report must match frontmatter schema + required sections.

  • init — create session from a named stack
  • next — reveal active step prompt (read-only, no side effects)
  • record — submit report, validate schema, advance to next step
  • status — show session progress
  • validate — check session integrity
lifecycle
lev stack init --stack sdlc-exec-validate lev stack next --session $ID # see step prompt # ... do the work ... lev stack record --session $ID --step exec-implement --report ./report.md lev stack next --session $ID # see next step # ... repeat until completed ...
Lev Exec

Profile-driven execution harness

Loads a named exec profile (adapter, model, loop control, until condition) and dispatches work through an adapter registry. Supports single-shot, until-loop, and graph-driven execution. Profiles compose from plugin defaults + project overlays + CLI flags.

  • profile — named YAML config (adapter, model, max_iterations, until)
  • adapter — runtime dispatch target (claude-agent-sdk, codex, ai-sdk)
  • until — semantic loop condition compiled to graph
  • graph — pre-compiled GraphDocument for FlowMind-driven execution
  • robot_mode — structured JSON output for phase handoff
lifecycle
lev exec "implement caching" --profile sdlc.flowmind.exec # adapter dispatches to claude-opus-4-6 # loops until semantic "done" condition # emits LevEvents on start/complete/fail # robot_mode wraps output in structured JSON

Prompt Stack: How It Actually Works

The runtime in plugins/prompt-stack/src/runtime.ts manages a session state machine. Each step is a full prompt loaded from prompts/steps/sdlc/*.md. Reports are validated against a schema contract before advancing.

1

init → createPromptStackSession()

Resolves stack from catalog YAML. Creates session.json in .lev/steering/sessions/. Sets activeStepIndex: 0. Returns session ID + first step metadata. Future steps hidden by ID only.

2

next → revealActivePromptStep()

Pure read. Returns the active step's full prompt text, before/after templates, orchestration envelope (subagent instructions + CLI commands), and project context paths. Does NOT modify session state.

3

Agent does the work

Reads the prompt. Implements/validates/routes per the step's intent. Writes a markdown report with YAML frontmatter matching the schema contract: session_id, stack_id, step_id, status, inputs, outputs + sections: Summary, Evidence, Outcome, Next.

4

record → recordPromptStackReport()

Validates report against schema. Guards: activeStepId === stepId (can't record wrong step). Creates receipt with SHA256 digests of prompt + report + contract. Increments activeStepIndex. Persists to session.json. If last step: writes completion manifest.

Lev Exec: How It Actually Works

The harness in core/harness/src/execution/runner.ts resolves a profile, selects an adapter, and dispatches work. Three execution modes: single-shot, until-loop (compiled to graph), and pre-compiled graph.

1

Parse args → resolve profile

CLI flags parsed into ExecOptions. If --profile given, loads YAML from plugins/*/profiles/ or .lev/exec-profiles/. Explicit CLI flags always override profile values. Resolution: built-in → plugin → project → CLI.

2

Select adapter → init

Registry selects adapter by name (claude-agent-sdk, codex, ai-sdk). Validates model compatibility. Calls adapter.init() with resolved options.

3

Dispatch execution mode

--graph: Walk FlowMind state machine via ecStart()/ecNext(). --until: Compile semantic condition to graph, loop via executeWithUntil(). Default: Single adapter.run() call. All paths emit LevEvents.

4

Result → exit code

Returns ExecResult with exit code, token usage, and optional structured output (robot_mode). Orchestration layer owns iteration semantics, circuit breakers, and budget enforcement.

Integration gaps

What's Missing Between Them

1. Profile ↔ Stack Wiring

Not Wired

Exec profiles declare prompt_stack: sdlc-exec-validate and prompt_stack_mode: compose_per_entity in YAML. These fields are parsed by profile-loader.ts but never consumed by the harness runner. The bridge: when exec starts with a stack-aware profile, it should init a stack session and use step prompts as the agent instruction for each phase.

2. Report Validation in Exec

Not Implemented

Exec's robot_mode: structured-json wraps adapter output in JSON, but there's no schema validation on that output. Stack has full schema validation (frontmatter + sections). Ideal: exec phases produce reports that stack validates before advancing. The validateProfile field in exec profiles hints at this but routes to a separate validation exec run, not stack schema checks.

3. Adapter Dispatch from Stack

Architecture Gap

Stack's orchestration envelope tells a subagent what CLI commands to run. It doesn't dispatch through exec adapters. For lev exec mode, the harness should be the caller — it reads the step prompt from the stack, dispatches via adapter, captures output, and records the report. Stack owns steering, exec owns dispatch.

4. Token/Event Cross-Reference

Separate Worlds

Exec emits LevEvents with executionId and tokenUsage. Stack creates report receipts with promptDigest and reportSha256. These two audit trails don't reference each other. Ideal: stack receipt includes the exec executionId, and exec events include the sessionId + stepId.

Target architecture

Ideal: Both Modes Working

The key insight: Prompt Stack and Lev Exec are not competing alternatives. Stack is the steering layer (what instruction, what schema, what step). Exec is the dispatch layer (which adapter, which model, how many iterations). They compose.
Mode A

Stack-only (Claude Code subagent) — Today's working mode. Orchestrator calls lev stack next, hands prompt to a subagent, subagent does the work, writes report, orchestrator calls lev stack record. No exec harness involved. Works for Claude Code sessions where the agent IS the runtime.

Status: working

Mode B

Exec-driven with stack steering — The target mode. lev exec "task" --profile sdlc.flowmind.exec inits a stack session internally, reads step prompts from the stack, dispatches each step through the adapter (claude-agent-sdk/codex), validates output against stack schema, and advances. The loop controller (--until) handles retry logic. Stack provides the what, exec provides the how.

Status: not wired — all pieces exist, bridge code missing

Bridge

What needs to happen: The harness runner (runner.ts) checks for prompt_stack in the resolved profile. If present, it calls createPromptStackSession() at exec start, uses revealActivePromptStep().prompt as the adapter task prompt, captures adapter output, formats as a stack-valid report, and calls recordPromptStackReport(). On retry (exec loop iteration), it reads the next step or re-runs the current one based on verdict routing.

Key files: core/harness/src/execution/runner.ts (dispatch), plugins/prompt-stack/src/runtime.ts (session API), plugins/core-sdlc/profiles/exec.yaml (profile with prompt_stack field)
