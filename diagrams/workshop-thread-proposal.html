Proposal

Workshop should be a thread, not a graveyard.

The problem is not that you lack research. The problem is that research, intake, analysis, POCs, reports, and promotions are split across multiple filesystem surfaces with no active thread model. The stopgap is not another CLI. It is a visual workshop control plane that makes the live work visible.

Working proposal: Recast workshop as a first-class Leviathan surface with one active object: the workshop thread. A thread owns a hypothesis, sources, research notes, current disposition, promotion target, and next action. The UI then becomes a thread browser plus a promotion lane.

Diagnosis

The data exists. The thread does not.

Right now you have at least four competing workshop truths: top-level workshop research, new `.lev/workshop` intake clones, `.lev/pm/intake` reports, and a `core-workshop` plugin that still assumes an older tracker-file world. That is why the workshop feels useful in your head and invisible on screen.

What exists
Top-level `workshop/` Deep corpus: `analysis`, `approved`, `extract`, `poc`, `pocs`, `reports`, templates, scripts.
`.lev/workshop/intake` Fresh source staging area for current repo intake and comparison work.
`.lev/pm/intake` Human-readable reports and synthesis artifacts that already behave like decision records.
`plugins/core-workshop` Legacy surface with commands, but it still points at old tracker assumptions instead of current thread truth.
What is missing
Active thread object A durable unit that binds source, analysis, hypothesis, disposition, and promotion target.
One obvious UI surface A page that answers: what is live, what is cold, what should be promoted, and what is stuck.
Promotion lane A visible route from intake to extract to integrate to approved instead of silent folder shuffling.
Recent work visibility Threads need heat, age, notes, last action, next action, and ownership so the workshop stops decaying into archaeology.
Why It Feels Dead

The current plugin is not the present tense of the workshop.

My read is that `plugins/core-workshop` does not currently function as the living workshop surface. It reads more like an older integration shell. The repo reality has moved faster than the plugin.

Stale contract

It points at old tracker files

The current commands still look for `SEMANTIC-TIER-MAPPING.yaml` and `IMPLEMENTATION-TRACKER.csv` under top-level `workshop/`, then fall back to mock data if they do not find them. That means the UI layer is not reading the same workshop truth you are using today.

Wrong abstraction

It lists tools, not threads

The plugin is oriented around tool classification and plugin listing. What you need now is not another taxonomy of tools. You need a workstream surface that turns research into active threads and promotion decisions.

Opportunity

Use it as scaffolding, not truth

The plugin can still help as a shell for routes or commands, but it should not remain the canonical model. The canonical object should be a workshop thread backed by manifests and rendered visually.

Workshop Thread

Reframe the workshop around one live object.

A workshop thread is the unit of gravity. Instead of browsing folders directly, you browse threads. Each thread may point to many files, but the UI should always answer the same questions: what is this, why does it matter, what evidence exists, what is its state, and what happens next?

01 · Intake
Source enters the system
Repo, paper, URL, transcript, or idea lands. A thread is created immediately, not just a folder.
02 · Evidence
Notes and reports attach
Thread links to staged clone, intake report, analysis markdown, screenshots, and any explainer artifacts.
03 · Position
Disposition becomes explicit
Extract, monitor, integrate, pass, vendor, or hold. Not hidden in prose or in your head.
04 · Promotion
Target path is named
If it matters, the thread gets a promotion target into `core`, `plugins`, `community`, `apps`, or `tooling`.
05 · Retain or cool
Heat decays visibly
Cold threads move to archive or library. They are still findable, but they stop cluttering the live board.
Identity

Thread title + slug

Human name, canonical slug, and type: repo, paper, protocol, product, concept, or benchmark.

Evidence

Source bundle

URLs, staged clone path, reports, screenshots, notes, and visual explainers live under one visible object.

State

Disposition + heat

Current decision, confidence, age, last touched time, and whether the thread is active, stalled, or archived.

Promotion

Next landing zone

Concrete target path and the delta required to move from inspiration to Lev-native implementation.

This is the move I would make next. Do not start by rebuilding the plugin. Start by generating a living HTML board from the existing filesystem surfaces and render it as the workshop thread wall. Then let that page tell you what the eventual Leviathan UI actually needs.

My Read

This should become the Leviathan UI wedge.

I think you are right that the workshop is purgatory right now. The answer is not to abandon it. The answer is to give it a first-class visual surface and let that surface define the next abstraction.

Why this matters

The workshop is already the research brain.

You already use it as the memory of what you have studied, cloned, compared, and half-promoted. If Leviathan is getting out of hand, the workshop wall is one of the best possible wedges for the UI: it has real data, real tension, and obvious decisions.

My proposal

Use visual-explainer as the stopgap, then harden what proves useful.

Make the workshop explainer page your daily surface for a week. Let it reveal what you actually click, what filters you need, what metadata is missing, and which thread operations deserve to become native UI.

Grounded in the current repo state: top-level `workshop/` remains the deep corpus, `.lev/workshop/intake/*` is where fresh intake clones are landing, `.lev/pm/intake/*` is the report ledger, and `plugins/core-workshop` is not yet aligned to that lived reality.
