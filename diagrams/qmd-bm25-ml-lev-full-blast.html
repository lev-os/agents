Retrieval Systems, Not Magic

QMD, BM25, and Lev

A full-stack explanation of lexical search, semantic search, reranking, and where Lev can split retrieval into cleaner layers.

The short version: BM25 finds the same words, embeddings find the same ideas, and rerankers read the candidates more carefully. qmd combines all three. Lev can benefit most by separating those layers instead of forcing one tool to do every job.

BM25 is classical IR, not a neural net qmd is hybrid retrieval Lev wants cleaner boundaries
What you should walk away with

You should be able to explain why qmd search, qmd vsearch, and qmd query exist as separate modes, and why BM25 Turbo is interesting for Lev without being an automatic qmd replacement.

  • Lexical retrieval Exact terms, file names, acronyms, code symbols, literal phrases.
  • Semantic retrieval Meaning similarity via embeddings and vector search.
  • Precision layer Chunk selection and reranking decide which candidates are actually best.

1 — BM25 Basics

What BM25 Actually Does

Rare Terms Matter

IDF: uncommon words carry more signal

If every document says system, that word barely helps ranking. If only a few say zephyr, that term becomes strong evidence.

BM25 rewards words that narrow the candidate set.

Diminishing Returns

Term frequency helps, but it saturates

Seeing kubernetes three times is usually better than once. Seeing it fifty times is not fifty times better.

BM25 avoids naive keyword stuffing by capping how much repetition helps.

Length Normalization

Long documents should not win by being long

A giant transcript naturally contains more words than a short design note. BM25 normalizes that so a focused short match can still beat a huge noisy file.

Mental Formula
BM25 score ≈ rare query terms × useful repetition × document length correction

That is why BM25 is excellent for code symbols, filenames, product names, acronyms, and literal phrases.

Important distinction

BM25 is not modern ML. It is classical information retrieval. It uses corpus statistics, not learned semantic representations.

2 — qmd Pipeline

What qmd Actually Does

qmd is not “just BM25.” It is a local hybrid retrieval pipeline that mixes lexical search, vector search, query expansion, rank fusion, chunking, and reranking.

End-to-End Search Flow
Stage 1

Index documents

Markdown files go into SQLite. qmd builds an FTS index and stores chunk/vector data.

Stage 2

Run BM25 probe

qmd checks for an obvious lexical win before spending more on expansion and reranking.

Stage 3

Expand the query

The local LLM can generate typed variants: lexical, vector, and HyDE-like semantic versions.

Stage 4

Search both worlds

Lexical expansions go to FTS/BM25. Semantic variants go to vector search.

Stage 5

Fuse ranked lists

RRF combines the candidate lists so lexical and semantic evidence can both matter.

Stage 6

Select best chunks

Documents are chunked so reranking focuses on the most relevant slices, not entire files.

Stage 7

Rerank

The reranker scores query-chunk pairs more precisely, which usually improves final ordering.

Stage 8

Return results

The final list is a blend of recall-oriented retrieval and precision-oriented reranking.

Why the BM25 Probe Matters

If the query is qmd no-rerank flag and a single document is obviously dominant, expansion and reranking are wasted effort. The BM25 probe lets qmd skip expensive stages when the lexical signal is already strong.

Why the Pipeline Is Layered

No single technique dominates every query. BM25 is strong for literals. Vectors are strong for paraphrase. Rerankers are strong when you already have good candidates. qmd stacks them because each fixes a different weakness.

3 — ML Techniques

The ML and Retrieval Techniques qmd Uses

Classical IR

BM25 / FTS

Not neural

Great for exact words, file paths, symbols, acronyms, literal phrases, and “the user knows the vocabulary.”

Representation Learning

Embeddings

Semantic

Maps text into dense vectors so similar meaning sits nearby even when the wording changes.

Vector Retrieval

Nearest-Neighbor Search

Recall

Finds chunks whose embeddings are closest to the query embedding. Good for “same idea, different words.”

LLM-Oriented Retrieval

Query Expansion

More compute

Generates multiple search variants so the system can search like a smarter user than the original query.

Retrieval Heuristic

HyDE-like Expansion

Semantic bridge

Generates answer-shaped text, then embeds that. Useful when the best retrieval vector is closer to an answer than to the literal question.

Rank Fusion

RRF

Robust blend

Combines multiple ranked lists without pretending their raw scores are directly comparable.

Document Preparation

Chunking

Critical plumbing

Large files become smaller semantic units so vector retrieval and reranking stay within context windows and stay specific.

Pairwise Scoring

Reranking

Precision

Given a query and a candidate chunk, the reranker asks the expensive but high-quality question: “how relevant is this pair, really?”

Best practical intuition

BM25 says same words. Embeddings say same idea. Reranking says now inspect the candidates carefully. qmd does all three.

Why this counts as ML

The ML parts are embeddings, expansion, and reranking. BM25 is not ML, but it often remains essential because lexical recall is still hard to beat for exact identifiers and literal terminology.

5 — Lev Unlocks

What This Unlocks for Lev

The real opportunity is architectural clarity. Lev does not need one tool to own lexical recall, semantic recall, and final precision at the same time.

Current mental model

One tool, many jobs
  • qmd as unified search box Keyword retrieval, vector retrieval, query expansion, reranking, and tool surface are bundled together.
  • Pros Convenient, local-first, one install, one CLI, good end-to-end default behavior.
  • Cons Harder to swap parts independently, harder to tune per workload, lexical performance tied to qmd’s internal architecture.

Unlocked Lev model

Split retrieval stack
  • Layer 1: lexical retrieval Fast BM25 service for exact terms, filters, and broad candidate generation.
  • Layer 2: semantic retrieval Embeddings and vector search for concept recall and fuzzy discovery.
  • Layer 3: precision layer Reranker or LLM-based evaluator decides final ordering after recall is already good.
Faster first pass

Lev could use a dedicated lexical service for code terms, plugin names, commands, paths, and exact identifiers without invoking the expensive semantic stack every time.

Better workload routing

Some queries are clearly lexical. Some are clearly semantic. Lev can route them intentionally instead of always using a unified heavy path.

Cleaner contracts

Once lexical retrieval is its own layer, Lev can benchmark it, cache it, or replace it without destabilizing embeddings and reranking.

Why BM25 Turbo is interesting

It is compelling as a dedicated lexical layer for Lev. That does not automatically mean it should replace qmd’s entire retrieval stack.

Why qmd still matters

qmd already knows how to do the semantic and precision parts: expansion, vector search, chunk selection, reranking, and agent-facing integration.

7 — Mental Model

The Simple ML Mental Model

One-sentence model

BM25 finds the same words.
Embeddings find the same ideas.
Rerankers judge the best candidates more carefully.

System design consequence

The best retrieval systems do not argue about “keyword search versus AI search.” They compose both. First make recall broad enough. Then make ranking precise enough.

Concrete example: querying for “how does auth work”

BM25 path: looks for docs that literally use words like auth, authentication, login, or file names containing those terms.

Vector path: can retrieve docs about session validation or token middleware even if they do not say auth explicitly.

Expansion path: may rewrite the query into multiple forms like “session validation,” “JWT middleware,” or “login flow.”

Reranker: decides which of the candidate chunks actually explains the mechanism instead of just mentioning adjacent concepts.
