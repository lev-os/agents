Abstract Interface First

A2UI is the contract. Everything else is an adapter.

The interface boundary cannot be tied to json-render, MCP Apps, WebMCP, or any single framework. Those are concrete protocol and runtime adapters below the abstract A2UI layer.

Operating rule

Build the A2UI contract around abstract concerns: shape, state, actions, transport, and host embedding.

  • json-render is one strong render/runtime adapter.
  • MCP Apps and WebMCP are host/transport adapters.
  • Other GenUI frameworks each map only part of the contract.
  • That is why the adapter layer is many-to-many, not one-to-one.
“Our abstract interface should not be tied to a lib.” binding design constraint

Adapter Model

Why the mapping is many-to-many

The clean way to think about this is to split the world into one abstract contract and several adapter families. No concrete system implements the whole stack the same way, so the contract has to decompose into facets.

Protocol / Framework Matrix

Where each thing fits, and what it is not

The key distinction is between abstract contract, host protocol, transport protocol, and render/runtime engine. The confusion comes from treating all of them as if they were the same layer.

Thing Primary role Artifact / shape Where it is strong How it maps under A2UI
A2UI abstract contract Interface facets, not one schema Owns the language of surfaces without tying us to a specific library Defines host, transport, spec, and state/action interfaces. Everything else implements some subset.
MCP Apps host protocol Tool + UI resource + inline host surface Embedding interactive UI inside MCP-capable clients Implements the host surface adapter and part of transport. It does not define your render grammar.
WebMCP web transport / host bridge Browser/web-facing MCP-ish runtime channel Making browser/web surfaces participate in tool/resource flows Implements web transport and browser-session integration. It still needs a render/runtime model above it.
json-render render/runtime engine Catalog, flat spec, state, actions, visibility, patches Guarded composition and interactive UI generation Best current implementation of the spec + state/action side of A2UI. It is not the abstract contract itself.
Other GenUI frameworks alt runtimes Framework-specific schemas, components, state semantics Good for comparison and alternate backends Adapter targets. They will diverge in schema shape, action model, and hosting assumptions, so mapping will never be 1:1.
polymorph / Sofia legacy substrate Primitive tree + templates + renderers Existing asset, design/Pencil path, compatibility Translate existing payloads into the A2UI contract instead of making this the canonical future runtime.

Component Consolidation

How to deal with the 300+ component estate

Consolidation is classification, not flattening everything into one catalog. Use the clean `@kingly/ui` Storybook as the reference surface, then sort the estate into the right buckets.

Reference surface

Kingly UI Storybook on 6007

  • Directly browser-validated in this session
  • Searchable component navigation is working
  • Use this as the clean primitive and recipe reference
Do classify into

Canonical buckets

  • A2UI primitives
  • Composite spec builders
  • Host-only widgets
  • Protocol-only adapters
  • Legacy translators
Do not classify into

One giant 1:1 mirror

  • Do not dump 344 components into the runtime catalog.
  • Do not preserve multiple resolution systems forever.
  • Do not tie the abstract contract to `json-render` specifics.
The catalog should contain the stable abstract language. The 300+ component estate should be mapped to it, not copied into it. consolidation rule

Execution Roadmap

Ordered to preserve abstraction and reduce runtime drift
Phase 0

Codify A2UI as abstract interfaces

  • Split the contract into host, transport, spec, and state/action facets.
  • Write the names of those interfaces before picking one implementation.
Phase 1

Choose the first concrete runtime

  • Use json-render as the first spec/state/action implementation.
  • Keep it explicitly below the abstract contract.
Phase 2

Unify the resolution index

  • Promote one canonical catalog / registry path.
  • Translate polymorph, Sofia widgets, and manual registries into it.
Phase 3

Prove the four real surfaces

  • Approval sheet
  • Diff viewer
  • Metrics board
  • Config form
Phase 4

Wire protocol adapters

  • MCP Apps host adapter
  • WebMCP / browser transport adapter
  • Runtime bridges for state, actions, and event streams
Phase 5

Converge the canvases

  • Collapse multiple canvas entry points behind the same A2UI host boundary.
  • Make runner truth and interface truth match.
