CQRS / ES overlay

CQRS/ES is the skeleton. Disentanglement is one async projection family on that skeleton.

The CQRS/ES landscape says Lev should keep immutable events as authority and derive read models from them. The disentanglement work says one of those read models should recover conversational structure and context heat from mixed chat, terminal, email, and agent data. These are compatible because the graph remains a projection, and soft inference stays downstream.

Canonical rule: event spine = truth, hard lineage = preserved projection, soft disentanglement = async enrichment, heat = view-layer scoring.
Macro model
CQRS / ES Authority lives in append-only events and canonical stores. Files, BD, graph, and memory are projections.
Micro model
Disentanglement Recover containers, reply edges, and heat for data sources that do not provide them directly.
Lev seam
core/graph Already exposes the right runtime shape: extract → classify → validate → apply.
Critical constraint
No source-truth overwrite Soft inferred lineage should become weighted graph links or claims, not canonical event ancestry.

2. Diagram Split
What each diagram contributes

CQRS / ES landscape

Supplies the control-plane rule: events are immutable authority, projections are downstream, dual writes are solved with outbox/CDC/event-first patterns, and read models can be live, inline, or async.

Disentanglement blueprint

Supplies the semantics: threads should be decomposed into container membership, hard lineage, soft inferred edges, and heat.

Overlay result

Lev should treat conversational structure as a graph projection family, not a replacement for canonical storage or the event spine.

4. Lev Components
Where this lands in the codebase
Runtime

core/graph/src/compositor.ts

Keep the main policy seam here. Hard-lineage extraction can stay deterministic; soft-edge classification can become a dedicated classify policy or adapter path.

Authority

core/graph/src/events/jsonl-store.ts

The JSONL event store already supports replay and traceLineage(). That is the right place for re-projectable inference inputs.

Query

core/graph/src/context/projector.ts

Extend scored entities with heat breakdowns instead of pushing explanation logic down into raw event storage.

Adapters

plugins/graph-adapters or new provenance adapter

Source-specific extraction for Slack, Discord, email, terminal, and agent runs belongs in adapters or projector-specific ingestion seams.

Fields that should stay canonical

  • source_entity
  • target_entity
  • causal_parent for hard replay lineage only
  • correlation_id for run/session grouping

Fields that should stay projected

  • soft_parent_id
  • edge_confidence
  • container_confidence
  • heat.* score breakdowns
5. Guardrails
Non-negotiables
Mesh point: event-first authority The overlay only works because both diagrams assume the event spine is canonical and the graph is a projection/query surface.
Mesh point: replayability Soft disentanglement should be rerun from the ledger when the model improves. That is exactly what CQRS/ES wants.
Mesh point: multi-view output The same graph state can feed BD, files, memory, dashboards, and context windows without changing source truth.
Do not write inferred edges back as canonical causality causal_parent should stay for replay-grade lineage. Inferred chat structure belongs in weighted links or claims.
Do not block writes on model inference Hard lineage can be inline; soft disentanglement and rich heatmapping should stay async so the write path remains simple and auditable.
Do not collapse all thread semantics into one field Keep source-native thread IDs, hard ancestry, soft ancestry, and heat as separate dimensions so views stay inspectable.
Short answer: the CQRS/ES diagram explains where authority and projections live. The disentanglement diagram explains what one projection family should compute. They fit by treating conversational structure and heat as replayable async graph projections over the existing event spine.
