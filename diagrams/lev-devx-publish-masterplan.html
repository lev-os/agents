1 — OSS Graybeard Gotchas

Hard-won lessons from Sindre Sorhus, Rich Harris, Evan You, Jordan Harband, TJ Holowaychuk, and the Node.js TSC. These are the landmines.

Gotcha What Happens How to Avoid Severity
workspace:* in published packages npm doesn't understand workspace:*. Consumers get cryptic resolution errors. Changesets transforms these automatically. But test e2e — publish dry-run → install in clean project. critical
"Works in monorepo, breaks after publish" Dev uses symlinks. npm uses resolution. Different systems, different bugs. Create a test-consumer/ project that npm installs published packages and runs smoke tests. Netflix built Niagara for this. critical
npm publish vs pnpm publish npm publish does NOT honor publishConfig for exports overrides. Ships raw .ts files. Lock down CI to only allow pnpm publish. Add a prepack script that validates dist/ exists. critical
Peer dep version hell LangChain's #1 complaint. Multiple @langchain/* packages need exact same core version. Use Changesets fixed groups — core packages version together. Already configured in .changeset/config.json. high
CLI startup time Node.js loads 80+ packages on every lev invocation = 2+ second startup Bundle CLI entrypoint with tsup/esbuild into single JS file for distribution. tsx for dev only. high
Premature package splitting 120 packages = 120 changelogs, 120 version bumps, quadratic maintenance Start with 8-10 public packages. private: true everything else. Split when real consumers need independent versioning. high
Lockfile merge conflicts 80+ packages + multiple contributors = pnpm-lock.yaml merge conflict every PR Use Renovate for automated dep updates. Minimize manual dependency changes. medium
Dual CJS/ESM Wrong exports conditions silently break half your consumers Ship ESM only ("type": "module" already set). CJS consumers can use dynamic import(). medium
curl | sh trust exercise Users executing remote code. One compromised CDN = owned machines. HTTPS only + TLS 1.2+. Checksum verification. Version-pinned URLs. Interactive by default. medium
Jordan Harband's rule: "The hard part isn't the tooling. It's the discipline. Changesets forces explicit communication. Semver forces you to think about what 'breaking' means. private: true forces you to decide what's ready."

3 — Dev Workflow

The Two-Mode Architecture

DEV MODE (monorepo)
pnpm lev --help
  • Resolves to node_modules/.bin/lev
  • tsx respawn shim runs .ts directly
  • Zero build step. Instant feedback.
  • Edit source → run again → changes reflected
PROD MODE (installed)
lev --help
  • Resolves from ~/.local/bin/lev (PATH)
  • Compiled JS bundle via tsup/esbuild
  • Fast startup (~200ms target)
  • Installed via npm i -g @lev-os/lev

Coexistence: pnpm automatically resolves workspace bins first. When you're in the monorepo, pnpm lev runs the dev version. Outside, lev on PATH is the installed version. No shell functions. No env vars. No conflicts.

What pnpm dev Should Do

Fix the broken scripts, then:

01 Build Registry poly scans config.yaml files
02 turbo dev tsc --watch on ~10 packages
03 Ready pnpm lev runs live .ts

The CLI itself doesn't need a watch — tsx runs source directly. The watch is for packages that emit dist/ consumed by other packages.

launchd — Plist exists, points to wrong path

The Unification Plan

User: lev daemon start
  │
  ▼
core/daemon (TS) ── poly routes "lev daemon *" here
  │
  ├─ 1. Register levd in poly registry.yaml
  ├─ 2. pmdaemon start levd (or brew services)
  │
  ▼
crates/levd (Rust) ── actual HTTP server
  ├── /health          → healthcheck
  ├── /status          → running processes
  ├── /declarations    → capability store
  └── /events/recent   → event bus tail

TS daemon = orchestrator. Rust daemon = server. One manages, the other serves. They just need to be wired together. Add levd to core/poly/registry.yaml and fix the launchd plist path.

Unix CLI Philosophy Checklist

  • --json on every command (machine-readable)
  • stdout = data, stderr = diagnostics
  • Exit codes must be accurate
  • Quiet by default, -v for verbose
  • Startup under 200ms (bundle for prod)
  • XDG paths (~/.config/lev/, ~/.local/share/lev/)
  • Composable through pipes
  • --provenance on all npm publishes
