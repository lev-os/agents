AG-UI vs json-render

Different layers, complementary roles

AG-UI and json-render are not competitors. AG-UI is about the agent↔user interaction protocol between frontend and backend. json-render is about the render/runtime model for guarded UI composition.

Thing Primary layer What it standardizes What it does not solve
A2UI abstract contract How A2UI-capable systems describe UI, state, actions, and capability boundaries without tying to one implementation. Does not itself choose a transport, host, or rendering engine.
AG-UI interaction protocol Event-driven agent↔frontend communication, bi-directional state sync, streaming agent execution into user-facing apps. Does not define your component grammar or rendering catalog. It can carry A2UI/UI intents, but it is not the view engine.
json-render render/runtime engine Component catalog, flat spec, state, actions, visibility, validation, patch-based progressive UI updates. Does not define the abstract contract above it, nor the host shell, nor workspace/layout/policy routing.
MCP Apps host/embed protocol How an MCP tool/server returns interactive UI resources that render inside compatible MCP clients. Does not define your UI grammar or your application state model.
WebMCP web transport / browser bridge How web/browser surfaces can participate in MCP-like interactions and AI-optimized web flows. Still needs a render/runtime model and an abstract UI contract above it.
Other GenUI frameworks alternate runtimes Framework-specific composition semantics, state models, and host assumptions. They vary too much to become the abstract contract. They are adapter targets.
AG-UI moves events json-render composes UI A2UI names the boundary MCP Apps embeds the result

Adapter Lattice

Why the adapter layer is not 1:1

The same abstract A2UI surface may use AG-UI for event transport, `json-render` for runtime composition, and MCP Apps for host embedding. Another surface may use a different runtime and still satisfy the same contract.

Milestones, Corrected

Where the “four surfaces” came from

Approval sheet, diff viewer, metrics board, and config form came from the first proving-ground milestone. They were never meant to be your north star; they were the smallest set of real surfaces that would validate the abstract contract plus runtime path.

Milestone A

Proving-ground surfaces

  • Approval sheet
  • Diff viewer
  • Metrics board
  • Config form
  • Purpose: prove host, spec, state, action, and transport loops.
Milestone B

Estate-powered compositions

  • Use the broader Kingly UI / migrated estate to build much richer surfaces.
  • Pull in dashboards, scene/canvas, chat, ops, review, diagnostics, media, and settings recipes.
Milestone C

Natural-language interface generation

  • User asks for an interface.
  • Lev GenUI composes the right surface from the estate.
  • The surface is interactive and can talk back through AgentPing when needed.
