Design Intent
Both config-based AND file-based routing are intentional. core/*/src/commands/ = file-based for lev core. .lev/config.yaml = config-based for project extensions. .lev/flows/*.flow.yaml = fractal auto-discovery. They are complementary, not competing. But the bridge between them was never finished.
2 — The Three Systems
Spec Intent
  • Config.yaml poly.sdk declares commands with handlers
  • Registry-builder merges config + filesystem into unified manifest
  • Proxy-factory consumes manifest → generates proxy binaries
  • Fractal ownership patterns auto-discover *.ts across workspace
Primary issue
Proxy-factory.js was declared but never created. The pipeline is half-built.
Code Reality
  • discoverCommands() walks core/*/src/commands/ at runtime
  • Dynamic import(handler) dispatches to found files
  • Config.yaml poly.sdk exists but nothing reads it for dispatch
  • Registry-builder runs on postinstall → output never consumed
Primary issue
Works fine for core commands. No path for project-local config-driven extensions.
Decision Required
  • Option A: Finish the pipeline (proxy-factory, wire registry consumption)
  • Option B: Formalize filesystem-only, kill dead spec references
  • Option C: Hybrid — filesystem for core, config for project extensions
Recommendation
Option C aligns with the original fractal config intent. See plan-research below.

4 — How It Actually Works

The Real CLI Dispatch Path

core/poly/bin/lev → surfaces/cli/index.ts
$ lev daemon status --json 1. bin/lev (respawn shim) → Ensures --import tsx in NODE_OPTIONS → execFileSync to surfaces/cli/index.ts 2. surfaces/cli/index.ts → Parse argv: command="daemon", args=["status","--json"] → Check hardcoded special routes (work, review, loop, autodev...) → Not special → discoverCommands() 3. discoverCommands() — surfaces/cli/discovery.ts → Walk core/*/src/commands/ for .ts files → Walk plugins/*/src/commands/ for .ts files → Return { commands: { daemon: { path, module } }, aliases } 4. Dispatch → import("core/poly/src/commands/daemon.ts") → handler(["status", "--json"]) Config.yaml poly.sdk section? ← NOT CONSULTED Registry-runtime.yaml? ← NOT CONSULTED FractalOwner/FractalEntry? ← NOT CONSULTED
Key Insight
The filesystem walk works perfectly for core commands. The gap is for project-local extensions — there's no mechanism for .lev/config.yaml to register additional commands that discoverCommands() would find.
5 — The Missing Pieces

What Was Never Built

proxy-factory.js

Declared in config.yaml line 349. Supposed to consume the registry manifest and generate proxy binaries for poly, watch, and kernel.

File does not exist. Not even a stub. The depends: [registry] chain is broken.

Registry Consumer

registry-builder.js successfully generates .build/registry-runtime.yaml on every pnpm install. 100+ entries with fractal owners and routes.

Nothing imports or reads this file. The CLI uses filesystem discovery instead. The registry is an orphaned artifact.

Fractal Governance Spec

FractalOwner and FractalEntry are well-typed in core/domain. They define ownership, glob patterns, default configs, and route resolution.

Zero documentation outside TypeScript JSDoc. No spec explains the governance model. No design doc explains why both config + filesystem exist.

Coherence Tests

Good test coverage for discoverCommands() (8+ tests). Good tests for CLI dispatch, aliases, blocklists.

Zero tests validate that config.yaml declarations match discovered commands. Zero tests for fractal auto-discovery round-trip. Zero tests for registry consumption.

6 — Open Questions

Decisions That Need Answers

  1. Was proxy-factory deliberately deferred or abandoned?
    Config.yaml still declares it. No commit removes it. No decision doc explains why it wasn't built. Is this a "get to it later" or a "we changed our mind"?
  2. Should config.yaml poly.sdk drive dispatch?
    Currently it's documentation cosplaying as config. Either wire it into discoverCommands() or remove it and use a real docs file. The current state creates false confidence.
  3. How do project-local extensions register CLI commands?
    The fractal config vision says .lev/config.yaml can declare commands. The CLI doesn't read it. Is there a planned hook point in discoverCommands() for project-scoped discovery?
  4. What role does registry-runtime.yaml play?
    It's generated on every install. 100+ entries. Nothing reads it. Should the SDK's call() and runExecutionRequest() resolve through it?
  5. Is the poly:binary-compiler handler real?
    Config.yaml declares it for fractals.binaries. SEA option exists in defaults. But bin/lev is a raw TypeScript respawn shim, not a compiled binary. Is compilation planned for production?
  6. Why didn't drift detection catch this?
    The drift scan grep'd for presence (file exists + config entry exists = PASS). It should have loaded the spec contract and checked: "does proxy-factory.js exist?" (NO) and "is registry-runtime.yaml consumed?" (NO).

8 — Path Forward
Recommendation: Research First, Then Decide

A plan-research-poly-build-pipeline.md entity has been created (P2, ready). It will produce a decision document answering: finish the pipeline, simplify it, or go hybrid.

The research should focus on three things:

1. Read registry-builder.js end-to-end — understand the merge logic and what it produces.

2. Clarify the user's intent for fractal config composition (config + filesystem = complementary).

3. Decide: wire discoverCommands() to also read project-scoped config, or formalize the filesystem-only approach and update the spec.
