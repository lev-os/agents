Plain-language explainer

gstack is a packaged software factory for one operator.

In simple terms, gstack turns a coding agent into a small specialist team. Instead of one generic AI chat, you get named roles: someone to rethink the product, someone to lock the architecture, someone to do code review, someone to QA the app in a real browser, and someone to ship. Underneath that role layer is one substantial runtime feature: a fast, persistent browser daemon that gives the agent durable eyes and hands.

01

The short answer

If you asked “what is gstack?” in one sentence: it is a repo of structured AI workflow skills, plus a headless browser system, designed to make an agent behave more like a disciplined engineering org than a raw autocomplete.

What it is

A role-based workflow layer

`gstack` ships named skills like /office-hours, /plan-eng-review, /review, /qa, and /ship. Each one gives the agent a specific job and tone instead of one generic prompt.

What it does

Pushes work through a repeatable sprint loop

It helps a user move from idea to plan to implementation to review to QA to release. The output of one skill is meant to feed the next so less falls through the cracks.

What makes it feel real

The browser is not a toy add-on

The browser subsystem keeps tabs, cookies, and state alive between commands. That lets the QA flow act like an actual tester instead of a fresh browser boot on every step.

02

How the sprint works

`gstack` is easiest to understand as a pipeline. It is not “some random commands.” It is an ordered loop for building and landing features.

Think

/office-hours

Reframe the problem and make sure the request is actually the right product wedge.

Plan

/plan-ceo-review

Push on ambition, scope, and the 10-star version of the idea.

Plan

/plan-eng-review

Lock architecture, data flow, tests, and failure modes.

Build

Agent implementation

After the reviews, the coding agent writes the feature with clearer context.

Review

/review + /codex

Find bugs, challenge the code, and get a second model’s opinion.

Test

/qa

Open a real browser, click through flows, and fix regressions.

Ship

/ship + /retro

Land the work and look back at what actually happened.

03

The specialist team it gives you

The point is not “more commands.” The point is using different mental postures on the same work. `gstack` packages those postures into named specialists.

Group Representative skills What they do Why that matters
Problem framing /office-hours /plan-ceo-review Reframe the product, challenge assumptions, find the better wedge. Stops the agent from coding the literal request when the real opportunity is bigger or narrower.
Technical planning /plan-eng-review /plan-design-review Work through architecture, diagrams, design quality, and missing edge states. Pushes crucial decisions earlier, when they are cheaper to fix.
Code scrutiny /review /codex Run staff-engineer review plus second-opinion or adversarial review from another model. Creates a more skeptical review loop than “the same model approves itself.”
Product testing /qa /qa-only /browse Test live sites in a real browser, find bugs, and optionally fix them. This is what makes the agent feel operational, not just textual.
Release and hygiene /ship /document-release /retro Land changes, update docs, and reflect on what shipped. Turns the workflow into a loop instead of a one-off coding session.
Safety tools /careful /freeze /guard Warn before destructive commands and limit edits to a safe scope. Keeps fast agent work from becoming reckless agent work.
04

The real engineering underneath it

A lot of the repo is markdown skill logic, but one part is an actual runtime system: the persistent browser service.

Agent

Claude / Codex / other skill host

Runs a skill that needs to inspect a page, click something, or verify a flow.

→
CLI

Compiled browser binary

Reads a state file, sends commands to a local server, and restarts the daemon when the version changes.

→
Server

Bun HTTP daemon

Accepts commands, keeps session state alive, and routes them into Playwright and Chromium.

→
Browser

Persistent Chromium session

Keeps cookies, tabs, and local state alive so the agent can work in sub-second loops instead of cold-starting every time.

Why it matters

State survives between commands

That means real login sessions, real browser history, and much faster test/debug loops. The agent can actually continue a flow instead of constantly starting from scratch.

Why it feels robust

Refs, restart logic, logs, and auth

The browser layer includes ref-based element targeting, stale-ref detection, buffered logs, localhost auth tokens, and auto-restart when the binary version changes.

05

Why people use it

Clarity

It gives the agent a job, not just a prompt

A named specialist is easier to reason about than “be helpful.” You know why the agent is acting a certain way and what kind of critique it should provide.

Coverage

It makes gaps harder to ignore

By splitting the flow into planning, review, QA, and release, it becomes harder for a feature to bypass testing, code review, or documentation.

Speed

The browser makes agent QA believable

The agent can inspect pages, click through flows, and keep context alive. That is a different class of capability from static code review alone.

06

What it is not

Not this

Not a graph-native runtime

`gstack` does not give you an auditable execution graph, a typed event spine, or a declarative workflow substrate. It is workflow packaging, not a universal runtime.

Not this

Not adaptive multi-agent orchestration

Its “many agents” story is mostly isolated sessions and named specialists. That is powerful, but it is different from a CDO-style adaptive synthesis graph.

Not this

Not a generic platform-agnostic substrate

It is deeply shaped by skill-host workflows, local files, and home-directory state. That makes it pragmatic, but also very specific.
