Skill overview

Visual Explainer turns technical answers into polished HTML pages.

It is not “just a diagram tool.” It is a rendering workflow for taking architecture, plans, reviews, comparisons, and dense technical prose, then restructuring them into a browser-native explanation with hierarchy, typography, navigation, and visual intent.

Input Architecture notes, plans, recaps, diffs, audits, docs, or anything that benefits from visual structure.
Output A self-contained .html file designed to be opened in a browser.
Bias Never fall back to ASCII tables or plain terminal diagrams when the skill is active.
Location Writes to ~/.agent/diagrams/ and opens the result immediately.
1. What It Is
Role

A visual explanation engine

The skill packages a technical explanation as a visual page, usually with cards, timelines, tables, flow layouts, or diagrams instead of raw prose.

Medium

HTML first

The output is browser-native, self-contained HTML. Fonts, styles, layout, and optional JS all live in one file.

Browser required
Default behavior

Visual by default

Even prose-heavy content gets reshaped into visual sections. Paragraphs become accent elements, not the whole page.

2. Best Uses
Strong fits
  • Architecture overviews and system maps
  • Diff reviews, plan reviews, and project recaps
  • Comparison tables, audits, and status matrices
  • Documentation that would be clearer as cards or flows
Behavior
  • If a table is large, it belongs in HTML, not terminal box drawing.
  • If the story has sections, it should usually get a nav and layout hierarchy.
  • If a diagram engine helps, use it. If not, CSS grid and semantic HTML win.
Shortcut summary

Use it any time a plain-text answer would technically work but would force the reader to do the layout and structure mentally.

3. Workflow
Step 1

Think

Commit to an audience, content type, and aesthetic instead of defaulting to generic “dark UI.”

Step 2

Structure

Choose the right medium: cards, Mermaid, tables, dashboards, timelines, or a hybrid.

Step 3

Style

Pick real typography and a coherent palette. Avoid violet-gradient AI defaults.

Step 4

Deliver

Write a self-contained file under ~/.agent/diagrams/ and open it automatically.

Step 5

Check

Run the squint test, swap test, both-theme check, overflow check, and browser-open check.

4. Output Contract
Write target

Persistent diagrams folder

Use descriptive filenames inside ~/.agent/diagrams/ so the output can be reopened or shared later.

Open action

Always view it

The workflow is incomplete if the HTML is written but not opened in the browser.

Fallback rule

No ASCII escape hatch

If the skill is loaded, the answer should stay visual. Terminal tables are explicitly the wrong output.

5. Guardrails
  • Use distinctive fonts and a real palette Named pairings and deliberate color choices are part of the skill, not decoration.
  • Use semantic HTML Tables should be real tables, code should preserve whitespace, and cards should have hierarchy.
  • Let the format match the content Use Mermaid when routing matters, CSS when card content matters, and hybrids when both matter.
  • No purple-gradient AI slop Avoid violet/indigo defaults, gradient headline text, and neon cyan-magenta dashboards.
  • No emoji-led section chrome The skill prefers typography, labels, dots, and layout over generic emoji signaling.
  • No “looks nice, says little” output If the page is pretty but misses the actual explanation, it fails the contract.
Short version: it is a visual delivery discipline. It takes technical content, chooses the right browser-native form, and enforces enough aesthetic constraints to avoid generic AI output.
