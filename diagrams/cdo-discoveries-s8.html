II

The Advocate's Unified Position

All six bucket reports converge on the same underlying architecture. The agreement is unusually strong across disparate concerns.

Advocate

The Lev architecture is a four-plane system with a five-type graph topology, a single protocol-agnostic event envelope, and a single executor with a handler registry.

Plane separation is non-negotiable. Every bucket confirms D33 (shearing layers): the control plane (FlowMind declarations), execution plane (executor + adapters), state plane (graph entities), and causality plane (LevEvent bus) must change at different rates. Any coupling inversion — fast layer depending on slow layer — is a structural defect, not a style preference.

B6 is not a "status enum mismatch." It is a coupling inversion where the execution plane's internal vocabulary leaked into the promotion check in the orchestration plane.

Advocate

Five canonical GraphNodeTypes — not nine, not three

The 9→5 reduction (agent, function, subgraph, terminal, hitl) has unanimous support across B1, B3, B4. The four removed types are not node types — they are execution policies masquerading as topology.

gate → compile-time shorthand that expands to conditional branches.
parallel + loop → OrchestrationOverlay annotations on subgraph nodes.
redirect → a next: edge reference.

Principle: GraphNodeType answers "what is this node," not "how do I traverse it."
Advocate

LevEvent is already 90% CloudEvents-compatible

The delta is surgical — add ProtocolDescriptor to LevEventMetadata (optional fields, non-breaking). The existing A2A dispatcher becomes the first ProtocolAdapter<AgentJob, unknown> implementation. AgentJob is promoted to core/domain as the Lev-native job payload type. Zero rebuild required; the code is promoted and renamed, not replaced.

Advocate

One executor with a handler registry replaces four diverging executors

Four executors accumulate bugs in four places. The NodeTypeHandler interface + AdapterRegistry replaces GraphFlowExecutor's switch/case dispatch. Stubs ship immediately (no regression), real handlers replace stubs incrementally. The FlowmindExecutor stays deprecated-but-running until GraphFlowExecutor proves full parity.

Advocate

CDO dual-brand: FlowMind = engine, CDO = stylesheet library

FlowMind is the engine brand (developer-facing runtime, compiler, kernel). CDO is the capability brand (product-facing pattern library, deliberation profiles). CDO stylesheets (production.cdo.yaml, development.cdo.yaml) are named FlowMind profiles in the fractal config system. No new primitives required.

III

The Critic's Challenges

Where the Advocate's position has structural risk. Three coupling inversions, two hidden dependencies, and one cross-bucket disagreement.

Critic — Coupling Inversion #1

Node type reduction requires executor completeness, but executor completeness requires node type stability

Reducing from 9 to 5 types requires updating GraphFlowExecutor to handle the new representations. But it currently cannot handle subgraph, hitl, parallel, or redirect — the very types involved in the reduction. If you reduce while the executor still doesn't handle subgraphs with orchestration overlays, you get silent failures.

Resolution: Do not reduce types until stub handlers for all 5 canonical types are registered. The ordering is B1 Phase 1 (stubs) → B4 node type reduction → B1 Phase 2 (real handlers).
Critic — Coupling Inversion #2

B2 says 3 types, B4 says 5 types — the only cross-bucket disagreement

Protocol bucket (B2) proposed 3 types for A2A job mapping simplicity. Topology bucket (B4) proposed 5 for completeness. This is a real disagreement.

Resolution: B4 wins (5 types). subgraph is a structural containment primitive — it's a namespace boundary, not an agent invocation. hitl is a category distinction, not a provider choice. B2's "3 types" is an A2A mapping heuristic, not a topology prescription.
Critic — Coupling Inversion #3

B6 fix: source AND bridge, not just bridge

Fixing only the promotion check (isTerminalSuccess()) is a patch that allows the semantic mismatch to propagate further. The true fix should normalize at the source (session-executor emits 'success') AND add the bridge at the promotion check as belt-and-suspenders.

Critic — Hidden Dependencies

Two things the Advocate underweighted

1. The decompiler (B7) is a prerequisite for node type reduction validation. You cannot validate that 9→5 is semantically lossless without a roundtrip test: parse(serialize(doc)) must produce a structurally equivalent GraphDocument. Not nice-to-have — primary validation tool.

2. hitl handler requires an agreed LevEvent schema for agent.hitl.requested. This sits at the intersection of B1 (executor), B2 (protocol), and B4 (topology). Build the event schema first, then the handler.

The Pragmatist's Assessment

What ships this quarter. What Sonnet can autodev overnight. What requires Opus-level design judgment.

Pragmatist

Wave 1 is 3-4 days of autodev work. Every task is Sonnet-safe.

B6 status fix (30 min), A16 YAML rename (1 hr), parser compat shim (30 min), CdoProperties deletion (30 min), decompiler MVP (1-2 days), terminal + redirect stubs (1 hr), hitl stub (2 hr), LevEvent metadata extension (1 hr), UsageReport type alias (1 hr).

Key insight: Wave 1 is entirely mechanical. No design judgment required. Ship it overnight.
Pragmatist

The dominant blocker for autodev is not complexity — it's the absence of interfaces

The NodeTypeHandler interface in Wave 2 is the unlock: once it exists, every handler is an independent autodev task. The ProtocolAdapter interface is the same pattern for the protocol layer. Each interface definition is an Opus/human task that enables a dozen Sonnet subtasks.

Pattern: One Opus task (define interface) unlocks twelve Sonnet tasks (implement handlers).
Pragmatist

D-W10 federation — gated on protocol envelope stability.

VI

The Wild Card's Provocations

What everyone else was too polite to say. Four provocations, each with a verdict.

Wild Card

Is 9→5 too conservative? Should we go to 3?

A pure 3-type model (agent, function, terminal) would eliminate subgraph (replaced by inline graph composition) and hitl (modeled as agent with provider: 'human').

Verdict: Do not go to 3. subgraph is the host type for the entire orchestration overlay design — without it, parallel and loop have nowhere to live. hitl deserves first-class status because human-in-the-loop is a category distinction, not a provider choice.
Wild Card

Should LevEvent be the ONLY type? Full monism.

What if every message — agent invocations, function calls, HITL pauses, loop ticks — is a LevEvent? AgentJob dissolves into LevEvent<{type: 'orchestration.job.dispatch'}>.

Verdict: Architecturally correct, operationally impractical. AgentJob has 23 tests against typed fields — dissolving it loses TypeScript type safety at the dispatch boundary. The pragmatic middle path (keep AgentJob as first-class, carry as LevEvent.data) gives 90% of the observability benefit. Revisit at Wave 4.
Wild Card

What if CDO stylesheets are the killer feature?

Every bucket treats CDO stylesheets as a minor roadmap item. But no agentic framework has a "profile system" that lets you swap from development.cdo.yaml (2 agents, loose gates) to production.cdo.yaml (5 agents, mandatory HITL, strict gates) with a single config change.

A developer who has never heard of FlowMind understands immediately: "I can tune my agent's thinking style like a stylesheet."

Implication: CDO stylesheet examples should ship as early as Wave 2, even before the full profile resolver. Documentation-first for this feature, not code-first.
Wild Card

What paradigm shift are we missing?

The six reports collectively describe an agentic programming model. But none of them makes observability the primary user-facing value proposition.

The events.jsonl file, the token:usage events, the episode crystals — these are the foundation of a system where you can ask "why did the agent make that decision?" and get a causal answer.

The missing framing: FlowMind is not just CSS for agents — it is CSS for agents with browser DevTools. The causality plane (LevEvent bus) is the DevTools. CDO stylesheets are the stylesheets. The REPL is the inspector. Episode crystals are the memory profiler. "Finally, debuggable agents."
Wild Card

What would a competitor copy first?

A well-resourced competitor reading these reports would immediately notice:

1. hitl as a first-class topology primitive, not a workaround
2. Shearing layers enforcement — a CI-enforced architectural immune system
3. The 4-plane separation — genuine plane separation, not layered monolith
4. CDO deliberation patterns (thinking-parliament, anti-groupthink) — concrete, runnable AI reasoning

Defense: They'd copy the thinking-parliament first — most demonstrable, most marketable, least architected. You can fake it without the runtime. The defense is making it executable only through FlowMind, proving infrastructure value. Ship Wave 1-2 before demoing CDO publicly.

Cross-Bucket Convergence

What emerged when 6 isolated agents looked at the same system from 6 different angles.

B1B2B3B4B5B6

B6 status fix is the universal blocker

All 6 agents independently flagged this. Not because they talked to each other — they can't. Because it genuinely blocks every downstream validation path.

B1B3B4

9→5 node type reduction is unanimous

Execution, vernacular, and topology buckets all arrived at the same 5 types independently. The principle "what IS it" vs "how to traverse it" converged from three angles.

B1B3

4 executors → 1 executor is structural, not cosmetic

Execution model and vernacular both identify the 4-executor divergence as the root cause of accumulated bugs. The handler registry is the escape.

B2B5

LevEvent is already the right envelope

Protocol and context buckets both conclude that LevEvent needs minimal extension, not a replacement. The delta is surgical metadata fields.

B5B6

CDO stylesheets compose through existing fractal config

Context and branding both confirm: named profiles in the fractal config system. No new primitives required. The system already supports cascade semantics.

B2B4

The one disagreement: 3 types vs 5 types

Protocol wanted 3 for A2A simplicity. Topology wanted 5 for completeness. Critic resolved: 5 types, with A2A adapter mapping agent→job.

VIII

The Deliberation Verdict

The architecture is correct. The execution sequence is the only thing requiring significant disagreement, and the Critic's sequencing correction resolves it.

1. 5 canonical GraphNodeTypes. Gate → compiler shorthand. Parallel/loop → OrchestrationOverlay. Redirect → next: edge.

2. B6 fix is Wave 1, Step 1. Source normalization AND bridge normalization. Both.

3. LevEvent + ProtocolAdapter is the protocol architecture. A2A dispatcher becomes reference implementation.

4. One executor with handler registry. Interface locked before handlers registered.

5. Dual-brand: FlowMind = engine, CDO = patterns. CDO stylesheets = named fractal config profiles.

6. Observability is the primary user-facing value. Every node emits a LevEvent. "Finally, debuggable agents."

7. Shearing layer enforcement (D33 lint rules) is a Wave 3 infrastructure investment that pays dividends across the entire timeline.
