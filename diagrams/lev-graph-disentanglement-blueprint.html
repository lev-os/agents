Lev graph blueprint

Thread disentanglement is a graph projection problem, not a chat-only model.

The research on IRC, Slack, Discord, email, terminal traces, and agent sessions all converges on one stable idea: do not model “thread” as one vague primitive. Model container membership, causal edges, and confidence/heat separately, then materialize thread views from that.

For core/graph: ingest events first, preserve hard lineage, infer soft lineage only for unresolved cases, and feed the result into ContextProjector.

1. Core Thesis
What the research actually says
Primitive

Do not store “thread” as a monolith

A thread is a view over two different facts: which events belong together, and which events cause or reply to which others.

When the platform helps

Email, Slack, Discord, LangSmith, and OTel already expose hard edges or containers. Treat those as constraints, not hints.

When the platform does not help

IRC, mixed terminal logs, and many agent traces need reply-link inference or session boundary estimation on the leftover unresolved stream.

2. Research Arc
Algorithm lineage
2008–2010

Pairwise reply scoring + graph clustering

Elsner/Charniak establish the base pattern: score candidate reply links in tangled IRC chat, then recover conversations as graph structure.

2019

Better data changes the field more than cleverer decoding

Kummerfeld’s large corpus makes evaluation more trustworthy and exposes how noisy older benchmarks were. This is still the canonical reference stack.

2021

Pretrained encoders + global matching

BERT with handcrafted structure remains strong; multitask and bipartite matching improve thread reconstruction beyond greedy local link choice.

2021–2022

Low-label and structure-aware models

Co-training, speaker-aware structure, and contrastive learning push performance, but all still depend on preserving explicit structure when it exists.

2023–2025

Domain transfer is still brittle

Drama, software engineering chat, and newer backbone comparisons show the same problem: the difficult part is not only local text understanding, but cross-domain discourse and conversation boundary drift.

3. Source Signals
What each source family contributes
Source Native signal Graph treatment Confidence lane
Email Hard edge Message-ID, References, In-Reply-To Store ancestry as direct lineage edges; use JWZ-style placeholder handling for missing parents. Very high unless headers are malformed.
Slack Hard container thread_ts Parent message anchors thread container. Do not re-infer if the field exists. High for threaded replies, low for channel chatter outside threads.
Discord Hard edge message_reference and explicit threads Keep reply edges and thread-channel containers separate; Discord has both. High when message content and history are available.
IRC Soft edge timing, mentions, lexical continuation Run reply-link inference over a rolling window; derive conversation nodes from recovered edges. Medium and heavily dataset/domain dependent.
Terminal sessions Hard session PTY, process group, markers, command/output sequence Sessionize first, then infer subthreads only if one terminal session contains concurrent tasks. High for session boundaries, medium for within-session task forks.
Agent runs Hard lineage thread_id, checkpoints, span links Preserve causal graph directly from run metadata; inference should be the fallback, not the default. High if every child run inherits lineage fields.
4. Lev Graph Mapping
How this should land in current core/graph
Current contract fit

Use the existing event spine

  • LevEvent remains the ingestion envelope.
  • GraphEvent.causal_parent becomes first-class for message lineage.
  • GraphReader.traverse() and traceLineage() become native read paths for reconstructed threads.
What to add
  • Event payload fields: container_id, hard_parent_id, soft_parent_id, edge_confidence, platform, actor_id.
  • Projection links: belongs_to, replies_to, forked_from, continued_after, span_link.
  • Container entity: use episode now or add conversation/thread at V1+.
Stage 1

Normalize

Convert email/chat/terminal/agent records into canonical LevEvent envelopes with platform metadata.

Stage 2

Sessionize

Derive hard containers first: thread, channel-topic, PTY session, run thread, or event batch.

Stage 3

Preserve hard edges

Apply reply/reference ancestry exactly when the platform provides it. No ML override on known lineage.

Stage 4

Infer soft edges

Run disentanglement only on unresolved windows. Emit weighted links instead of mutating raw truth.

Stage 5

Project views

Materialize thread views, lineage views, and heat-ranked context windows from graph state.

5. Context Heatmapping
Heat is not attention
Use

Structural heat

Recency, container continuity, hard-parent ancestry, and graph neighborhood density. This can be computed without model internals and should be the default production signal.

Use

Attribution heat

When an LLM or ranker is involved, use attribution-style methods or perturbation-based responsibility tests, not raw attention scores.

Do not use alone

Attention heat

Useful for debugging. Weak as an explanation contract. It should never be the only heat source in Lev graph decisions.

Suggested projector extension

Extend ScoredEntity, not the whole runtime

The existing ContextProjector already scores entities across past/present/future. Add a heat decomposition so the score becomes inspectable:

  • heat.recency
  • heat.lineage
  • heat.container
  • heat.retrieval
  • heat.attribution
Result

Explainable context assembly

The projected context window can say not only “this entity is hot,” but why it stayed hot: recent reply ancestry, session continuity, semantic relevance, or explicit tool-run lineage.

This is the seam between disentanglement research and practical Lev context selection.

6. Delivery Path
What to do now vs later
Phase 1: deterministic ingestion Build adapters that emit canonical lineage fields for Slack, Discord, email, terminal, and agent traces.
Phase 2: soft-edge projector Run disentanglement only where hard edges are absent. Emit weighted graph links, not destructive rewrites.
Phase 3: heat-aware context views Drive ContextProjector and lineage views from the combined hard+soft graph.
Do not make graph the source of truth Keep with the current Lev direction: event log and canonical stores are authoritative, graph is a projection/query plane.
Do not run ML over everything If the platform already knows the parent or thread container, inference is wasted risk.
Do not treat attention as explanation Attention visualizers are inspection tools. Heat for production needs explicit scoring or attribution evidence.
Short implementation read: event-first ingestion, graph projection, hard-edge preservation, soft-edge inference only where needed, then heat-ranked context assembly.
