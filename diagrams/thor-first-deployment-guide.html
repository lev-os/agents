Jetson Thor to First Deployment

The shortest reliable path from a Mac is: create the Thor installer USB on macOS, flash the dev kit directly, finish first-boot networking, install JetPack components on the device, then deploy this repo from the Mac over SSH.

1. Choose Path

Do not over-engineer the host side yet. Thor’s current official quick start explicitly supports a bootable installation USB made from a Mac, Windows, or Linux machine. That makes the USB installer the correct first attempt, not UTM and not Docker.

Path Use It When Why Status
Mac-native Thor USB installer Fresh setup, first flash, fastest path to a working Thor NVIDIA says Thor can install BSP from a bootable USB without a host Ubuntu PC recommended
UTM + Ubuntu 24.04 You later need SDK Manager or custom flash.sh workflows UTM supports USB device connection to VMs, but it adds another layer to debug fallback
Docker Desktop on macOS Only if you already solved USB/IP and know exactly why you need it Docker Desktop does not support direct USB passthrough; USB/IP is a workaround, not the first-line Thor flash path avoid first
Decision: Start with the USB installer. Only build a UTM Ubuntu VM if you later need host-driven flashing or BSP customization beyond the stock Thor installer flow.

2. Mac Prep

Everything in this section happens on the Mac. The goal is to leave the Mac with exactly two things ready: the bootable Thor installer USB and a local repo checkout that can deploy to Thor once networking is up.

Need

Thor installer media

Download the latest Thor Jetson ISO from NVIDIA’s Thor quick start / JetPack download flow. The guide currently references the Jetson ISO (r38.4.0) path for the JetPack 7 era.

Need

Balena Etcher

Local path: /Applications/balenaEtcher.app. Official download: balenaEtcher.

Need

Deployment workspace

Keep this repo local on the Mac. The canonical deployment model here is Docker-first build on Mac, native runtime on Thor via SSH/rsync.

What to have on hand
  • A 16 GB or larger USB flash drive
  • Thor dev kit, power supply, display or headless serial path
  • USB keyboard and mouse for easiest first setup
  • Ethernet cable if you want stable first-deploy networking

3. Create the USB Installer

This is the part where people lose time by dragging the ISO into Finder. Thor’s installer must be written as a bootable USB, not copied as a normal file.

01

Open Balena Etcher and select the downloaded Thor Jetson ISO.

02

Select the target USB flash drive carefully; it will be overwritten.

03

Flash the USB and let Etcher validate the write before ejecting it.

04

Label the USB so it does not get confused with ordinary install media.

Critical: NVIDIA explicitly warns that this is not a “Live USB” flow. The USB exists only to install the Thor BSP onto NVMe.

4. Flash Thor

For a first-time board, use the monitor-attached path unless you already know the Thor headless serial path. It removes several variables while you are still getting the hardware stable.

Monitor-attached path
  • Attach HDMI or DisplayPort monitor.
  • Attach keyboard and mouse.
  • Insert the bootable Thor USB into USB-A or USB-C.
  • Power Thor from the bundled USB-C supply.
  • Press the power button and let it boot from the USB stick.
What you select
  • If needed, hit Esc repeatedly to inspect UEFI boot order.
  • Select the Thor installer from the USB stick.
  • Choose Flash Jetson Thor AGX Developer Kit on NVMe.
  • Then choose the current Thor NVMe flash option for the ISO version you wrote.
Timing

NVIDIA says the BSP installation itself takes a little under 10 minutes, followed by an automatic UEFI firmware update and another reboot.

Easy-to-miss failure

Starting with newer ISOs, UEFI may keep USB first in boot order after install. Remove the installer USB after the firmware update sequence or you can loop back into the installer.

5. First Boot, Networking, and SSH

Your first deployment is blocked until the Mac can reach Thor over SSH. So this section is not optional housekeeping; it is the bridge from hardware setup to actual software rollout.

OEM config

Create the first user

During oem-config, set the username, password, timezone, and hostname. This guide assumes a deploy user like naac.

Network

Prefer wired Ethernet

Headless CUI setup uses enP2p1s0: Ethernet PCI. Let DHCP assign an address first, then reserve it or convert to static netplan later.

SSH

Turn reachability into a stable alias

Install and enable openssh-server if needed, copy your key, then add a stable host entry on the Mac once the board’s IP is known.

Thor-side commands
on Thor
sudo apt update
sudo apt install openssh-server
sudo systemctl enable --now ssh
hostname -I
Mac-side commands
on Mac
ssh-keygen -t ed25519
ssh-copy-id naac@<thor-ip>
ssh naac@<thor-ip>

6. Install JetPack Components and Satisfy the Thor Bootstrap Gate

Flashing the BSP is not the end. To get from “Thor boots Ubuntu” to “this repo can deploy,” there are two more layers: JetPack components on Thor, and the deployment prerequisites this repo expects.

JetPack components

NVIDIA’s package-management path says to install JetPack components on the device itself once Jetson Linux is running:

on Thor
sudo apt update
sudo apt install nvidia-jetpack
Repo bootstrap gate

The repo’s first Thor deployment gate is VG-14. It requires uv, pmdaemon, curl, /opt/naac/releases, /opt/naac/models, and /etc/systemd/system/pmdaemon.service.

Important repo reality: this repo has canonical deploy and rollback scripts, but it does not yet expose a single authoritative “bootstrap Thor host” script in the checked-in operational path. Treat VG-14 as the stop/go gate before your first deploy attempt.
Gate to run from Mac
VG-14 bootstrap check
ssh naac@<thor-ip> 'command -v uv && command -v pmdaemon && command -v curl && test -d /opt/naac/releases && test -d /opt/naac/models && test -f /etc/systemd/system/pmdaemon.service'
Runtime contract from this repo
  • config/profiles/thor_prod.yaml expects shared_memory transport.
  • Whisper backend path is /opt/naac/models/whisper-atc-tensorrt.
  • deploy/pmdaemon.yaml expects services to run from /opt/naac/current.

7. First Deployment From the Mac

Once Thor passes bootstrap and SSH works, the first release is driven from this Mac. The canonical path in this repo is: build and validate locally, eject artifacts, rsync them to Thor, activate via symlink swap, restart pmdaemon, then health-check.

Predeploy

Run the heavy path at least once

The repo’s full compose-backed end-to-end gate is just test-e2e-full. That is the most meaningful local predeploy signal before touching Thor.

Deploy

Use the canonical script

Prefer deploy/thor/deploy.sh over the older just deploy-thor helper because the script handles build, artifact extraction, rsync, activation, and health.

Host identity

Set the real Thor IP

The repo defaults THOR_HOST=thor, which is only a placeholder unless your local SSH config resolves it. Use the real IP or a real SSH alias.

Canonical first deploy
from the repo root on Mac
just test-e2e-full

THOR_HOST=<thor-ip> THOR_USER=naac ./deploy/thor/deploy.sh
What the deploy script does
  • Builds the Docker deploy image unless --skip-build is used
  • Extracts release artifacts into ./artifacts/<version>
  • Rsyncs to /opt/naac/releases/<version> on Thor
  • Swaps /opt/naac/current to the new release
  • Restarts pmdaemon
  • Checks http://localhost:8001/health on Thor
Current repo caveat

The deploy and rollback scripts only smoke-check :8001/health themselves. The canonical gate for signoff in this repo is still VG-15, which checks 8001, 8002, and 8003.

Current runtime caveat

The Thor-target TensorRT and intent cascade paths are still partially stubbed in-repo, and there is model-path drift between some build scripts and runtime expectations. Treat the first deployment as infrastructure bring-up until those paths are reconciled.

8. Verify Health and Keep Rollback Ready

A deployment is not “done” just because rsync finished. The canonical activation gate in this repo is all three service health endpoints responding after the release switch.

Activation gate
VG-15 deploy activation
ssh naac@<thor-ip> 'readlink /opt/naac/current && curl -fsS http://localhost:8001/health && curl -fsS http://localhost:8002/health && curl -fsS http://localhost:8003/health'
Rollback rehearsal
VG-16 rollback visibility
THOR_HOST=<thor-ip> THOR_USER=naac ./deploy/thor/rollback.sh --list
Healthy means

/opt/naac/current points at the expected release and 8001, 8002, and 8003 all respond.

If activation fails

Stop treating the rollout as successful. Inspect logs, keep the previous release visible, and do not proceed into manual QA.

If rollback is missing

A first deployment without a visible previous release is an ops smell. The repo explicitly treats rollback as a deployment gate, not an optional later step.

9. Sources and Repo Authority

This guide combines current NVIDIA docs with this repo’s canonical deployment artifacts. It deliberately ignores older JetPack 6-era Thor task notes when they conflict with the current Thor installer path.

NVIDIA Thor Quick Start Mac/Windows/Linux host can create the installer USB; no host Ubuntu PC required for the initial BSP install.
NVIDIA JetPack Install / Setup Install JetPack components on-device with sudo apt install nvidia-jetpack.
SDK Manager System Requirements Ubuntu 24.04 is a supported host OS for SDK Manager, but that is no longer the fastest path for Thor first flash.
NVIDIA SDK Manager Matrix JetPack 7.x host compatibility and CLI/docker packaging details.
Docker Desktop FAQ Direct USB passthrough is not supported on Docker Desktop.
Docker USB/IP USB/IP exists, but it is a workaround path and not a first-line Thor flashing recommendation.
UTM macOS USB behavior UTM can prompt to connect USB devices to a VM, which makes it a fallback path for Ubuntu-in-VM workflows.
