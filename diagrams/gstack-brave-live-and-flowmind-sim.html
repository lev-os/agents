What the live Brave probe tells us

Filesystem

The DB is there and readable

The Brave cookie DB exists locally and could be opened without the temp-copy fallback. So the failure is not “Brave is impossible.”

Keychain

The secret lookup succeeds

A direct call to security find-generic-password -s "Brave Safe Storage" -w returned successfully. The environment can access the encryption secret.

Reality

Some rows still fail to decrypt

A raw one-row decrypt repro hit a WRONG_FINAL_BLOCK_LENGTH error, but the actual `gstack` importer still succeeded on 8 of 9 rows for `.github.com`. That means per-row failures are normal enough that the importer tolerates them.

02

The actual cookie/session flow

flowchart TD
  A["Run cookie-import-browser"] --> B["Find installed Chromium browsers"]
  B --> C["List domains from Cookies SQLite DB"]
  C --> D["User picks domains in localhost UI"]
  D --> E["Read Brave Safe Storage secret from Keychain"]
  E --> F["PBKDF2 derive AES key"]
  F --> G["Decrypt v10 cookie rows"]
  G --> H["Convert to Playwright cookies"]
  H --> I["Add to current browser context"]
  I --> J["QA browser is authenticated"]

  C -. DB busy .-> K["Copy DB to /tmp and reopen"]
  E -. deny/timeout .-> L["Keychain error"]
  G -. bad row .-> M["Skip failed cookie and continue"]
Why it feels seamless

`gstack` does not make you think about auth artifacts. It turns “import cookies from Brave” into “make the QA browser logged in now.”

Why it still feels brittle

That path depends on SQLite access, the right profile, the Keychain secret, and per-row decryption all being close enough to the current browser format to work.

Simulated seat outputs, verbatim

office-hours
STATUS: DONE

You're calling this a "daily briefing app," but your own pain points say otherwise.

What you described is a personal chief-of-staff assistant whose first visible output happens to be a morning briefing.

Three wedges:

1. Briefing-first
   Ship tomorrow. Pull events, detect stale logistics, and generate genuinely useful prep bullets.

2. Relationship-first
   Build the meeting graph first: who this person is, why they matter, what happened last time, and what to ask next.

3. Full chief-of-staff loop
   Calendar hygiene, prep generation, CRM memory, and time-prioritization in one system.

Recommendation:
Choose wedge 1. It is the only version that creates immediate user trust without forcing you to solve the whole assistant problem in one sprint.
eng-review
STATUS: DONE_WITH_CONCERNS

Architecture verdict: viable, but not implementation-ready yet.

Top gaps:

1. Enrichment is treated as if it were synchronous and reliable. It is neither.
2. Duplicate-event normalization is underspecified, which means trust will collapse the first time two calendars disagree.
3. There is no confidence gate on generated prep, so low-quality output will still ship.

Minimum fixes before build:

- make enrichment a background step with explicit retry + timeout
- define duplicate-event merge rules with test fixtures
- add a confidence threshold that degrades to "raw facts only" when prep quality is low
qa-report
STATUS: DONE_WITH_CONCERNS

QA summary: core flow is understandable, but not ship-ready.

Findings:

1. Critical — preview spinner can hang forever if enrichment returns partial data.
2. High — exclusion toggle updates local UI state, but the regenerated brief still includes the excluded event after refresh.
3. Medium — test delivery success toast appears before the request completes, which creates a false-positive feel.

Health score:
6.5 / 10

Recommendation:
Do not ship from this branch until exclusion persistence and preview timeout behavior are fixed.
cross-model-challenge
STATUS: DONE

Adversarial take:
You are not deciding whether to ship a rough alpha. You are deciding whether to normalize lying to yourself about broken state.

The critical bug is not cosmetic:
the preview can hang forever.

The high bug is not cosmetic:
the exclusion rule is not trustworthy across refresh.

That means the product's two core promises — "this is accurate" and "this respects what I changed" — are both unstable.

Recommendation:
Do not ship even an internal alpha unless the timeout cap and exclusion persistence are fixed first.
