01

What the verdict actually means

This is not a polite “maybe later.” It is a narrow, concrete yes: formalize what `gstack` is good at, and leave the rest behind.

What gstack really is

Specialist workflow pack + one serious browser runtime

The role system is static specialist routing: CEO review, eng review, QA, review, Codex challenge. The strongest engineering asset is the persistent browser daemon with refs, restart logic, session carryover, and buffered logs.

What Lev actually needs

Graph-visible seats, explicit proposals, clean ownership

Lev cannot absorb prompt-first orchestration. Every borrowed pattern has to land as FlowMind/session contracts, explicit workflow nodes, plugin-owned runtime behavior, and `LevEvent` provenance.

Operational takeaway

Extract seats and worker patterns, reject the control plane

Keep `gstack` as intake reference. Pull the seat-pack ergonomics, review-readiness surface, and browser worker concepts forward into the roadmap. Do not import `~/.gstack`, shell preamble governance, or slash-command workflow glue as canonical runtime.

Lev docs and architecture

Confirmed the four-plane boundary: FlowMind declares, Orchestration dispatches, Graph persists, Event Bus records. Also confirmed Step 10 and Track D are the right insertion points for specialist seats and graph-visible workflow nodes.

Repo Scan

gstack runtime vs skill glue

Separated the durable browser daemon from the mostly prompt-driven specialist workflow layer. Parallelism exists, but as human-managed isolated workspaces, not shared adaptive graph orchestration.

Fit Scan

Disposition pressure test

Forced the intake into one of five outcomes. `extract` won because it captures nearly all near-term value without violating ownership or control-plane rules.

Synthesis

Strong common ground, low residual ambiguity

All scans agreed that `gstack`'s specialist seat patterns are useful and its browser daemon is technically mature, but the orchestration model is too implicit, host-centric, and non-graph-native to become Lev's runtime substrate.

04

What to borrow, what to reject

Borrow

Patterns Lev should formalize

  • Specialist seat packs as named workflow nodes: office-hours, eng review, QA, codex challenge.
  • Stage-aware routing as a surface affordance once it compiles into explicit proposals or sessions.
  • Review-readiness as a dashboard-visible status layer with freshness/staleness tracking.
  • Cross-model challenge seats as explicit dissent/adversarial nodes inside CDO/A2A orchestration.
  • Persistent browser worker semantics for lifecycle, restart, refs, auth, and buffered logging.
Reject

Patterns that would hurt Lev if copied directly

  • Prompt-first control plane hidden in shell preambles and markdown skill glue.
  • Host-centric state under ~/.gstack, ~/.claude, or ~/.codex.
  • Static slash-command orchestration passed off as graph-native multi-agent execution.
  • Workspace concurrency ≠ CDO; isolated workspaces are useful, but they are not adaptive synthesis loops.
  • Policy in prompts instead of runtime when the same behavior should be enforced by declarations or code.
05

Where it lands on the roadmap

The value is not abstract. It plugs into specific roadmap surfaces that already exist.

Step 10

Workflow engine: all skills as runnable and visualizable workflows

Best fit for the specialist seats. The dashboard should render roles like office-hours, eng-review, qa, and codex-challenge as real nodes with inputs, outputs, and status, not as freeform prompt rituals.

Track D

FlowMind program-driven loop + Full A2A subagent orchestration

This is where cross-model challenge seats belong. The useful `gstack` idea is not “use Codex sometimes,” it is “treat second-opinion and adversarial passes as named seats in the graph.”

Steps 7 + 9

levd WebSocket adapter and agent polling loop

The browser worker lessons matter here: fast round-trips, durable session state, actionable failures, auto-restart, and structured logs all improve how dashboard-triggered workers feel when they run live.

Plugin boundary

plugins/browser-cascade/**

The browser daemon should not leak into core orchestration. Lev already has a boundary for browser work. That is the correct extraction target.
