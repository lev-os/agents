Lev Poly Architecture Truth Map

Spec vs. reality for every core module. What they say they do, what they actually do, and where the gaps live.

Generated 2026-03-21 from docs/specs/*, docs/design/*, core/poly/**, core/daemon/**

S1: What Each Module IS

One-to-three sentence JTBD per module. Spec claim, code reality, and gap assessment.

poly @lev-os/poly

Spec says: Poly is a pure binder and codegen pipeline. It reads module SDK declarations, builds a unified registry, generates control surfaces (CLI, MCP, HTTP, gRPC, WS), and routes commands to owning packages. Zero business logic.

Code does: The CLI surface (surfaces/cli/index.ts) has 8 hardcoded if-blocks for work, review, gather, sdlc, loop, autodev, flowmind, and exec that bypass the discovery/registry system entirely. The "~10 LOC build artifact" binary is actually 46 lines of tsx-respawn logic. Command discovery is filesystem-based, not registry/config.yaml-driven. The 20 command handlers total 4,579 LOC in commands/. The codegen pipeline (adapter-gen, proto-gen, zod-gen, binary-compiler) exists but the surfaces are hand-written, not generated.

Gap: The core spec promise -- "add a config.yaml declaration, all surfaces get the command automatically" -- is not implemented. Surfaces are hand-coded. The CLI is a conventional if/else router, not a codegen output. The codegen pipeline exists in isolation from the actual dispatch path.

daemon @lev-os/daemon

Spec says: Daemon owns process supervision, dependency DAG ordering, health monitoring, task queues, worker pools, and adapter-based process management (process, brew, pmdaemon). Poly routes daemon commands; daemon implements them.

Code does: 11,155 LOC across 20 source files. Full supervisor with dependency graph, health monitor, queue manager, worker pool (including warm pool), three adapters (process, brew, pmdaemon), OTel telemetry bridge, orchestrator integration, and a 966-line CLI lifecycle handler. The ownership boundary is clean -- core/poly/src/commands/daemon.ts is 152 lines of pure delegation.

Gap: Aligned. The spec-to-code boundary is the cleanest in the entire system. The poly daemon handler delegates to @lev-os/daemon with zero leaked implementation.

exec @lev-os/exec

Spec says: Stateless factory (createExec(dispatch)) producing frozen ExecEngine instances. Per-provider concurrency via ProviderSemaphore, canonical LevEvent emission, structured ExecResult envelopes. Plus a DI-driven client facade for registry-driven execution.

Code does: The factory/semaphore/event pattern is implemented. client.ts provides the DI facade with lazy transport bootstrap. Poly's SDK wires into it via createExec(call).

Gap: Aligned. Exec is structurally sound and the ownership boundary with poly is well-defined.

build @lev-os/build

Spec says: Build-time-only toolchain. Codegen, FlowMind compilation, skills indexing, source aggregation, markdown sanitization. No runtime package may depend on it. Install/onboarding is being migrated out to @lev-os/onboarding.

Code does: Poly uses lazy re-export shims (wizard.ts, registry-loader.ts) to break the cycle. The boundary is respected -- build does not leak into runtime paths.

Gap: Aligned. The SDK-first mandate (typed options in, typed results out) is the pattern.

config @lev-os/config

Spec says: Four-scope resolution chain (system, project, module, env). XDG-compliant paths. Schema validation with strict key rejection. Fractal config composition.

Code does: Config resolution, merge, XDG paths, and schema validation live in core/config/src/. Poly's config-apply command delegates properly.

Gap: Aligned. The monorepo wiring gap (flowmind can't resolve @lev-os/config) is documented and worked around.

event-bus @lev-os/event-bus

Spec says: Canonical event spine: pub/sub with pattern matching, JSONL persistence, durable queues, checkpoint/replay, session recovery, XState v5 aggregate machines (Idea, Session, Synth, PR), lifecycle hooks, workflow loading, bridges, frontmatter management.

Code does: 11 subsystems implemented across events, context, runtime, aggregates, policy, bridge, frontmatter, actions, guards, types, and workflow loader. The spec catalogs every file and subsystem exhaustively.

Gap: Aligned. The spec is at parity with code. Ephemeral store was pruned (tick 164).

flowmind @lev-os/flowmind

Spec says: Declaration execution and policy plane. Owns the kernel (declaration + execution), execution contracts, and the executor. FlowMind drives everything -- agents, workflows, policies, harness configs are all FlowMind declarations.

Code does: core/flowmind/src/kernel/** and core/flowmind/src/execution-contract/** exist. The CLI surface hardcodes a flowmind if-block rather than discovering it through the registry. AI provider factory with model profiles works.

Gap: Partial. FlowMind is the declared policy plane but the "drives everything" aspiration is not fully realized. The CLI bypass (hardcoded if-block) undercuts the registry-first model.

domain @lev-os/domain

Spec says: Stable shared contracts -- types, LevEvent factory, ExecTransport interface. The type spine that every other module imports from.

Code does: event-source.ts provides LevEvent factory. ExecTransport is defined here for DI. Domain is the shared type boundary.

Gap: Aligned.

logger @lev-os/logger

Spec says: Pino-based structured logging. Zero workspace dependencies. Standalone package imported directly, not through poly.

Code does: Standalone structured logging. Design doc explicitly says dead re-exports in poly should be deleted.

Gap: Aligned.

S2: The Poly Problem

The design doc says "each surface reads the same registry and produces a different wire format." Here is how it actually works.

Spec: Registry-driven dispatch (the dream)
user: lev X --> poly bin/lev --> createCliSurface(registry) --> registry lookup --> module handler --> response

Every command flows through the same registry. Add to config.yaml, all surfaces get it. ~10 LOC binary.

Reality: Hardcoded if-blocks + filesystem discovery
user: lev X --> bin/lev (46 LOC tsx respawn) --> surfaces/cli/index.ts
if X == 'work' --> @lev-os/core-sdlc (dynamic import)
if X == 'review' --> @lev-os/core-sdlc (dynamic import)
if X == 'gather' --> handlers/gather.ts (lazy re-export)
if X == 'sdlc' --> @lev-os/core-sdlc (dynamic import)
if X == 'loop' --> @lev-os/core-sdlc (dynamic import)
if X == 'autodev' --> @lev-os/core-sdlc (dynamic import)
if X == 'flowmind' --> @lev-os/flowmind (dynamic import)
if X == 'exec' --> core/harness (hardcoded path)
else discoverCommands() --> filesystem scan core/*/src/commands/ --> registry.commands[X]

8 commands hardcoded before the registry is even consulted. Discovery scans filesystem, not config.yaml poly.sdk declarations. MCP/HTTP/gRPC/WS surfaces are hand-written, not codegen outputs.

Why the if-blocks exist

Each hardcoded if-block exists to break a circular dependency. The comment in the code says it explicitly: // Handler imports are lazy (dynamic) to avoid circular deps: poly -> core-sdlc -> poly/surfaces/cli/utils. The exec block also needs special handling because "exec is a harness command, not a file-discovered command." These are pragmatic workarounds, but they violate the spec's registry-first promise.

Discovery is filesystem-based, not config-driven

discoverCommands() walks core/*/src/commands/*.ts and plugins/*/src/commands/*.ts to find handlers. It never reads config.yaml poly.sdk declarations. The config.yaml-driven registry described in the design doc and spec is not wired into CLI dispatch at all.

S3: What Does the User Actually Want?

BDD-style scenarios extracted from the specs and design docs, capturing the real intent behind the architecture.

Given poly is a binder that reads config.yaml poly.sdk declarations

When I add a handler declaration to any module's config.yaml

Then all surfaces (CLI, MCP, HTTP, gRPC, WS) automatically discover and expose that command without touching poly source code

Given daemon manages process lifecycle and poly only routes

When I run lev daemon start api

Then poly parses args and delegates entirely to @lev-os/daemon, which starts the process through the configured adapter, respects the dependency DAG, and emits health check events

Given exec owns the execution engine as a stateless factory

When any surface (CLI, MCP, HTTP) needs to execute a provider call

Then it calls createExec(dispatch).exec(), gets per-provider concurrency control, backpressure handling, and canonical exec.started/exec.completed/exec.failed LevEvents -- without reimplementing any of that logic

Given the codegen pipeline generates proto, Zod, and adapter stubs from service manifests

When I run lev-dev codegen

Then the generated CLI surface, MCP tools, HTTP endpoints, and gRPC services all reflect the current registry state -- and I never hand-write surface code

Given a TypeScript, Python, Go, or Rust package declares handlers via config.yaml

When the registry builder scans the workspace

Then all handlers appear in the unified registry with typed schemas, and I can call Go from TypeScript or Python from Rust through poly SDK call()

Given FlowMind is the declaration and policy plane

When I define a .flow.yaml workflow with stages and gates

Then the kernel executes it deterministically, emits LevEvents at each transition, and the event-bus persists/replays the full history

Given event-bus owns persistence, replay, and recovery

When a gateway process crashes mid-session

Then session recovery detects the orphan, replays from the last checkpoint, and resumes execution without data loss

Given the plugin validator enforces the leviathan contract

When a plugin at plugins/X/ declares daemon, MCP, and SDK handlers in its manifest

Then the registry builder discovers it, the daemon starts it, and lev X:command routes through poly to the plugin's handler with namespace isolation

S5: Does Any of This Make Sense?

The Architecture Is Sound

  • The ownership map is well-defined and mostly respected. Seven of nine core modules are at parity between spec and code.
  • The daemon boundary is the gold standard -- 152 LOC of delegation in poly, 11,155 LOC of implementation in daemon. That's a clean cut.
  • The exec factory pattern (stateless, frozen, per-provider semaphore) is elegant and correctly isolated.
  • The event-bus is genuinely comprehensive: 11 subsystems, XState v5 aggregates, checkpoint/replay, session recovery. It's the most mature module.
  • LevEvent as the only inter-module event schema is the right call. One envelope, one spine, no legacy variants.

The Implementation Has One Structural Lie

  • The spec says: "Control surfaces are codegen outputs, not hand-written adapters." The code says: every surface is hand-written. The codegen pipeline (proto-generator, zod-generator, adapter-generator) exists and works, but it is not wired into the actual CLI/MCP/HTTP/gRPC/WS dispatch path.
  • The CLI surface has 8 hardcoded if-blocks that bypass the registry. These exist for valid reasons (circular dependency breaking), but they mean adding a new command requires editing surfaces/cli/index.ts, not config.yaml.
  • Discovery (discoverCommands()) scans the filesystem for src/commands/*.ts, not config.yaml poly.sdk declarations. This is conventional and works, but it's not the config-driven registry the specs describe.
  • The other surfaces (MCP, HTTP, gRPC, WS) each have hand-written index.ts + server.ts pairs. They are not generated from the same registry the CLI uses.
The gap is not a design flaw -- it's an execution gap. The architecture is right. The codegen pipeline exists. The surfaces exist. They just aren't connected yet. The if-blocks are pragmatic cycle-breakers. The filesystem discovery is a working bootstrap. The path forward is clear: wire the codegen pipeline into surface generation, collapse the if-blocks into registry entries, and make config.yaml the single source of truth for command registration.

Does Poly Need to Be Split?

  • No. Poly's scope is correct: binder + codegen + surfaces + SDK transport. That's one coherent concern.
  • What it needs is subtraction, not division. Move config-apply (205 LOC), doctor (593 LOC), and cdo (364 LOC) to their owning modules. That's 1,162 LOC of misplaced business logic.
  • Then close the codegen loop: make the surfaces actual codegen outputs. The pipeline is there. The templates are there. The connection is missing.

The Simplest Path to Making the Spec True

  • Phase 1 (hours): Move config-apply.ts, doctor.ts, and cdo.ts to their owning modules. Replace with thin re-export shims in poly. Brings poly commands from 4,579 to 3,417 LOC of actual routing/formatting.
  • Phase 2 (days): Collapse the 8 hardcoded if-blocks into registry entries. Each module that needs early routing (work, review, exec, etc.) declares itself in config.yaml with a handler URI. The CLI surface reads the unified registry first, filesystem-discovers second.
  • Phase 3 (week): Wire the codegen pipeline into surface generation. The proto-generator, zod-generator, and adapter-generator already exist. Make them emit the CLI dispatch table, MCP tool definitions, HTTP router, and gRPC stubs from the same registry manifest. Each surface becomes a thin runtime that loads codegen output.
  • Phase 4 (validation): Add the test infrastructure gate: 4 language test packages (TS/Py/Go/Rust), each with 5 handlers + 3 dep-backed handlers = 20 handlers in the registry. Prove the full cross-language matrix works. The spec already has 28 BDD scenarios for this.
The architecture is sound and well-specified. The daemon boundary proves the team can do clean cuts. The remaining work is mechanical: connect the codegen pipeline to the surfaces, move misplaced business logic to owning modules, and replace the hardcoded if-blocks with registry entries. None of this requires rethinking the architecture -- it requires finishing the implementation.
