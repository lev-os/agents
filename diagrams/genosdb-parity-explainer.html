GenosDB Parity / Lev Architecture

One-to-one parity without turning Lev into GenosDB.

The analysis landed on a strict shape: GenosDB is a strong parity target for browser-local graph state, realtime sync, signed operations, and collaboration transport, but the product itself must not become a new fused core module. The correct move is plugin-first parity plus a small set of reusable engine contracts.

Core Read

Lev already has the hard boundaries that matter: core/graph for state, core/event-bus for replay, core/flowmind for policy, core/orchestration for execution, and AgentPing for host surfaces.

That means the winning question was not “how do we import GenosDB?” but “which capabilities become plugins, and which missing contracts are reusable enough to justify a core upgrade?”

01 — Decision
Rejected shape vs chosen shape
The whole analysis hangs on one architectural decision: parity is a capability mapping problem, not a product-port problem.
Rejected

“Make a new core GenosDB module”

  • Fuses graph, replay, transport, auth, and host concerns into one runtime product.
  • Violates plugin-first policy and makes browser assumptions feel core-owned.
  • Pushes WebRTC, Nostr, OPFS, and WebAuthn pressure into the engine too early.
  • Overfits Lev to one external product shape instead of upgrading reusable contracts.
Chosen

Plugin-first parity with reusable contract upgrades

  • Keep state, replay, policy, execution, and host ownership where they already belong.
  • Deliver browser-local and realtime features as plugins.
  • Add only transport-agnostic contracts that future plugins also need.
  • Treat GenosDB as a high-signal parity reference, not a dependency or template for core packaging.
02 — Split
The minimum stable decomposition
Four plugins are enough to cover the public GenosDB surface without collapsing Lev’s ownership planes.
Plugin 1

core-browser-store

OPFS/IndexedDB persistence, worker lifecycle, BroadcastChannel tab sync, reactive get/map parity.

Plugin 2

core-realtime

Rooms, channels, media streams, relay discovery, TURN config, and cellular mesh transport behavior.

Plugin 3

core-browser-auth

WebAuthn-backed keys, signed operations, verifier-only nodes, RBAC/ACL helpers, secure local record wrappers.

Plugin 4

core-query-pack

Radix and inverted indexes, geo, NLQ, audit, and governance as optional add-ons instead of baseline runtime features.

Why this is the smallest shape
  • Storage/query reactivity is one concern.
  • Peer transport is a second concern.
  • Browser-held identity and signed ops are a third concern.
  • Optional enrichments are a fourth concern.
What it avoids
  • No new core/genosdb ownership island.
  • No WebRTC or relay logic in core/poly.
  • No second event spine beside core/event-bus.
  • No Ethereum/WebAuthn assumptions becoming universal core identity.
03 — Core Deltas
What the engine actually needs
The analysis only justified four reusable core upgrades, and all four are neutral contracts rather than GenosDB-specific implementations.
K1

Realtime collaboration transport

Lev has host and surface language, but not a neutral room/channel/media/session transport contract that WebRTC, libp2p, or Nostr-backed plugins can all target.

Contract Gap Core
K2

Signed operations + browser identity proofs

Lev needs a provider-neutral contract for signed mutation envelopes, verifier-only nodes, and delegated permission evaluation.

Contract Gap Core
K3

Reactive graph subscriptions

GenosDB’s live get(id, cb) and map(query, cb) semantics need a neutral Lev contract so browser and dashboard plugins don’t invent incompatible live-view APIs.

Contract Gap Core
K4

Browser-local canonical storage clarification

spec-storage-adapters already defines a storage role. It now needs to say how browser-local canonical adapters work for host-side workspaces without weakening XDG rules for core/runtime state.

What maps where
The parity spec classified every visible GenosDB feature. This matrix is the quick read for architecture review.

Storage OPFS + worker persistence plugin-native plugins/core-browser-store Strong parity target, but strictly browser/plugin owned.
Graph node graph + recursive $edge native core/graph Lev already owns the right semantics; parity is mostly exposure and projection work.

Packaging / typings dist-only, empty typings, proprietary core rejected none Parity is capability-level only; Lev should not copy this packaging model.

Bottom line

GenosDB should not become a core module. Its best value to Lev is as a parity target that forces the engine to expose cleaner reusable contracts for browser-local state, collaboration transport, signed operations, and live graph views. Everything else stays plugin-first.
