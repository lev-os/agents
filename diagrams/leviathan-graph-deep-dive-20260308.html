Key Point

The graph is not the product.

The graph is the structural intermediate that lets Lev compare and reason across media once each medium has an adapter.

FlowMind's Role

Declarative control

FlowMind is interesting here because it can potentially formalize the prompt-analysis loop itself: prompt to structure, structure to workflow, workflow back to prompt.

Current Reality

External reference first

The Python compilers are still the working reference implementations. Lev has the right pieces, but not native parity yet.

What is real
  • Compiler and executor code exists in core/flowmind.
  • Architecturally, FlowMind is already framed as control/policy plane.
  • Good substrate for declarative classification workflows.
What is not real yet
  • Prompt-analysis parity is not native inside FlowMind yet.
  • core/graph does not yet act as a fully policy-shaped universal graph runtime.
  • The docs promise more than the current graph runtime currently delivers.
4 — Our Graph Theory In Plain English
Nodes

What counts as a thing

Prompts, files, entities, claims, evidence, views, episodes, or runtime artifacts can all become nodes if the adapter says they matter.

Edges

What counts as a relation

Overlap, provenance, ownership, derivation, similarity, causality, and workflow links are all candidate edge types.

Seams

Where translation happens

The most important design problem is not “graphs are cool.” It is choosing stable seams between raw media, extracted structures, graph entities, and materialized views.

Views

What the graph is for

The graph exists so multiple downstream operations become shared: compare, cluster, traverse, explain, publish, and feed workflows.

The Art

The hard part is deciding:

  • what is a node vs just evidence
  • what is an edge vs derived similarity
  • what is source of truth vs materialized projection
  • what gets normalized away vs preserved
Why the current metrics are fragile

PromptOS and Leviathan are being extracted with different ontologies and different artifact mixes, so density comparisons are useful signals, not final truth.

Nanograph

Local typed graph database

Nanograph is a graph DB with a schema language, query files, seed data, and a local database directory.

What it did

Validated the export path

It let us prove that the Leviathan harness graph can be rendered into a real queryable local graph DB artifact.

What it did not do

Not the native Lev graph

It did not replace core/graph, and it did not prove itself as the final storage backend.

7 — What We Actually Proved
Proved
  • The prompt-analysis POC works today as external Python tooling.
  • Prompt corpora are highly convergent under the current PromptOS extractor.
  • Leviathan can analyze itself through a parallel harness-graph compiler.
  • Leviathan harness outputs can be exported into Nanograph and queried.
Not Proved
  • That native Lev/FlowMind already reproduces the prompt-analysis loop.
  • That Nanograph should be the graph backend.
  • That the current density comparison is extractor-robust enough for big product claims.
  • That the universal graph vision is fully implemented today.
