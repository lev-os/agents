2 / Renderer model

How this should work

Your instinct is right: define the shell once, then let agents submit payloads that mutate state and trigger ultra-fast rebuilds of the same page.

Stable shell

shell.mustache plus the partials define the layout family. This is the expensive design work. It should change rarely.

layout zones widgets

Schema-driven content

schema.json decides which widgets exist and where they bind. If an agent wants a new panel, diff widget, inbox lane, or metrics strip, it changes schema and data.

schema switch widget registry bindings

Append-only state

current.json is the materialized latest state; ticks.jsonl, mail.jsonl, and actions.jsonl are the durable logs. The page is rebuilt from those, not reinvented.

3 / Daemon vs Astro vs React tree

What this actually is

You asked whether this is different from Astro. It is, but not because the final output differs. It is different because the control loop and data model differ.

Comparison
Approach Good at Bad at My take here
Astro-style static site build Multi-page content sites, docs, islands, publishing Session-native append-only interaction model is not its natural center Overkill for v0. Fine later if this becomes a broader publishing surface.
Tiny local build daemon Ultra-fast rebuilds from schema/data mutations into static HTML Less batteries included; you own the contract Best fit now. The POC already proves this direction.
React tree / VDOM host Very interactive widgets, local stateful complexity Easy to overbuild before the session-shell contract is stable Good later for richer widgets or AgentPing-hosted surfaces, not required for the shell core.
BertUI-style ultra-fast static pipeline Exactly the “schema in, static page out very quickly” instinct you described You still need your own session grammar, asset conventions, and bridge semantics The pattern is right even if we don’t copy the stack literally.
4 / Plugin question

Should this be a plugin?

Not yet. Right now this is exactly a custom local renderer. That is not a failure; it is the right phase.

Why it should stay local now

The contract is still moving: session hierarchy, widget schema, mailbox semantics, AgentPing bridge rules, and artifact mirroring are not stable enough to promote. If you pluginize now, you will harden the wrong interface.

still moving renderer not settled bridge not settled

What promotion would look like later

Split the reusable renderer/compiler into plugins/genui or plugins/core-genui. Then let Lev use AgentPing via a platform plugin beneath it. That keeps the fractal inversion you described without overloading AgentPing with the abstraction burden.

renderer plugin AgentPing platform plugin LevUI IR stays above
My recommendation

Keep the shell local in .lev/gen-ui. Use a tiny watch daemon as the primary runtime. Emit static HTML every time. Treat the schema as the session grammar and AgentPing as the interaction/host bridge. When the grammar stops moving, then promote the reusable parts into a plugin.
